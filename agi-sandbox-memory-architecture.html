CareLink AGI — Sandbox & Memory Architecture

Generated 2026-04-28. Companion to docs/agi-architecture-overview.html. This doc focuses on the sandbox-and-memory layer specifically.
Captures the design discussion that led to 1 sandbox = 1 session = N agents, the ephemeral session memory + long-term canonical memory model, and the pre-task fetch / post-task update sync semantics. Includes options considered, trade-offs evaluated, and what we eventually landed on.

The question we set out to answer

Where does state live in the AGI runtime, and how does it persist across sessions when many concurrent sessions across many pharmacies need to share learned context?

This collapses into two cardinality questions and one architectural question:

  1. Session ↔ Sandbox — does each session get its own sandbox, or do sandboxes get pooled and reused across sessions?
  2. Agent ↔ Sandbox — does each agent (orchestrator, each expert) get its own sandbox, or do they share?
  3. Memory persistence — given the cardinality answers, how does memory survive past a session's lifetime and become available to future sessions?
What we landed on
1 sandbox = 1 session = N agents, with ephemeral session memory + long-term canonical memory fetched pre-task and updated post-task.
  • Session ↔ Sandbox is 1:1 — each session has its own ephemeral sandbox, created at session start and destroyed at session end
  • Agent ↔ Sandbox is N:1 — orchestrator and all experts (intake, billing, dur, etc.) share the same sandbox within a session
  • Memory is two-tier: canonical (durable, pharmacy-scoped) lives at /memory/<org>/...; per-session projection lives inside the sandbox at workspace/memories/
  • Pre-task fetch: at session start, a memory selector copies a scoped subset of canonical into the sandbox + writes a manifest
  • Post-task update: at session end, a merger diffs sandbox vs manifest and atomic-appends additions to canonical with signed entry blocks
  • 14-day TTL on canonical entries via Inngest cron sweep

Why this model holds up

Property we wanted How 1:1:N + two-tier memory delivers it
Cross-session learning available pharmacy-wide Canonical is pharmacy-scoped and persistent; every session at that pharmacy projects from it at start
Within-session consistency across experts Many agents share one sandbox → intake's note is readable by billing later in the same session
No cross-org leakage Canonical store is partitioned by org_id; sandbox path includes org_id; org-routing middleware locks every request to one DB
Pharmacy PHI stays at pharmacy Patient/prescriber/insurance buckets live at /memory/<org>/pharmacy/<P>/...; selector won't project across pharmacy boundaries
Cold-revive across machine reboots Sandbox lives on shared FS; any backend replica can revive it; lease columns route subsequent ops to whichever replica owns it now
Deterministic merge boundary Exactly one snapshot per session, exactly one merge per session — no coordination across sandboxes needed
Cleanup is well-defined Sandbox lifetime = session lifetime; finalize_session's shutil.rmtree already handles this

Options explored — the cardinality decision tree

Two independent axes. Click any option to expand its trade-offs.

Axis A — Session ↔ Sandbox

How many sessions can share a sandbox?

Chosen A1 — One session = one sandbox Today's behavior, kept

Pros

  • Clean lifecycle — sandbox dies when session ends; cold-revive reuses by path stored on extra_data["workspace"]
  • Per-session bearer rotation works trivially (token written into apis.json at sandbox creation)
  • Memory snapshot tied to a session start is unambiguous (manifest at workspace/.memory-manifest.json)
  • Lease has a 1:1 thing to point at
  • Take-over semantics are obvious (lease moves; sandbox stays)

Cons

  • ~150-500ms setup cost per session (skills copy is the slowest piece)
  • Disk churn on high-volume deployments
  • No warm cache of pre-built sandboxes

Verdict

The bearer-token rotation, cold-revive, and snapshot semantics arguments are decisive. Setup cost is real but small — sessions are minute-scale, not millisecond-scale. Reserve pooling for the day skills copy becomes a measurable bottleneck.

Rejected A2 — One sandbox = many sessions (sandbox pool) Pre-warmed pool, sessions check out a sandbox

Pros

  • Amortize setup cost — pre-warm a pool of workspaces ready for instant session start
  • Less FS churn, friendlier on shared-FS substrates (Filestore IOPs)

Cons (significant)

  • Bearer rotation becomes ugly — can't bake into apis.json since multiple sessions share. Either every restish call needs an env-var override of bearer, or some token-injection layer rewrites apis.json mid-flight (race conditions)
  • Memory snapshot can't be a per-sandbox manifest; would need per-call snapshot files. Manifest semantics fragment
  • Cold-revive gets weird — which sandbox in the pool was this session bound to? Have to invent a session→sandbox lookup
  • Take-over and lease semantics need an extra layer (lease points to session, session points to sandbox slot, sandbox slot points to FS path)
  • One session's transient files (proposals/, in-progress memories/) visible to a sibling sharing the sandbox — cross-session contamination risk even within the same pharmacy

Verdict

Performance wins are real but small (skills copy is 100-200ms). The semantic regressions are large and load-bearing. Rejected.

Deferred A3 — Hybrid: shared sandbox root with per-session subdirs Shared skills tree via symlinks; per-session writable dirs

Shape

/sandbox-pool/<pool_id>/ for shared skills/agents; /sandbox-pool/<pool_id>/sessions/<sid>/ for per-session memories, proposals, and apis.json. Skills/agents read-only via symlinks.

Pros

  • Amortizes the heavy copies (skills are ~3,200 LoC of bundled Python + MDs)
  • Preserves session isolation for writable content

Cons

  • Complexity for marginal gain — the skills copy is the slow part today; the rest is microseconds

Verdict

Worth holding in our pocket as a future optimization. Deferred: revisit if skills copy time crosses 2-3% of total session latency.

Axis B — Agent ↔ Sandbox

How many agent processes can share a sandbox within one session?

Rejected B1 — One agent = one sandbox (per-spawn isolation) Every orchestrator and every expert spawn gets its own workspace

Pros

  • Hard isolation — experts can't read each other's mid-session writes, can't see other experts' MDs they aren't authorized for, can't see proposals from sibling experts
  • Crash isolation — a runaway expert can't pollute the orchestrator's workspace
  • Cleaner answer if we ever do parallel expert dispatch (no race on memory file writes)

Cons (significant)

  • Heavy: skills copy + agent MD copy + apis.json write times N agents per session. For an 8-step rx-flow run, that's 9 workspace builds (orchestrator + 8 experts)
  • Memory snapshot semantics fragment — does each agent get its own snapshot? Or does the orchestrator's snapshot get re-projected into each expert? Both are awkward
  • Within-session learning across experts (intake's note → billing's read) becomes impossible without an explicit cross-agent shared store, which is basically re-inventing the shared sandbox
  • Bearer rotation per agent? Probably not needed (same session = same actor) but the per-agent envelope makes it possible, which can quietly become a thing someone implements when they shouldn't

Verdict

The within-session memory propagation is a real feature, not a bug. Hard isolation buys us very little for sequential dispatch. Rejected for v1.

Chosen B2 — One sandbox = many agents (orchestrator + experts share) Today's behavior, kept

Pros

  • Lightweight — sandbox built once per session, all agents share
  • Within-session memory propagation works naturally (intake writes memories/drugs/<ndc>.md; billing reads it)
  • Workspace IS the unit of isolation — clean mental model
  • Matches Claude Agent SDK's default subagent semantics (Task tool inherits cwd)

Cons

  • Experts can read sibling experts' working files. Today this is mostly cosmetic since they share the same memory scope by design. Would matter under parallel dispatch (race condition)
  • An expert could in principle Read another expert's MD file (since .claude/agents/ contains all of them). Mostly informational disclosure, not a privilege escalation

Verdict

The right answer for sequential dispatch (which is what we do today). Chosen.

Deferred B3 — Hybrid: orchestrator workspace + per-expert subdir Orchestrator gets the full workspace; each expert gets workspace/agents/<expert_id>/

Shape

Skills + memories + apis.json shared via symlinks at the parent. Per-expert proposals + transient files isolated.

Pros

  • Per-expert proposals + transient files isolated
  • Memory file writes go to per-expert dirs that the merger reconciles at session end
  • Parallel dispatch becomes safe (each expert writes to its own subtree)

Cons

  • Memory merge gets two-stage: per-expert merge into session-level workspace at expert-end, then session-level merge into canonical at session-end
  • _spawn_expert_agent complexity grows (per-expert cwd + symlink farm)

Verdict

The natural escape hatch the day we want parallel expert dispatch. Design the v1 memory merger and _spawn_expert_agent so B3 is a future drop-in, not a redesign. Deferred.

How memory persistence intersects with the cardinality choices

The memory model wasn't a separate axis — it falls out of the cardinality decisions:

  • Sandbox is the unit of memory consistency. 1:1 with session means snapshot-then-merge has exactly one well-defined boundary.
  • Many agents share one snapshot. Intra-session writes are visible to later experts in the same session.
  • Cross-session learning requires a separate persistent layer above sandboxes — that's the canonical store.
  • Pre-task fetch / post-task update is the natural sync model for "ephemeral session sandbox + persistent canonical store."

If we'd chosen A2 (sandbox pool), this would have to be re-thought; if we'd chosen B1 (per-agent sandbox), the merge step would need cross-agent coordination. The 1:1:N cardinality is what makes the memory model simple.

Sandbox lifecycle — the 8 states

A sandbox passes through up to 8 distinct states from creation to destruction. The sandbox persists across multiple subprocess restarts — it's the durable backing store; subprocesses come and go.

Click any state below for details.

1 provisioning ~150-500ms — workspace built; memory snapshot taken

Triggered by POST /sessions after access checks pass. _prepare_workspace at service.py:1531-1560 creates the directory tree:

  1. mkdtemp (or mkdir of session-id-named subdir on shared FS) — atomic
  2. Skills copy → .claude/skills/ (slowest step today; ~100-200ms)
  3. Agent MDs copy → .claude/agents/
  4. apis.json write → .config/restish/ and Library/Application Support/restish/
  5. Memory snapshot — selector reads canonical, copies subset to memories/, writes manifest at .memory-manifest.json
  6. Workspace path stamped on session row — extra_data["workspace"] = str(workspace)
  7. Lease acquired

If any step fails: partial workspace exists, session row may not have extra_data["workspace"] populated. Sweeper cleans these up: orphan workspace dir whose corresponding session row doesn't have it on extra_data → rmtree.

2 active Lease held by replica X; orchestrator subprocess running
  • One subprocess (orchestrator) is alive
  • Workspace is being read/written: experts read memories/, write to proposals/ and memories/
  • apis.json bearer is the actor's bearer (1h TTL by default)
  • Workspace is NOT being moved or duplicated — every spawn (_spawn_expert_agent) reuses the same path via kwargs["cwd"] = str(workspace)
  • Heartbeat extends lease_expires_at every ~15s
3 waiting_hot Subprocess parked at HITL gate; lease still held
  • Subprocess is alive but in await wait_for_decision_or_timeout
  • No state writes (the gate just blocks)
  • Workspace untouched
  • Lease still held by the original replica
  • Heartbeat continues — if it stops, lease expires after 60s

The sandbox doesn't change shape across the active ↔ waiting_hot transition. The orchestrator is just paused.

4 waiting_cold Subprocess EXITED; lease released; workspace persists on shared FS

The critical state. The cold-park decision means:

  • SDK subprocess exits cleanly via continue_=False, stopReason=...
  • Lease released (heartbeat stopped or explicit release)
  • Workspace stays on disk — this is non-negotiable
  • DB row's extra_data["workspace"] still points at it
  • Session row status flips to waiting_cold
  • No replica "owns" the session anymore

Cost: zero CPU/RAM. Just disk (~50-200MB). This state can persist for minutes (operator stepped away to take a phone call) to days (operator went on vacation). Without a hard TTL, sessions accumulate here.

5 cold_reviving Any replica picks up the parked session; spawns fresh SDK client

Trigger: HITL answer arrives, OR cold approval decided, OR explicit re-acquire.

  1. Inngest event fires (with org_id and session_id)
  2. Any backend replica handles the event — read OrgX DB → session row → extra_data["workspace"]
  3. Workspace must be readable from this replica — this is why shared FS is mandatory
  4. Mint a fresh bearer (the original 1h token has expired) → rewrite apis.json
  5. Acquire lease (force=true since prior released)
  6. Spawn fresh ClaudeSDKClient with resume=session_id and cwd=<the same workspace>
  7. Drain via drive_client

The SDK CLI's own session state lives at <workspace>/.claude/projects/<sanitized-cwd>/<session_id>.jsonl (see comment at service.py:496-499). That's why the workspace path can't change — the SDK reads its own JSONL from there to continue.

6 take_over Lease changes hands; sandbox stays
  1. Bob's frontend POST /sessions/{sid}/lease with force=true
  2. Backend marks lease revocation pending
  3. SSE event sent to Alice's frontend → her tab cleans up
  4. Replica holding active SDK client drains gracefully (finishes current turn or hits quiescent boundary)
  5. Lease updated (new holder = Bob's machine)
  6. Same workspace path — replica now driving the session reads from shared FS
  7. New ClaudeSDKClient spawned with resume=session_id

The sandbox doesn't move. The lease moves.

7 finalizing → terminated Memory merge runs; audit flushed; sandbox rmtree'd

finalize_session runs (service.py:1480-1524):

  1. Cancel any background drain task
  2. Stop events tee, flush audit
  3. store.complete(session_id, audit=audit) — DB row updated with terminal status
  4. Memory merger runs — diff workspace vs manifest, append entries to canonical
  5. proposals.upload_workspace_proposals(workspace) — to GCS
  6. shutil.rmtree(workspace, ignore_errors=True) — sandbox is gone
  7. Lease released

After this, the session is terminated. The DB row stays (with audit events on extra_data + transcript path on GCS). The workspace is gone forever.

8 abandoned TTL sweep on long-parked cold sessions; memory merge skipped

If a session sits in waiting_cold past 7 days:

  1. Cron sweep finds it
  2. Audit event session_abandoned
  3. Workspace rmtree
  4. Session row → abandoned

This is the only state transition that doesn't go through finalize_session. Memory merge does NOT run for abandoned sessions — anything the operator wrote during the session is lost. By design: 7-day-old in-flight notes are stale anyway, and avoiding the merge prevents a "ghost author" entry from appearing in canonical memory.

Why the sandbox outlives its subprocesses

The most important property of the lifecycle: the sandbox persists across the active ↔ waiting_cold transition. The subprocess dies. The Python ClaudeSDKClient object is gone. But the workspace dir on shared FS, with all its skills/agents/memories/proposals, stays.

That's why cold-revive on a different replica works. That's why take-over to a different machine works. That's why the 1:1 session-to-sandbox cardinality — combined with shared FS — gives us multi-replica deployments and machine-portable sessions for free.

Memory model — two-tier with pre-task fetch / post-task update

Two distinct physical paths, both on the same shared FS mount:

/var/lib/carelink/agi/
├── memory/<org_id>/ # CANONICAL (durable, pharmacy-scoped)
│   ├── org/
│   │   ├── tools/<verb>.md # org-shared, append-only entries
│   │   └── drugs/<ndc>.md
│   └── pharmacy/<P>/
│       ├── patients/<pid>.md # pharmacy PHI scope
│       ├── prescribers/<npi>.md
│       └── insurance/<bin_pcn>.md
│
└── workspace/<org_id>/<P>/<sid>/ # SANDBOX (per-session, ephemeral)
    ├── .memory-manifest.json # what was projected + when
    ├── memories/
    │   ├── tools/<verb>.md # copies of selected canonical keys
    │   ├── drugs/<ndc>.md
    │   ├── patients/<pid>.md
    │   ├── prescribers/<npi>.md
    │   └── insurance/<bin_pcn>.md
    └── ...rest of sandbox (skills, agents, proposals, etc.)

Canonical = the durable, pharmacy-scoped truth

  • Lives across sessions. Survives restarts, deploys, machine failures.
  • Append-only — entries dated and signed
  • Swept by TTL (14 days)
  • The thing that "carries learning" between sessions
  • Per-org partition is structural (different DB-per-org tenants have different FS roots); per-pharmacy partition is policy (within an org)

Projection = the per-session view inside the sandbox

  • Created at session start (pre-task fetch)
  • Merged back at session end (post-task update)
  • Destroyed with the sandbox
  • The session never reads canonical directly during execution — only the projection
Key insight: the session is operating against a frozen snapshot, not a live view. This is what makes concurrent sessions safe — they don't see each other's mid-flight writes. Cross-session learning happens at the merge boundary, not during execution.

What gets projected, by bucket

A mature pharmacy might accumulate 10,000 patient memory files. Copying all of them into every sandbox would mean 100MB+ snapshots and minutes of FS work. So projection is scoped by relevance, not "copy everything in pharmacy scope."

Bucket Volume Projection strategy Why
tools/<verb> Bounded (~30-50 verbs) All eager Small total volume; every expert may need any verb's quirks
drugs/<ndc> Unbounded over time By context manifest — only NDCs the session declares it'll touch Can't project all; orchestrator knows the relevant NDC at session-creation time
patients/<pid> Unbounded By context manifest — usually exactly 1 patient per session Same reason; orchestrator knows the patient
prescribers/<npi> Unbounded By context manifest — 1-2 prescribers per session Same
insurance/<bin_pcn> Bounded by payers ever seen (~100s) By context manifest Could project all but most are irrelevant; manifest is cleaner

Plain copy. ~1-50 KB per memory file × ~5-50 files = a few MB at most per session. Negligible.

What about workflows without context?

Chat workflow doesn't know upfront which NDC or patient the user will ask about. Two options:

  • Project only tools/ (bounded) and let the LLM operate without persistent drug/patient context. Acceptable for ad-hoc chat. v1 default.
  • Lazy-fetch on miss (v2 feature): a Read hook intercepts attempts to read a non-projected memory file, fetches from canonical, copies into workspace. Adds first-read latency but unlocks chat-style sessions.

Memory sync flow — three phases

The full lifecycle of memory state across a session, plus what concurrent sessions look like.

Phase 1 — Pre-task fetch (session start)

  1. Session creation handler invokes the memory selector with (workflow, pharmacy_id, role, context_dict)
  2. Selector walks canonical, reads matching keys
  3. Files copied to <workspace>/memories/<bucket>/<key>.md
  4. Manifest written to <workspace>/.memory-manifest.json with byte ranges + source mtimes + SHA256
  5. Audit event memory_snapshot with manifest summary

Phase 2 — During the session (read/write)

Once projection is done, everything happens locally in the sandbox.

Reads

Experts open memories/<bucket>/<key>.md like any other file. The SDK CLI's Bash + Read tool see whatever's in the workspace. There is no fetch-on-read, no canonical lookup, no special hook (in v1).

If a key wasn't projected, the file simply doesn't exist. The expert gets FileNotFoundError and proceeds without that prior context. Most expert MDs already handle this gracefully ("if you have prior notes... otherwise...").

Writes (within session)

Experts append notes to memory files. Writes go to the workspace, not canonical.

Session 1, Expert intake (T+0:30): writes to workspace_1/memories/patients/pt-9c.md:
  "Patient confirmed DOB transposed (1948 vs 1984). Verified twice."

Session 1, Expert billing (T+2:15): reads workspace_1/memories/patients/pt-9c.md
  → SEES intake's note (intra-session propagation works)

This is the intra-session memory propagation that makes the model useful. Earlier experts in a session set context that later experts in the same session benefit from.

Concurrent sessions don't see each other's writes

This is the session-as-unit-of-consistency property. Within a session = strongly consistent. Across sessions = eventually consistent at session-end merge time.

Phase 3 — Post-task update (merger at session end)

The merger reconciles workspace memories back to canonical.

What the merger does, file by file

for each <bucket>/<key>.md in workspace/memories/:
    if not in manifest:
        # Net-new key created during session
        new_entry = wrap_in_entry(file_content, actor, role, session_id, ts)
        atomic_append(canonical_path, new_entry)
        continue

    snapshot_bytes = manifest[key].snapshot_bytes
    snapshot_sha = manifest[key].snapshot_sha256
    current_bytes = workspace_file.size
    current_sha = sha256(workspace_file[:snapshot_bytes])

    if current_sha != snapshot_sha:
        # Pre-snapshot bytes were modified — VIOLATION of append-only
        log_warning("Append-only violated; ignoring modified pre-snapshot content")
        # Don't propagate — fall through to extract additions only

    if current_bytes > snapshot_bytes:
        added_content = workspace_file[snapshot_bytes:]
        new_entry = wrap_in_entry(added_content, actor, role, session_id, ts)
        atomic_append(canonical_path, new_entry)

    # If current_bytes == snapshot_bytes: no additions, skip

The append-only invariant is enforced at the merger, not at the OS level. If an expert tried to delete or modify pre-existing canonical content (via Edit tool), the merger silently drops the change. Only the post-snapshot additions get propagated.

The wrapping

Every appended block is wrapped with in-band metadata:

<!-- entry: actor=alice@orgX.io role=technician session=8f3a... ts=2026-04-28T14:14:30Z -->
Patient confirmed DOB transposed (1948 vs 1984). Verified twice.
<!-- /entry -->

Atomic append (the production-critical part)

Plain open(path, 'a') is not safe under concurrent writers — partial content can be visible to readers mid-append.

def atomic_append(canonical_path, new_entry):
    # Use a per-file fcntl lock
    with open(canonical_path, 'r+b') as f:
        fcntl.flock(f, fcntl.LOCK_EX)
        f.seek(0, os.SEEK_END)
        f.write(new_entry.encode())
        f.flush()
        os.fsync(f.fileno())
        fcntl.flock(f, fcntl.LOCK_UN)

This serializes concurrent merges to the same file. Sub-millisecond contention.

For files that don't exist (net-new keys created during session), use atomic create:

def atomic_create(canonical_path, content):
    tmp_path = canonical_path + ".tmp." + uuid()
    with open(tmp_path, 'w') as f:
        f.write(content)
        f.flush()
        os.fsync(f.fileno())
    os.rename(tmp_path, canonical_path)  # atomic on POSIX

Concurrent sessions — the model in one picture

Crash during merge — recovery

If the merger crashes mid-process:

  • Some files have been merged to canonical (those entries are committed)
  • Some files have not
  • The session's extra_data["merge_completed"] is False

Recovery: cron sweeper finds (status=finalizing, merge_completed=false, finalized_at < now - 5min). Marks session as merge_failed. Workspace stays for forensic review (don't rmtree). Operations team or a re-run script can inspect and complete the merge if desired.

In the worst case, some session writes are lost. Better than corrupting canonical.

Per backend replica (realistic)

Realistic peak utilization across a fleet: 10-20% of users active at any moment. For a 4-replica deployment serving ~100 mid-sized orgs:

Bottlenecks at scale

1. Skills copy on session start (first to hurt)

At ~150-300ms per session for the skills tree, if creation rate hits ~5-10 sessions/sec sustained, you'll burn close to a full CPU just on shutil.copytree. This is the single biggest perf cliff before everything else.

  • Mitigation: shared symlink tree (the A3 hybrid we deferred). Skills live once at <workspace_base>/<org>/_skills/; each session symlinks .claude/skills/ to it.

2. Resident RAM per replica

At ~150MB per orchestrator × 300 active = ~45GB RAM per replica.

  • Mitigation: cap max_turns per workflow; cold-park aggressively (subprocess dies, state survives in DB + workspace); horizontal scale; replica sizing 32-64GB.

3. Subprocess count + fd limits

OS-level limits on processes per user and open fds. A replica running 300 concurrent active sessions ≈ 600 subprocesses + ~30K fds.

  • Mitigation: ulimit -n 65536 at deploy time; cgroup limits per backend container; cold-park turns "many concurrent active" into "many concurrent waiting DB rows."

4. Cold-parked sandbox accumulation on shared FS

At ~100MB each × 5,000 cold sessions across the fleet = 500GB just for parked workspaces.

  • Mitigation: hard TTL on cold-park (7 days); per-pharmacy quota; monitor + alert.

5. Cold-revive storm

End of pharmacist's lunch — 30 pending approvals processed in 60 seconds → 30 cold-revive Inngest events.

  • Mitigation: Inngest concurrency=[Concurrency(scope="fn", limit=3)] already in place; per-pharmacy concurrency throttle; backpressure visible in UI.

6. Filestore IOPs (shared FS)

GCP Filestore standard tier: ~10K IOPs sustained per TB. 500 sessions/min × 200 IOPs/sandbox-create = 1.6K IOPs/sec sustained.

  • Mitigation: symlink skills tree; per-org Filestore mounts; premium tier for bursty orgs.

Edge cases to plan for

  1. Pharmacy with synchronized lunch break — at 12:30 PM all 12 employees finish their morning Rx work, click "submit for approval" simultaneously. 12 sessions all flip to waiting_cold within 5 minutes. Cold-revive storm at 1:00 PM.
  2. Single user with multiple browser tabs — Alice opens 3 tabs each starting an rx-flow session. 3 sandboxes, 3 leases, all on Laptop 1's machine_id. Lease design handles this fine but the user might expect "continue my last session" — surface a session picker in the UI rather than auto-resuming.
  3. The session that never ends — orchestrator hits some edge case where it loops between two experts indefinitely. max_turns cap saves you, but worth surfacing as an audit event for review.
  4. Workspace mid-state during take-over — Bob takes over Alice's session while an expert is in the middle of a Bash call. Process is killed; tool_result was never returned to orchestrator. Resume needs to roll back to the last committed orchestrator turn (the SDK's resume=session_id does this naturally, but verify under test).
  5. Memory write race within a session under future parallel dispatch — if two experts dispatched in parallel both write to memories/drugs/<ndc>.md, the file gets interleaved bytes. fcntl flock at merge time is fine; in-session mid-flight is a real race. Today sequential, no problem. Future = B3 hybrid.
  6. Cold-revive on a different replica than original — works because shared FS, but the new replica's cl._agi_clients[sid] dict starts empty. Sandbox is on disk; the live ClaudeSDKClient Python object has to be re-created on whichever replica picks up the Inngest event. That's the whole point of the lease design.
  7. Pharmacy onboarding races — admin creates pharmacy → user created → user starts session before the bootstrap step has provisioned the pharmacy memory directory. Bootstrap must be idempotent and synchronous with pharmacy creation, not async-eventual.
  8. Org migration / DB cutover — moving from shared-DB to DB-per-org. Existing sessions have to either be drained (let them finish) or migrated (move row + workspace to new DB / FS partition). Drain is cleaner. Plan for a freeze window.

Multi-replica failure scenarios

Replica holding the lease crashes (deploy, OOM, segfault)

  1. Replica X is hosting session_id S with active lease
  2. Replica X dies. No graceful drain.
  3. Heartbeat stops. After 60s, lease_expires_at < now()
  4. Lease sweeper (cron, every 30s) reclaims by setting lease_holder_* to NULL
  5. Session row remains valid; workspace remains on shared FS
  6. Next user-facing event acquires lease cleanly
  7. Possibly: in-flight assistant turn was lost. SDK's resume=session_id recovers up to the last committed turn in the JSONL transcript.

Backend deploys mid-session (rolling restart)

Same as above, but graceful — replica drains active sessions before exiting:

  1. Replica X gets SIGTERM
  2. Each active session: cancel drain task, finalize cleanly OR cold-park if mid-turn
  3. If cold-parked: state preserved on disk + DB; new replica picks up on next request
  4. If finalized: session terminates cleanly; sandbox rmtree'd

Inngest worker picks up cold-revive on a replica that's never seen this session

  1. Inngest event agi/cold_approval_requested {org_id, session_id, approval_id} fires
  2. Any backend replica handles it (Inngest distributes via concurrency=3 limit)
  3. Replica resolves the org's DB connection → reads session row → reads workspace path
  4. Workspace is on shared FS, so replica can read it
  5. Acquires lease, spawns SDK client with resume, drains

The sandbox is the source of truth. The replica is interchangeable.

Future enhancements that don't change the core model

  • Memory size cap per key — refuse appends past 50KB; oldest entries auto-pruned
  • Per-pharmacy memory quota — total memory bytes per pharmacy capped; alert at 80%
  • Manifest entry-level redaction — selector includes file but redacts certain entries by role
  • Per-bucket TTL overrides — different buckets might have different retention (drugs longer than insurance, e.g.)
  • Audit-only read mode — for compliance reviews; reads canonical without spawning a session

What we explicitly chose NOT to ship in v1

  • Lazy fetch on miss — chat workflow projects only tools/ for now
  • Memory pinning — all entries TTL at 14 days uniformly
  • Cross-org platform priors — each org learns independently (option A)
  • Per-expert sandbox subdirs (B3) — sequential dispatch only; design v1 to make B3 a drop-in
  • Sandbox pool / pre-warmed cache (A2/A3) — reserved for skills-copy bottleneck
  • Per-bucket TTL overrides — uniform 14d
  • Memory entry promotion — no escape hatch from TTL today

The sandbox model in one sentence

The sandbox is the unit of memory consistency, the durable backing for cold-revive, and the perimeter for in-session agent collaboration. 1:1 with session, N:1 with agents, ephemeral by design — with canonical memory above as the long-term cross-session learning layer.
