Executive summary

In one paragraph

An engineer on a parallel branch built a working per-session sandbox primitive: each chat session gets its own isolated working directory backed by Google Cloud Storage, with optional fast-read access via gcsfuse. The library half (~1,000 lines of code + 666 lines of tests) is high quality and drops cleanly into our current PR 5 plan with a single targeted refactor. The consumer half (a parallel AI service named "carelink-agi") conflicts with our locked architectural decision and should NOT be pulled in. The workspaces/ library

In one paragraph

This is the part worth taking. Four files, ~1,000 lines of code with virtually no CareLink-specific imports, plus a 666-line test suite. The library packages "give every chat session its own isolated working directory" into a clean Python interface: a manager class that creates the directory, copies in skills/memories/transcripts at start, snapshots changes to GCS at every turn, and reaps stale dirs on a timer. The design is solid; a few rough edges around error handling need fixing before merge.

§ Public API surface

Exported from __init__.py:21-31:

  • HydrateOptions — frozen dataclass: skills, agents, memories, session_state. Defaults pull all skills, no agents, all memories, prior session state (manager.py:100-117).
  • WorkspaceManager — owner of the per-session round trip. Methods: hydrate, checkpoint, finalize, sweep_cold, layout_for, session_exists_locally, fuse_available, write_restish_config. Constructor: (org_id, local_root, gcs_client, fuse_root) (manager.py:131-178).
  • WorkspaceManifest — dataclass; persisted as sessions/<sid>/manifest.json in GCS (manifest.py:52-107).
  • WorkspaceFile — manifest entry: path, sha256, size, mtime, classification, blob_path (manifest.py:20-49).
  • SandboxLayout — frozen path bundle: skills_dir, agents_dir, memories_dir, artifacts_dir, proposals_dir, scratch_dir, restish_config_linux, restish_config_macos, transcript_file() (schemas.py:74-139).

§ Lease semantics — or lack thereof

There is no lease in any meaningful distributed sense. What exists:

self._locks: dict[str, asyncio.Lock]      # manager.py:172
def _lock_for(session_id): ...            # manager.py:704-709

In-process only. Two replicas of the FastAPI service or one FastAPI + one Inngest worker both holding session_id="X" will both hydrate, both checkpoint. Last writer wins on every blob. No fencing token, no GCS object lease, no Postgres row lock. _locks only shrinks on finalize — so swept sessions leak lock entries until process restart (minor memory leak in long-running pods).

§ GCS+FUSE interaction

  • Reads: prefer FUSE if fuse_root.is_dir(); else GCS API (manager.py:400-407). Per-call check, no caching of "is FUSE alive". A degraded-but-mounted gcsfuse silently produces empty hydrates.
  • Writes: always GCSClient.upload_blob — explicitly bypassing FUSE because of "gcsfuse upload-on-close quirks" (module docstring manager.py:18-21).
  • Mount: assumed pre-mounted by sidecar/systemd. Manager never shells out to gcsfuse.
  • Hash-diff before copy (manager.py:423-427): SHA256 short-circuit on identical files. Reads every byte of every file twice (source hash + target hash) on each idempotent re-hydrate — measurable per-session cold-start cost.
  • Cross-session memory propagation depends on gcsfuse stat-cache TTL (production: 10 min). The test at test_workspace_manager.py:621-647 simulates this manually by writing the GCS blob into fuse_root after checkpoint. Real gcsfuse with default TTL = up to 10 min lag.

§ Manifest semantics

  • Stored as gs://<bucket>/sessions/<session_id>/manifest.json only — not on local disk (manager.py:641-642).
  • Per-checkpoint flow: load manifest → walk sandbox → upload changed files → upload new manifest LAST. The "manifest last" ordering is the only consistency guarantee (manager.py:471-550).
  • bump() runs before manifest upload (manager.py:521); on upload failure, the in-memory manifest still advertises version=N+1 while bucket stays at version=N. Misleading return value.
  • _load_manifest catches all exceptions and returns WorkspaceManifest.empty(...) (manager.py:644-652, manifest.py:106-107). Cannot distinguish 404 (legit empty) from 503 (transient → should retry, not nuke). Data loss

§ Idempotency & retry guards

Extensive (15 sites identified). Everything is logged-and-skipped, never retried with backoff. The "next checkpoint will retry" pattern is consistent for per-file uploads and the manifest itself. No if-generation-match on manifest upload — overwrite is unconditional. No timeout / deadline on any GCS call. A hung gcsfuse mount blocks the lock indefinitely.

App integration (carelink-agi/app/*)

In one paragraph

The library is wired into a separate FastAPI service called "carelink-agi" — different from our current backend. Sessions are sandboxed per-request with a clever trick: a fresh job hydrates under its job_id because the SDK hasn't issued a session_id yet, then mirrors itself once the SDK identifies the session. Skills moved entirely to GCS (no in-process cache; every GET /skills re-lists the bucket). A new background sweeper reaps stale local sandboxes on a 1-hour timer. There's no startup health check for GCS auth — failures only surface on the first request.

§ Sandbox-key bifurcation

The trickiest mechanic in the diff (jobs.py:498-508,746-748,781-803; chat.py:151-154):

sandbox_key = job.session_id or job.job_id   # fresh job → job_id; on resume → session_id
hydrate under sandbox_key
... SDK runs ...
SDK init emits session_id
job.session_id = session_id
end_of_turn: if job_id != session_id:
    shutil.copytree(<job_id>/, <session_id>/)   # MIRROR, not move
    checkpoint under session_id

Both <job_id>/ and <session_id>/ live in /tmp until the sweeper reaps the orphan. Sharp edge Orphaned <job_id>/ is protected by the sweeper (_build_active_predicate at sweeper.py:36-53 checks both keys), but only via JobManager — chat sessions are NOT tracked there.

§ Sweeper analysis

  • Lifecycle: single asyncio task; while-loop with asyncio.sleep(interval_seconds); default interval=1h, max_age=24h (main.py:89-96).
  • What it sweeps: only /tmp/agi-sandboxes/<sid>/ on local FS. Never the bucket (explicit invariant sweeper.py:5-6).
  • "Is hot" predicate: _build_active_predicate (sweeper.py:36-53) walks jobs_manager.list() for jobs in pending|running|waiting_hot|waiting_cold matching either job_id or session_id.
  • Sharp edge Chat sessions are NOT tracked in JobManager. A long-running chat stream that's actively streaming tokens has no entry in jobs_manager.list(). Protection is implicit via mtime updates (the SDK writes during streaming).
  • Failure handling: try/except Exception at sweeper.py:91-92 logs and continues. CancelledError re-raised. Net: very robust.

§ Skills loading: old vs new

Old: SkillsRegistry(agent_workspace=…) — filesystem-backed. New: SkillsRegistry(gcs_client=…, templates_prefix="templates/skills") — every CRUD method is a direct GCS call (skills.py:99-329).

  • No in-process cache. Every GET /skills re-lists the bucket.
  • Per-session local copies. Each session's sandbox gets its own shutil.copy2'd copy of the templates.
  • Sharp edge Checkpoint walker REFUSES to push .claude/skills/ back (manager.py:634-635). An agent that edits a skill via Bash sees the write succeed locally; the change vanishes on next hydrate of any session. Skill mutation must go through the HTTP CRUD endpoints. Intentional but a documented footgun.

The deployment is dev-VM-only — not Cloud Run-ready. § gcsfuse flags

Identical between manual script and systemd unit:

  • --implicit-dirs — required because the bucket has no placeholder objects
  • --file-mode=0644 / --dir-mode=0755 — readable but not world-writable
  • --stat-cache-ttl=10m / --type-cache-ttl=10m — this is the cross-session memory propagation lag ceiling. A new memory written by Session A becomes visible to Session B no sooner than the cache TTL expires.
  • No write caching, no atomic-write mode, no --max-conns-per-host tuning.

§ Benchmark methodology

scripts/bench-sandbox.py runs 6 timed steps via time.perf_counter(): cold hydrate, warm hydrate, single-artifact checkpoint, 100-bulk checkpoint, transcript checkpoint, idempotent re-checkpoint.

  • No repeats, no median/p95, no warm/cold separation in real mode, no GCS-roundtrip isolation, no concurrency.
  • Budgets in the docstring are aspirational — script doesn't assert them.
  • --fake mode uses an in-process dict (zero IO), so its numbers don't predict real-mode behavior.
  • No actual measured numbers in the runbook.

Sharp edges 10 findings

In one paragraph

Ten distinct issues, ranked by data-loss risk. The top four (S1–S4) must be fixed before any production use of the cherry-picked library; the remaining six (S5–S10) are operability/security concerns that can be follow-ups. Each card has a click-to-expand fix sketch.

Filter:
S1 _load_manifest swallow-and-empty over real manifest Data loss

workspaces/manager.py:644-652. A bare except Exception → empty() cannot distinguish 404 (legit empty) from 503 (transient → should retry, not nuke). A transient GCS read failure during _checkpoint will produce an empty manifest, which is then uploaded over the real one via the unconditional manifest write (manager.py:537-542).

Recoverable only if GCS object versioning is enabled — which the bucket setup script does enable, but with no lifecycle, so the recovery is "go find the noncurrent version".

Fix: Distinguish 404 from 5xx in _load_manifest; on 5xx, raise + log + abort the checkpoint (let the next checkpoint retry).
S2 No if-generation-match on manifest upload Data loss

manager.py:537-542. Two concurrent checkpoints from different pods/replicas both download the same manifest, both bump to v(N+1), both upload. Last writer wins on files; the other's blob uploads remain orphaned in the bucket. The per-session asyncio.Lock does NOT span processes.

Fix: Pass if_generation_match=existing.generation on upload_blob; on precondition-failed, re-load and retry-with-merge or fail loudly.
S3 finalize() race window Data loss

manager.py:251-256. Lock released between checkpoint (which acquires/releases internally) and rmtree. A concurrent hydrate(session_id="X") could land between, creating a sandbox that gets rmtree'd.

Fix: Hold the lock continuously across both operations; or have finalize re-check session_exists_locally before rmtree.
S4 WorkspaceManager is one-org-per-process High

manager.py:147, carelink-agi/app/main.py:67. org_id is baked at construction time. Cannot serve multiple orgs from one carelink-agi process today. Incompatible with the locked 5-scope hierarchy (Platform→Org→Pharmacy→User→Session).

Fix (for cherry-pick): Replace org_id: str constructor param with a scope_resolver: Callable[[Session], MemoryScope] that produces the path bundle dynamically. ~4 hours.
S5 Memory cross-session propagation depends on gcsfuse cache TTL Medium

Test at test_workspace_manager.py:621-647 simulates this manually by writing the GCS blob into fuse_root after checkpoint. Production uses --stat-cache-ttl=10m (mount-sandbox-fuse.sh:53-54). Real cross-session memory propagation can lag up to 10 minutes. Not covered by tests; not flagged in the runbook.

Fix: Document the cache TTL in the runbook; for memory-merger logic in PR 6, prefer the GCS API path over FUSE for read-after-write within a single session boundary; consider lower TTL for memory paths specifically.
S6 Skills written by an agent are silently dropped Medium

manager.py:634-637 — _classify returns (None, "") for .claude/skills/** and .claude/agents/**. An agent that edits a skill via Bash inside its sandbox sees the write succeed; the content vanishes on next hydrate.

Fix: This is intentional (skills must go through SkillsRegistry). Document for agent authors. Consider a checkpoint-time WARNING log when checkpoint sees touched files in those paths.
S7 No startup health gate Medium

carelink-agi/app/main.py:54 constructs GCSClient(settings.sandbox_bucket) without probing. skills_registry.ensure_root() is now a no-op. First failure surfaces at first request.

Fix: Probe a healthcheck/probe.txt upload+delete during lifespan startup (mirroring what setup-sandbox-bucket.sh:18-30 already does in shell). Fail fast with a clear log line.
S8 Sweeper is JobManager-only; chat sessions protected only by 24h TTL Medium

sweeper.py:36-53. _build_active_predicate checks JobManager.list(); chat sessions aren't tracked there. A long-running chat with reduced TTL (e.g. for tests) could in principle be reaped mid-stream.

Fix: Register live chat sessions in a process-wide registry that the sweeper consults; or update mtime continuously during a chat turn so the TTL effectively follows liveness.
S9 systemd unit + IaC gaps High

scripts/sandbox-fuse.service. No hardening; hardcoded User=sangeet + HOME=/home/sangeet; bucket creation, IAM, gcsfuse install, kernel module, VM provisioning all out-of-band; manual ADC user-credential rotation = service degradation until human re-auths.

Fix: ~30-min hardening pass (PrivateTmp, NoNewPrivileges, ProtectSystem=strict, ProtectHome=tmpfs, capability bounds). Templated user/HOME via Environment=. Service-account migration via workload identity for any non-dev environment. Terraform/Pulumi modules for bucket + IAM + lifecycle policy.
S10 Versioning on + no lifecycle = unbounded cost growth Medium

scripts/setup-sandbox-bucket.sh:35. No max-versions, no age cap. Likely the dominant long-term cost driver.

Fix: GCS object lifecycle policy: NumNewerVersions=N (e.g. 10) for memory and transcript paths; explicit retention only on session manifests if needed for audit.
