Vercel: Single-user, no tenancy at all

Vercel

Their apps/web/lib/db/schema.ts has 14+ tables. Every domain row is FK'd directly to users.id. No organizations, no teams, no tenant_id. By design — the app is an open-source clone-template for an individual developer's "personal Devin." For multi-tenant retrofit: nothing reusable. Forking would mean rebuilding the schema.

Implication for us: not a forkable foundation. The patterns inside (lease, subagent dispatch) are reusable; the chassis isn't.

CareLink today: Single-org singleton, deployed once

CareLink now

CareLink.__init__ takes a config with org_id: str. 26+ services constructed with that org_id baked in (self.db, self._authz, self._config.org_id at every constructor). Today's deploy is one process, one org_id, one DB.

This works fine because we run ONE org today (Demo Pharmacy / Asepha). The architecture is implicitly single-tenant — never had to think about it.

Implication: the per-instance target preserves this. Every singleton stays single-org. No 26-service refactor needed.

CareLink target: Per-instance tenancy

CareLink target

Each org gets their own deployment: own backend process, own DB, own FS root, own GCS bucket, own Inngest worker, own Permit.io tenant. The CareLink Python package + the FastAPI backend ship as a deployable unit; provisioning a new org = provisioning a new instance.

Inside an instance: pharmacies are first-class entities in the DB. Users (employees) are members of one or more pharmacies via PharmacyStaff. PHI boundary = pharmacy. Cross-org = physically impossible (no shared anything).

What this resolves: the singleton refactor concern (G1, C1) is RESOLVED — every instance is single-org by construction. What this introduces: deployment provisioning, rolling updates across N instances, per-org Permit.io tenant management.

Vercel: task tool + AI SDK

Vercel

packages/agent/tools/task.ts dispatches to SUBAGENT_REGISTRY (3 entries: explorer, executor, design). Each subagent is its own ToolLoopAgent with its own toolset. Parent's history isolated from child's tool calls — only summary returns. Uses AI SDK's experimental_context for context propagation.

Implication: the field has converged on this pattern. Confirmation our invoke_expert shape is correct, not novel.

CareLink today: invoke_expert MCP + Pydantic

CareLink now

tools.py:682-792 invoke_expert. In-process MCP tool. Spawns expert subprocess via _spawn_expert_agent with the expert MD body as system prompt. Returns Pydantic-validated BaseExpertOutput on success, typed envelope {"status": "failed", ...} on second drift. Single-shot retry with schema-injected nudge.

Phase D spike (2026-04-27) concluded native agents= can't replace this — typed envelope substitution requires MCP, not built-in Agent tool.

Implication: stays canonical. Same shape as Vercel's, with stronger typing.

CareLink target: invoke_expert preserved

CareLink target

Phase D verdict means we keep invoke_expert. Future: workflow YAML declares which experts the orchestrator can invoke. The orchestrator's invoke_expert enum is filtered at session creation by InvokerContext (L3 access layer). MCP boundary enforces.

Vercel: No persistent memory

Vercel

Sandbox filesystem IS the memory. Per-session state in chat_messages (JSONB), sessions.cachedDiff (JSONB), sandbox snapshots. Skills are static MD prompts (closer to slash-commands than memory). No vector store, no episodic.

Implication: their model is "the repo is your memory." Useless for our cross-session pharmacy-operational learning. Don't look here.

CareLink today: Aspirational memories

CareLink now

Every expert MD references memories/<bucket>/<key>.md. service.py creates the dirs at session start, populates them as workspace, then rmtree's at finalize_session. No persistence. The "cross-run learning" is prose — operationally false.

Implication: implementing a real memory layer is the largest single load-bearing piece of the multi-tenancy work.

CareLink target: Append-only with TTL

CareLink target

Per-pharmacy memories (patients, prescribers, insurance) + per-org memories (tools, drugs). Append-only entries with comment-fence signing. 14d TTL via Inngest cron. Read-side projection at session start (manifest = audit record).

Substrate undecided:

  • Filesystem (current design): open(path, 'a') on shared FS. Closer to today's MD prose feel. Concurrency on NFS/Filestore is unreliable; scale concerns at 1000s of files per pharmacy.
  • Postgres rows (alternative): one row per (pharmacy_id, bucket, key, entry). ACID, cheap TTL via DELETE. LLMs read via restish carelink list-memories. Less natural; more durable.

Per-instance tenancy doesn't change the substrate decision — it just means each org's substrate is isolated.

Vercel: Inline question, not durable park

Vercel

ask_user_question is a client-side tool with no execute. Workflow yields with state: "input-available". UI renders form. Answer comes back as next user msg in the same chat. Cross-machine: yes, via Vercel Workflow durability. But it's chat-bound interactivity, not "park for 3 days waiting on a fax response."

Reusable shape: their questions[] schema is a clean inline-HITL UX standard. Worth borrowing for fast in-chat HITL inside the AI assistant workflow (separate from durable approvals).

CareLink today: Two gates + Inngest cold-revive

CareLink now

UserQuestionGate (mints question_id on arm) + ToolApprovalGate (parks Bash). Hot path: 60s in-memory wait. Cold path: subprocess exits, status flips to waiting_cold, Inngest func_cold_approval_drain resumes when decision arrives. Battle-tested (cancel-engine sprint shipped 2 weeks ago).

Today: queue is session-scoped — answer routes back to the originating session by ID. Multi-machine handoff works through Inngest re-spawning.

CareLink target: Pharmacy-scoped queue

CareLink target

GET /pharmacies/{pid}/approvals → list of pending approvals at that pharmacy. Any qualified user (role ∩ required_roles ≠ ∅) can decide. Cross-machine "for free" — Bob's laptop sees Alice's approval row in the queue. Both initiator and approver audited.

Implementation: extends today's gates. Add pharmacy_id column to agi_pending_approvals. New router endpoint. Race condition handled via SELECT FOR UPDATE + status check.

Vercel: TS directive-based, no registry

Vercel

"use workflow" + "use step" directives at function entry. Vercel Workflow SDK statically analyzes and generates a manifest. Two workflows hardcoded: runAgentWorkflow (chat) and sandboxLifecycleWorkflow (hibernation). Adding a new one = TS file + directive. No version field, no rollback.

Implication: opposite direction from us. Their build-time codegen is platform-bound. Not portable.

CareLink today: Hardcoded rx-flow + 3 TS prompts

CareLink now

rx-flow is the only "workflow." Its orchestrator prompt lives in system-prompt-templates.ts (1,225 LoC, 3 orchestrators after cancel-engine sprint: newrx-orchestrator, rx-renewal-response-orchestrator, rx-cancel-rx-orchestrator). Backend at routers/agi.py:208 plumbs body.system_prompt through.

Schema changes to BaseExpertOutput can silently desync from what the frontend tells the LLM.

CareLink target: YAML manifests + registry + selector

CareLink target

Workflow becomes a top-level entity: id, description, orchestrator_prompt, experts[], required_roles, memory_scope, hitl_approval_required_for_verbs. Three workflows planned: rx-flow, claims-flow (future), chat (orchestrator-only).

Adding a workflow = data, not code. Drop YAML + orchestrator MD, reference existing experts, org admin enables it.

Open question: YAML vs Python WorkflowSpec dataclass. YAML wins on "non-engineers can edit" (overstated for engineering-grade workflows). Dataclass wins on IDE help, type safety, no new tooling. If YAML, mirror Vercel's SKILL.md frontmatter shape for ecosystem compatibility (~50 LoC of Pydantic).

Vercel: userId equality only

Vercel

chat.ts: if (sessionRecord.userId !== params.userId) throw new Error("Unauthorized"). That's the entire model. Plus users.isAdmin for an admin panel.

Implication: wrong-fit by 6 layers. Don't reference for access control.

CareLink today: Permit.io at backend, not at AGI

CareLink now

PermitClient.check() exists at integrations/permit/client.py:24. Role permissions cached. get_permissions_matrix exposed for admin UI. Used at backend endpoints (e.g., cl.patients.create checks).

But: AGI doesn't query Permit.io for verb projection. The orchestrator can't fail-fast deny unauthorized verbs — that gate doesn't exist. Backend RBAC is the ONLY layer today.

CareLink target: 6-layer projection

CareLink target

L0 (org reachable, structural via per-instance) + L1 (JWT validated) + L2 (workflow startable) + L3 (expert invocable, filtered at MCP boundary) + L4 (verb permitted, PreToolUse hook) + L5 (approver authorized) + L6 (Backend RBAC, authoritative).

InvokerContext projected from Permit.io once at session start. Cached for session lifetime. Permit.io webhook → invalidate. Layers 0-5 fail-fast advisory; L6 is single source of truth.

Per-instance simplifies: L0 = "is this user reaching the right instance?" = handled by subdomain routing at the load balancer, not by application code.

POC-C validates this entire surface.

Vercel: claimLifecycleLease (reference)

Vercel

apps/web/app/workflows/sandbox-lifecycle.ts::claimLifecycleLease. ~30 LoC. Postgres compare-and-swap: write lifecycleRunId to session row, re-read to verify ownership. Take-over via same CAS. Clean reference; not portable (TS/Drizzle), but the shape is what we want.

Direct reuse: read this before implementing sessions/lease.py.

CareLink today: In-memory dict (process-local)

CareLink now

cl._agi_clients[session_id] = client at routers/agi.py:257-259. Python dict on the CareLink singleton. Today this works because we run a single uvicorn. Two replicas behind a load balancer would break sessions immediately.

Implication: safe for now. Per-instance + single-replica-per-instance also keeps this safe. The lease is only required when an instance scales horizontally.

CareLink target: DB-backed lease + heartbeat

CareLink target

15s heartbeat extends, 60s expiry. Take-over via force-acquire. Lease columns on agi_sessions: holder_machine_id, holder_actor_id, expires_at.

Per-instance reframes priority: if an instance is single-replica (most small orgs), the lease is unnecessary. Defer this until an instance actually scales horizontally. Today the in-memory dict is fine per-instance.

Vercel: Vercel Workflow (proprietary)

Vercel

Vercel Workflow SDK provides workflow-step-level checkpointing. If a worker dies mid-stream, the run resumes. Platform-bound.

Implication: their durability story is locked to Vercel platform. Not portable. We have Inngest for the same job — both ours and theirs converged on "durable function runtime" but different vendors.

CareLink today: Inngest cold-revive

CareLink now

integrations/inngest/agi_cold_revive.py (887 LoC). Two functions: func_cold_revive_drain (HITL question answers) + func_cold_approval_drain (approval decisions). retries=2, per-function concurrency, timeouts. Battle-tested by the cancel-engine sprint.

Plus the new sessions/sweeper.py (171 LoC, shipped 2026-04-28) for 24h chat session cleanup. Pattern is reusable for the 14d memory TTL sweeper.

CareLink target: Same Inngest pattern

CareLink target

Inngest stays. Add memory TTL sweeper cron (every 6h, walks each pharmacy's memory tree). Per-instance: each org's Inngest worker only sees that org's events. No cross-instance dispatch needed.

CareLink target

Vercel: No typed envelopes

Vercel

Each subagent ends with an XML output template. Orchestrator parses tags textually at runtime. When a subagent forgets to close a tag or renames a field, nothing fails fast. They accept this; their domain (code generation) is loss-tolerant.

Pydantic only at HTTP boundary, not between agents.

CareLink today: Pydantic at every boundary

CareLink now

HITL routing depends on status == "hitl_required" + human_questions shape. Drift catches at the boundary.

CareLink target: Same convention

CareLink target

Separate vercel-labs/skills repo + npx skills CLI + a "skills ecosystem" Vercel is pushing. If we adopt this same frontmatter shape on our workflow manifests, we get free interop with the broader skills ecosystem. Total port: ~50 LoC of Pydantic.

What converged independently (the "we're not weird" finding)

  • Subagent dispatch as a tool call — Vercel's task, ours invoke_expert. Same shape, independent arrival.
  • Subagent context isolation — parent's history doesn't include child's tool_use blocks; only the summary returns. Both implementations.
  • Lease via DB compare-and-swap — their claimLifecycleLease is the same shape we proposed. ~30 LoC reference.
  • Skill manifest format — both have YAML frontmatter on MD files for capability declarations.
  • Durability via external workflow runtime — them: Vercel Workflow. Us: Inngest. Same pattern, different vendor.

Where they have nothing to teach us

  • Multi-tenancy — single-user-only.
  • Cross-session memory — sandbox FS only.
  • Access control — userId equality.
  • Workflow definition format — TS directives + build-time codegen.
  • HITL durability — chat-bound only.

Each piece expands here with its purpose and what's per-instance vs shared.

CareLink Python package — shared, sealed

The codebase. Same git ref, same Docker image, deployed per-org. No per-org forks.

Per-org configuration is environment + DB + Permit.io tenant — never code.

Expert MD catalog — shared

8 expert MDs (intake, data-entry, clinical-checklist, dur, billing, fulfillment, shipping, rejection-classifier) ship inside the package. All instances run the same set.

Org-specific behavior comes from workflow YAML overrides (disable an expert; raise approval thresholds), not from forking the MDs.

Skill library — shared

SKILL.md files inside the package. Bundled at deploy time. Same versions everywhere.

Workflow templates — shared

rx-flow, claims-flow (future), chat. YAML manifests OR Python dataclasses, decided in POC-D. Per-instance overrides apply at session creation, not at the manifest level.

Permission templates — shared

Default role × verb policy ships with the package. Org admins can override per their Permit.io tenant.

Each instance's Permit.io tenant is initialized from the templates at provisioning.

FastAPI backend — per-instance

Single uvicorn process. org_id baked at startup from environment. No multi-org routing inside; subdomain routing happens at the load balancer.

Existing code unchanged. The 26+ services constructed with config.org_id stay exactly as they are.

AGI runtime — per-instance

Orchestrator + experts spawn from the same singleton. Sessions, workspaces, and memory all live within this instance's storage.

Postgres DB — per-instance, isolated

orgx_db. AGI tables (agi_sessions, agi_pending_approvals, audit log) live alongside core tables (users, pharmacies, patients, etc.).

Cross-org SQL injection physically impossible — there's no connection to another org's DB.

Alembic migrations: per-instance, run during deploy. N migrations across N instances is just standard rolling deploy.

Filesystem — per-instance

Workspace tmpdirs (today: /tmp/agi-workspace/, target: /var/lib/carelink/workspace/) + canonical memory store (target: /var/lib/carelink/memory/).

Per-instance means no shared FS mount required. Big simplification: the design's "shared FS for cross-replica cold-revive" is unnecessary if the instance is single-replica.

Inngest worker — per-instance

Each instance's Inngest worker only sees that instance's events. Per-org event isolation = no cross-org event leakage.

Adds one TTL sweeper cron for the 14d memory expiration (when memory layer ships).

Permit.io tenant — per-instance

Each instance has its own Permit.io tenant key. Roles + permissions managed per-org.

Provisioning a new org instance = provisioning a new Permit.io tenant + seeding from platform templates.

Org Y backend — fully isolated

Same code as Org X. Different process, different DB, different env. Bug in Org X's deploy doesn't affect Org Y.

Org Y AGI — fully isolated

Cannot see Org X's sessions, memories, audit. Period.

Org Y DB — fully isolated

Separate Postgres connection. Cross-org analytics requires explicit ETL pipelines (future concern, not v1).

Org Y FS — fully isolated

Separate mount or separate disk. No path collision with Org X.

Org Y Inngest — fully isolated

Separate event stream. Cold-revive events for Org X never reach Org Y's worker.

Org Y Permit.io — fully isolated

Different tenant key, different policy. Org Y's admin doesn't see Org X's roles.

Pharmacies inside an instance

Pharmacies are first-class entities in the org's DB. The PHI boundary is policy-enforced (Permit.io rule + read-side memory scoping), not structural.

LayerBoundary typeWhat protects it
OrgStructural (per-instance)Separate process, DB, FS, Inngest, Permit.io tenant. Cross-org physically impossible.
PharmacyPolicyPermit.io pharmacy_id attribute on permissions + read-side memory scoping at session start. Cross-pharmacy within same org requires the user to have a PharmacyStaff entry at both.

Users at multiple pharmacies (your example)

"Carol can be admin@P1 and pharmacist@P2 in Org X." This works because:

  • User row in OrgX's DB. PharmacyStaff many-to-many junction with role-per-pharmacy.
  • At login: Carol picks which pharmacy she's acting at (or default = primary). Session pinned to that pharmacy_id at creation.
  • To switch pharmacies mid-session: re-auth (open question #1). Simple model, avoids context-switch edge cases.
  • Permit.io check: Carol@P1 has admin role → matrix says she can create-pt. Carol@P2 has pharmacist role → matrix says she needs HITL for reverse-claim.

What's still open

  • Provisioning model: k8s namespace, Cloud Run service, Fly app, or Vercel project per org? Each has tradeoffs.
  • Auth-to-instance routing: subdomain (orgx.carelink.com), tenant header, or central auth that redirects? Probably subdomain.
  • Updates / rolling release: N instances need coordinated alembic + code deploy. Standard ops problem.
  • Resource cost per instance: small org = wasted overhead. Multi-tenant lite (e.g. multiple small orgs share a process via DB switching) is a future concern.

Gaps map — 15 findings, with per-instance impact

Each gap from the deep dive, mapped to severity and how the per-instance tenancy clarification affects it. Filter to focus.

G1 CareLink singleton refactor for multi-org critical resolved by per-instance

The singleton bakes org_id at construction across 26+ services. The earlier "DB-per-org with routing middleware" framing implied a months-scale singleton refactor. Per-instance resolves this: each instance is single-org by deployment, the singleton stays exactly as it is.

Resolved by: instance model. Action: update HTML design doc to clarify the intent is "deploy-per-org" not "route-per-org."
G12 POC migration path is missing critical unchanged

The design has 6 sequencing options, none of them is a 1-2 week POC scope. We need to pick a slice that validates the riskiest pattern with the smallest blast radius.

Resolved by: picking POC-C (or POC-A+C+E stack). Tab 3 has the candidates.
G3 invoke_expert filter for L3 (expert invocable) major unchanged

Today invoke_expert accepts ANY expert_id. To enforce L3, we need to either constrain the MCP tool's enum at session-spawn (rebuild MCP per session) or runtime-check inside the tool against InvokerContext. The doc says "filter at MCP boundary" without saying which.

Addressed by: a follow-up POC after C lands. Add the runtime check first (lower risk), then move to enum constraint if the LLM is confused by deny-after-call patterns.
G4 Memory append-only on shared FS has concurrency holes major reduced

open(path, 'a') on NFS/Filestore is not atomic with respect to other writers. flock semantics on managed NFS unreliable. Two simultaneous merges to patient-9c.md can interleave bytes. "Diff vs manifest" silently drops edits to existing entries — LLM doesn't know.

Per-instance reduction: within an instance, contention is bounded by that instance's session count. One small org may never see concurrent writes to the same key. The problem is real but smaller.

Resolved by: choose Postgres substrate (POC-B) instead of FS, OR add per-key file locks via Postgres advisory locks (FS substrate).
G5 HITL approval queue race condition major unchanged

"Any qualified user at this pharmacy can pick up" — but if 2 pharmacists click "approve" simultaneously, behavior is undefined. Need SELECT FOR UPDATE + status check + atomic transition.

Addressed by: POC-E.
G2 JWT trust ordering is wrong in the doc major unchanged

L0 says "structural via DB connection routing." L1 says "WorkOS JWT validation." But you can't trust JWT.org_id until L1 validates the signature. Spec ordering is wrong.

Per-instance update: L0 becomes "is this user reaching the right instance?" — handled by subdomain routing at the load balancer (validation comes from JWT signature too). L1 still must precede any trust of JWT claims.

Resolved by: 1-line spec correction in the design HTML doc.
G7 Workflow versioning + expert MD drift major unchanged

Pin workflow version at session creation — fine. But workflow YAML references expert MDs by ID. If MD content changes mid-session, is the running session reading the old or new MD? Today MDs are loaded at registry init.

Addressed by: snapshot the expert MD content hashes alongside the workflow version pin. Add to POC-D.
G9 Permit.io projection cache invalidation major unchanged

"Cached for session lifetime" — but permission revocation mid-session has zero spec. 4h ceiling for re-auth is mentioned in flow detail, not design rules. Permit.io webhook → invalidate cache → next check refreshes — none of this is wired today.

Addressed by: extension to POC-C (cache invalidation strategy).
G14 6-layer access control latency budget major reduced

L4 (PreToolUse on Bash) fires per tool_use. If it consults Permit.io live, that's network round-trips. Verify projection covers ALL verbs.

Per-instance reduction: Permit.io tenant is per-instance, lower latency baseline.

Resolved by: POC-C's projection covers all verbs cached; L4 reads from cache.
G7L Lease abandonment + multi-replica minor resolved by per-instance

The lease design only matters if an instance scales horizontally. Per-instance with single-replica makes the in-memory dict safe.

Resolved by: defer DB-backed lease until horizontal scaling is needed for a single instance.
G15 Shared FS infrastructure dependency minor resolved by per-instance

The original design said "shared FS for cross-replica cold-revive." Per-instance with single-replica means no shared FS mount needed. Local /var/lib/carelink/ is fine.

Resolved by: don't provision GCP Filestore for v1. Add it only when an instance needs horizontal scaling.
G6 Memory TTL vs pinning interaction minor unchanged

14d TTL is naive. Pinning is "Phase 2." Some memories should be evergreen. If we ship TTL first, we'll lose memories that should never expire.

Addressed by: ship TTL with a "pinned" flag column from day one. Default false. Phase 2 surfaces pinning UX.
G8 Workflow versioning shape unclear minor unchanged

Pin workflow version at session creation. But the version field itself isn't specced in the YAML.

Addressed by: add version: int to YAML schema in POC-D. Bump on breaking change.
G10 Audit log fan-out to global warehouse minor reduced

Per-instance: each org's audit lives in its own DB. HIPAA platform-wide audit becomes ETL from N instances to a global warehouse — cleaner shape.

Addressed by: future ETL pipeline. Each instance's audit is HIPAA-compliant in isolation.
G11 Chat workflow has zero experts (chat.yaml) minor unchanged

Chat orchestrator answers directly. But it declares memory_scope.read: [tools, drugs] — for an orchestrator with no invoke_expert calls, what does it do with these memories? No Pydantic envelope to detect drift either.

Addressed by: defer chat workflow until POC-D rx-flow shape is proven. Chat is a special-case workflow, not the canonical one.
G13 Role changes mid-session minor unchanged

If Carol is demoted from pharmacist to technician at P1 mid-session, current session may continue with cached old permissions.

Addressed by: Permit.io webhook → invalidate cache → next L4 check refreshes. Tied to G9.
