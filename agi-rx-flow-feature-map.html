Recommended sequencing for the remaining 9 experts

Order optimized by leverage of new infrastructure + dependency chains + user-visible value.

Remaining order

  1. doc-intake-assessor (Group C) — unlocks the entire fax/document leg; reuses existing Gemini classifier; first multi-orchestrator template lands; significant business value (PA responses, lab results, demographic updates all flow through this)
  2. rx-flow-renewal-response follow-up: build rx-flow-refill-readiness — completes the Approved branch (currently HITL queue). High volume.
  3. rx-flow-change-response — high-volume NCPDP message; same pattern as renewal-response
  4. rx-flow-prior-auth + 3 new PA skills — biggest unknown. Plan a separate skill-creation phase. Unlocks the rejection-classifier→PA branch + the doc-intake PA Response branch
  5. rx-flow-transfer (one expert, 3 modes) — moderately common; structurally identical to renewal-response
  6. rx-flow-dur-review — gates DUR overrides; depends on whether operator UI is enough or LLM reasoning adds value
  7. rx-flow-history-response + rx-flow-newrx-response-denied — lowest volume; parallelizable
  8. rx-flow-prescriber-outreach — last; depends on PA + DUR-review consumers wanting it

Each future Group A expert is ~3–4 days with one developer using the cancel-rx + renewal-response template. Group B/C are larger (1–2 weeks each for PA + doc-intake due to the new skill suites).

Process for building the next expert

Playbook validated 2026-04-27 with cancel-rx + renewal-response.
  1. Pre-flight (master): read existing patterns (rejection-classifier MD, contracts, new_rx Inngest fn, agi_cold_revive.py bridge, AGIService.create_session signature). Verify the matching skills exist + are current. Confirm the relevant Inngest event is already mapped at inbound.py:23-33.
  2. Build in worktree: spawn an Opus agent with isolation: "worktree" and a brief that pins base SHA, forbids commits/pushes/destructive git, lists exact deliverables (expert MD, contract classes, orchestrator template, Inngest handler, tests, sibling-collision note), and includes a stop-and-surface clause.
  3. Verify + test in parallel (after worktree returns): two opus agents — one read-only verification (contracts, MD frontmatter, sacred-touch, body-vs-frontmatter), one testing (pyflakes + pytest narrow + tsc smoke).
  4. Collate to main: copy worktree's diff; for shared files (contracts append, __init__.py register, templates append), merge manually. Apply any nit fixes inline.
  5. Stage, surface, STOP: never commit on user's behalf. Stage + draft message + wait for explicit git commit from user.
  6. Worktree cleanup: after user confirms commit, git worktree remove --force <path> && git branch -D <branch>.

For 2+ experts in parallel, sibling worktrees deliberately duplicate setup files (data-entry source ext, bridge wire-up). Collator dedups at merge time.

Open questions deferred until consumers force a decision

  1. PA expert split point — at what MD body length does it become unwieldy? Locked decision says "lean out later if needed". Likely trigger: when MD exceeds ~600 lines of body content (preamble inherited).
  2. Refill-readiness orchestrator template — is it triggered ONLY from renewal-response orchestrator, or does it also have its own entry point (IVR patient refill, scheduled refill)? If multiple entry points, dedicated template; if one, can be invoked inline.
  3. Doc-intake bridges to Gemini — fine for parsing; what about classification confidence? Claude classifier as fallback when Gemini confidence < threshold? Cost trade-off.
  4. Transfer expert mode discriminator — input shape: top-level mode: Literal["request", "response", "confirm"] field, or three separate input classes inheriting a common base? Lean toward the former (one input class, branch in body).
  5. Prescriber-outreach durability — IVR autonomous calls are async + long-running. Should the bridge wait? Or return outreach_initiated=true immediately and let the IVR side complete asynchronously? Lean toward the latter.
  6. Linked-records work item type — for cancel-rx we use rx.cancel.linked_records_review. For renewal-response and change-response, do we need parallel types, or one generic rx.linked_records_review with a category?
