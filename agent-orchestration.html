🔒 Harness & Restrictions

Three enforcement layers prevent experts from going rogue or crossing domain boundaries. Every expert operates inside a sandbox defined by its registration.

🔒 Tool Scoping

Each expert gets ONLY its tools via allowed_tools in the agent session options. If it tries to call a tool outside its set, the call is rejected.

Intake Agent:

✗ submit-claim (billing only)
✗ screen-ddi (DUR only)
✗ purchase-label (shipping only)
✗ update-prescription (orchestrator only)

DUR Agent:

✗ create-patient (intake only)
✗ submit-claim (billing only)
✗ create-shipment (shipping only)

Billing Agent:

✗ screen-dur (DUR only)
✗ print-rx-label (fulfillment only)
✗ create-patient (intake only)

📐 Typed Contracts

Every expert declares input/output schemas. The orchestrator validates BEFORE dispatch and AFTER return. Malformed output triggers retry, then escalation.

Intake → must return:

{ patient_id, prescriber_id, product_id,
  insurance_matched, needs_human,
  human_questions[], created_records[] }

DUR → must return:

{ findings[], overall_severity,
  needs_human, human_questions[] }

Billing → must return:

{ status, paid_amount?, copay?,
  reject_codes[]?, needs_human }

🚦 Phase Gating

Experts cannot advance the workflow. Only the orchestrator controls state transitions.

Only orchestrator can call:

✓ update_rx_status(rx_id, status)
✓ emit_report(phase, data)
✓ request_human_review(question)
✓ spawn_sub_workflow(type, context)

When an expert returns needs_human: true, it's the orchestrator that decides whether to escalate, retry with more context, or override.

Experts cannot create work items in arbitrary categories, trigger other workflows, or modify records outside their scope.

How Restrictions Are Enforced
Tool allowlist: The Claude Agent SDK accepts allowed_tools in session options. Tools not in the list are invisible to the agent — it can't even see them in its tool catalog.
Scope guard hook: A PreToolUse hook inspects every tool call before execution. If the agent somehow references a tool outside its allowlist, the hook rejects the call with an error message.
Workspace isolation: Each expert session gets its own workspace directory with only its skill files. No access to other experts' prompts or tools.
Output validation: After the expert returns, the orchestrator validates the output against the declared JSON schema. Schema violations trigger one retry with an error hint, then escalation to the operator.
Turn limits: Each expert has a max turn count. If it exceeds the limit without producing valid output, the session is terminated and the orchestrator escalates.
Option A: Inngest Orchestrates, Claude Reasons

The durable execution engine (Inngest) owns the phase graph. Claude is invoked INSIDE each step as a reasoning engine. The orchestrator is deterministic Python code. Claude handles judgment calls within domain experts.

Key idea: Inngest is the process that never dies — it handles crash recovery, multi-day waits, and state persistence. Claude is ephemeral — each expert invocation is a short session that starts, reasons, returns a result, and ends.

What Lives Where

Clear separation between durable execution and intelligent reasoning
Inngest owns:
• Phase graph (which step runs after which)
• State persistence (rx_context survives crashes)
• Multi-day waits (PA approval, HITL gates)
• Retry logic (if a step fails, memoized retry)
• Event routing (rx/new_rx → this function)
• Concurrency (rate limits, parallel Rx flows)
Claude owns:
• Fuzzy matching (patient matching with confidence)
• Interpretation (sig codes, rejection classification)
• Clinical reasoning (DUR severity assessment)
• Decision support (when results are ambiguous)
• Natural language (drafting outreach messages)
invoke_expert() is the bridge:
• Looks up expert in registry
• Creates a Claude agent session with scoped tools
• Sends the input as a prompt
• Collects the structured result
• Validates against output schema
• Returns to Inngest step

Sub-workflow pattern (PA, REMS)

When a phase needs to spawn a multi-day saga:

1. Current Inngest step calls step.send_event("pa/initiate")
2. A SEPARATE Inngest function handles PA lifecycle
3. Current function calls step.wait_for_event("pa/completed")
4. Inngest releases all resources — zero cost while waiting
5. Days/weeks later: PA function fires pa/completed event
6. Original Rx function wakes up, resumes from where it left off
7. Context is intact because Inngest memoized all previous steps
Option B: Claude Orchestrates Everything

A single Claude session acts as both the orchestrator AND the execution engine. It holds the phase graph in its system prompt, calls domain experts as subagents, and manages the full lifecycle.

This is what the current monolith prompt does — but decomposed into subagents instead of doing everything inline. Claude holds the workflow state in its conversation context.

What Lives Where

Claude does the thinking AND the routing
Claude owns:
• Phase graph (encoded in system prompt)
• Decision logic (what to do after each phase)
• Context management (in conversation history)
• HITL interaction (via AskUserQuestion tool)
• Report generation (emit_report tool)
• Expert invocation (invoke_expert tool)
External system needed for:
• Multi-day waits (Claude can't hold a session open)
• State persistence across session boundaries
• Event triggers (something must wake Claude back up)
• Crash recovery (if Claude session dies mid-flow)

The session boundary problem

Claude sessions are ephemeral. When the Rx flow hits a multi-day wait (PA approval, prescriber callback), the session MUST end. This means:

1. Before ending: serialize rx_context to database
2. Create a trigger: "when event X arrives, start new session"
3. New session loads serialized context from DB
4. System prompt says: "You are resuming from Phase N"
5. Claude re-orients and continues
Risk: Context loss across session boundaries. The new session doesn't have the conversation history from the previous session — only the serialized context dict. Nuanced reasoning from earlier turns (why a particular match was chosen, operator preferences) may be lost.

The orchestrator's invoke_expert tool

When the orchestrator calls invoke_expert(), a tool handler:

1. Looks up the expert in the registry
2. Creates a NEW Claude subagent session
3. Gives it the expert's scoped prompt + tools
4. Sends the input data as the query
5. Waits for the subagent to complete
6. Validates output against schema
7. Returns structured result to orchestrator

This is a nested Claude session — technically feasible via the MCP tool server pattern. The subagent runs in its own workspace with its own tool allowlist.

Option C: Claude Orchestrates + Durable DAG Memory

Claude is the orchestrator AND the DAG planner. The workflow graph lives in a mutable JSONB memory file that Claude reads and writes. Every decision, expert result, and state transition is checkpointed. Session crashes are survivable. Claude can modify the DAG mid-flow based on discoveries.

Key insight: The DAG is data, not code. Claude reads the DAG, decides the next node, invokes the expert, writes the result, and checkpoints state. If the session dies, a new session reads the file and resumes from the last checkpoint. This gives you durability (from state serialization) AND non-determinism (Claude can replan).
Execution Flow

The saga lives longer than any single Claude session. Each session is short-lived — it reads state, makes one or more decisions, checkpoints, then ends. External triggers wake new sessions when work arrives (human response, fax, scheduled event). The JSONB memory file is the thread that ties them together.

What Lives Where

The JSONB file is the thread, Claude sessions are short
JSONB Memory File owns:
• Current DAG (nodes, edges, statuses)
• Accumulated context from all completed phases
• Decision log (natural language audit trail)
• Expert result cache (idempotency)
• Pending human input (question + context)
• Resume triggers (which events wake this saga)
• Version counter (optimistic locking)
Claude Session owns (ephemeral):
• Reading the saga and understanding current state
• Making the next decision
• Calling domain experts
• Modifying the DAG if needed
• Writing checkpoints
• Each session lives for seconds to minutes
External handlers own:
• Event routing (which events wake which sagas)
• Saga discovery (find sagas with matching resume_triggers)
• Session spawning (start a new Claude session with saga_id)
• Optional cron sweeper (wake stalled sagas, retry failed)

Session boundary pattern

A session does the minimum work possible then ends:
1. Read saga → understand state
2. Execute 1-3 phases (until HITL, sub-saga, or done)
3. Checkpoint final state
4. Register resume_trigger if pausing
5. Session ends — release all resources
Why end the session even without HITL? Cost control. If Claude's done for now, don't keep the session alive. Next decision = new session + read saga. Cache keeps previous expert results.

DAG modification example

During clinical phase, expert returns: "REMS enrollment required, patient not enrolled."

Claude responds by:
1. Appending rems_enrollment node to DAG
2. Rerouting edges: clinical → rems_enrollment → billing
3. Checkpointing modified DAG with version++
4. Decision log: "REMS required, inserted rems_enrollment node before billing"
5. Advances to rems_enrollment (new node)
Original 8-node DAG becomes 9-node DAG. Persists. Next session sees the modified graph as truth.

Crash Recovery

If Claude's session dies at any point, a new session can resume. The memory file is the single source of truth.

💥 Crash scenario

Session dies mid-flow
1. Session checkpoints "about to invoke billing-agent"
2. Invokes billing-agent, gets paid response
3. 💥 Session crashes before checkpointing result
4. Memory file still shows: billing = in_progress
5. Expert cache is empty (result wasn't saved)
6. But the billing result IS in the Claude session history (lost)
Mitigation: Checkpoint BEFORE AND AFTER every expert invocation. Use expert_cache keyed by input hash — if the same input is seen again, reuse the cached result.

🔄 Recovery flow

New session picks up
1. Monitor detects saga is running but session is dead
2. Starts new orchestrator session with saga_id
3. New session reads memory file
4. Sees: current_node=billing, status=in_progress
5. Checks expert_cache for billing input hash
6. Cache miss → re-invoke billing-agent (idempotent)
7. Checkpoints result this time
8. Continues to next node
Key property: Domain experts must be idempotent. Submitting the same claim twice should not double-bill — use the claim's request_id to dedupe server-side.
Dynamic DAG Modification

This is where Claude's flexibility pays off. The DAG is not static — Claude can add, remove, or reorder nodes based on what it discovers. All modifications are checkpointed.

Before DUR

Standard 8-node DAG
intake → data_entry → clinical → billing
→ dur_review → fulfillment → pv2 → shipping

After DUR finds critical DDI

Claude inserted 2 new nodes
intake → data_entry → clinical → billing
→ dur_review → prescriber_outreach
→ dur_reconfirm → fulfillment → pv2 → shipping
Decision log entry: "Critical drug-drug interaction (contraindicated) detected with patient's active warfarin. Inserting prescriber_outreach node to request clarification, then dur_reconfirm to re-validate before proceeding to fulfillment."
Multi-Day Waits Without Sessions

🔌 Saga enters paused state

1. Expert returns { needs_human: true, question: "..." }
2. Claude checkpoints: status = "awaiting_human"
3. Writes pending_human_input to memory
4. Adds resume_trigger: { event: "human_response", saga_id }
5. Claude session ends (no cost)
6. Saga row in DB now shows: paused, awaiting_human

⏰ When trigger fires

1. Operator responds OR PA fax arrives (days later)
2. Event handler finds matching resume_trigger
3. Updates memory file with human_response
4. Starts new Claude session with saga_id
5. Claude reads memory: "I was waiting for human input, here's the response"
6. Continues from where it left off
7. Checkpoints new state, invokes next expert
Why this works without Inngest: The memory file IS the durable state. External event handlers (Postgres LISTEN/NOTIFY, a cron sweeper, webhook handlers) wake up paused sagas by spawning new Claude sessions. No long-running process needed — just trigger → spawn → resume.

Recommendation

Option A — Inngest

Best for:

• High-volume production
• Well-defined rigid workflows
• Strong regulatory audit needs
• Teams with existing Inngest infra

Trade-off:

Rigid. Phase graph in code requires redeploys to change. Not adaptive to edge cases.

Option B — Pure Claude

Best for:

• Early prototyping
• Undefined workflows
• Short-running tasks only
• Exploratory automation

Trade-off:

Fragile. No durability. Can't survive session death or multi-day waits. Not production-ready.

⭐ Option C — Claude + DAG Memory

Best for:

• Adaptive workflows
• Non-linear paths (DAG modifications)
• Durability without Inngest overhead
• Teams that want Claude's flexibility with production reliability

Trade-off:

Requires discipline: idempotent experts, atomic checkpoints, careful memory file schema versioning.
HITL: How Domain Experts Get Human Input

When a domain expert needs operator input, it does NOT ask directly. The orchestrator is the only agent that interacts with humans. This keeps HITL consistent and auditable.

Anti-pattern: Letting domain experts ask users directly. This breaks encapsulation, duplicates HITL logic across experts, and makes the operator experience inconsistent (different experts phrasing questions differently).

4. Orchestrator receives request
Sees needs_human=true. Decides: escalate to operator, skip, or override based on confidence / policy.
👤
5. Orchestrator calls AskUserQuestion
ONLY the orchestrator holds this tool. Formats the question. In Option C: checkpoints saga as "awaiting_human", ends session.
💬
6. Operator responds
Answer stored. Saga wakes up (Option A: step.wait_for_event resumes / Option C: new session triggered).
🔃
7. Orchestrator re-invokes the expert
invoke_expert("intake-agent", { ...original_input, human_decision: "selected Patient A" }). Expert continues.
Why this pattern works:
• Single HITL surface — operators see one consistent question format
• Audit trail — every question + answer logged in decision_log
• Checkpointable — saga pauses at the orchestrator level, not inside an expert
• Retry-safe — re-invoking the expert with the same human_decision is idempotent
• Policy-enforceable — orchestrator can apply org rules (auto-escalate to supervisor, default to "create new" if inactivity)
Expert output contract for HITL:
{
  needs_human: true,
  human_questions: [{
    question: "...",
    options: [{label, description}],
    input_field: "human_decision"
  }],
  partial_state: { candidates: [...] },
  resume_hint: "re-invoke with human_decision"
}
Edge case: in-session HITL for Option B. If the orchestrator is a live Claude session (Option B), it can call AskUserQuestion synchronously and wait for the response in-conversation. But this only works for sub-second operator responses. For anything longer, switch to Option A or C — session ends, resume trigger registered.
Hybrid approach: Start with Option B to prototype the Rx flow decomposition. Once the phase graph stabilizes, migrate to Option C (Claude + DAG memory) for adaptive production workloads. Fall back to Option A (Inngest) only when you need the operational maturity of a dedicated workflow engine. The invoke_expert() function is identical across all three options — only the caller and state persistence changes.
