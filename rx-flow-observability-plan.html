Overview

The mental model

Claude Code emits observability data through four separate channels. Each is independently enableable; each unlocks a different slice of the reference stats. The cheapest channel covers ~80% of the surface.

  1. JSONL transcripts. Every session writes one long .jsonl file, one typed event per line. Primary at ~/.claude/projects/<project-hash>/<session-id>.jsonl, subagents at /private/tmp/claude-501/<project-hash>/<session>/tasks/<agent-id>.output. Always on.
  2. OpenTelemetry. Claude Code has first-class OTEL support. Set a handful of env vars and it emits claude_code.* metrics and logs to any OTLP endpoint.
  3. Hooks. ~/.claude/settings.json can register PreToolUse / PostToolUse / SessionStart / SessionEnd / UserPromptSubmit / Notification hooks that pipe JSON into a shell command. Right place for bespoke event streams.
  4. Agent SDK events. claude_agent_sdk.query() (used inside our invoke_expert custom tool) yields typed messages (AssistantMessage, UserMessage, ResultMessage, SystemMessage). A thin wrapper produces exact state-flip counts and per-expert latencies.
Rule of thumb: JSONL gives you retroactive analysis of anything that has already happened. OTEL gives you live streams. Hooks give you bespoke logging. SDK events give you the fine-grained state machine that produces the “status flipped 38 times” number.

Non-goals for this doc

  • No code changes in this PR — this is plan-only.
  • No new Python or npm dependencies.
  • No installed services (local collector is optional; direct-to-hosted-OTLP works).
  • No edits to .claude/settings.json here — Tier 2 lands that.

Tier 0 — JSONL transcript parser

Reproduce ~80% of the reference stats block from any completed session, retroactively, with zero Claude Code configuration.

TIER 050-line Python parser over existing JSONL∼1 hour

Goal
Given a session id or a path to a .jsonl transcript, print a stats block that mirrors the reference — tool-call counts, error count, thinking/text chars, AskUserQuestion rounds, wall-clock.
Data source
  • Primary session → ~/.claude/projects/<project-hash>/<session-id>.jsonl
  • Subagent outputs → /private/tmp/claude-501/<project-hash>/<session>/tasks/<agent-id>.output
  • Each line is a typed event: user, assistant, tool_use, tool_result, thinking
Implementation
A ~50-line Python script at scripts/observability/parse_transcript.py (ships in a later code PR, not this doc). Reads JSONL line by line, groups by event type, counts, sums, prints a table. Accepts either a session id OR a path.
Covers
  • Tool calls by name (filter tool_use, group by name)
  • tool_results count + error count (tool_result.is_error === true)
  • Thinking block count + total chars (thinking events)
  • Assistant text delta count + chars (assistant.message.content[].type === "text")
  • AskUserQuestion rounds (tool_use.name === "AskUserQuestion")
  • Wall-clock = last event ts − first event ts
Does NOT cover
Per-state duration (running vs. waiting_hot), state-flip counts, live streaming. Those need Tier 1 or Tier 3.
Landing position
Before PR 2.1 — no dependency on the registry or any expert. Ship as a drive-by during Phase 0–1 or as a small follow-on to PR 1.
Why this first: it retroactively explains the 14 subagent transcripts that already exist from the Phase 0–1 session. One hour of work unlocks 80% of the reference stats block on history you already have.

Tier 1 — OpenTelemetry env-var flip

Live streaming metrics via Claude Code’s built-in OTEL support.

TIER 1Set 4 env vars, stand up a local collector∼30 min + 1 h dashboard

Goal
Token burn, tool usage, session duration, API errors streaming live into Grafana (or any OTLP backend).
Env vars
CLAUDE_CODE_ENABLE_TELEMETRY=1
OTEL_METRICS_EXPORTER=otlp
OTEL_LOGS_EXPORTER=otlp
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
Collector
Simplest path: brew install otelcol-contrib, drop a 20-line otel-collector-config.yaml at scripts/observability/otel-collector-config.yaml with an OTLP receiver + Prometheus exporter. Pair with a Grafana container or Grafana Cloud. Easier remote option: point directly at Honeycomb or Grafana Cloud and skip the local collector.
Covers
  • Token counters (input / output / cache hits / cache creation) as live gauges
  • Tool usage counters by tool name (matches the “Bash: 114, Read: 20, …” breakdown)
  • Session duration histograms
  • User decision counters (approved / denied permission prompts)
  • API error counters by error class
All metrics prefix claude_code.*.
Does NOT cover
Fine-grained state transitions (running ↔ waiting_hot ↔ idle), thinking/output char totals. Those still come from Tier 0 or Tier 3.
Landing position
Alongside PR 2.1 — registry loader PR is the first time multiple experts might run concurrently in a session; live metrics become worth having. Could also land earlier — the env vars are safe to set on any version.
Minimal starter dashboard: 4 Grafana panels — (1) tool usage by name, (2) token burn over time, (3) error rate, (4) active sessions. Build the rest as needs surface.

Tier 2 — Hooks for bespoke event logging

Capture events the built-in channels don’t emit — per-command Bash duration, per-tool success/failure at the tool boundary, approval-prompt outcomes.

TIER 2settings.json hooks + 15-line sink script∼45 min

Goal
Which Bash commands fail most? How long does pytest take? When did the user get an approval prompt and did they approve or deny? Hooks answer all three.
Implementation
Add a hooks block to ~/.claude/settings.json registering PreToolUse + PostToolUse commands that pipe the JSON stdin into a sink script at scripts/observability/hook-sink.sh. Sink appends one line per event to ~/claude-code-events.jsonl (or forwards to the Tier 1 OTEL collector).
Covers
  • Per-command Bash duration + exit code
  • Per-tool success/failure timestamped to the second
  • SessionStart / SessionEnd markers for session-level derived metrics
  • UserPromptSubmit markers for “time since last user message” windows
  • Notification events (the moments the CLI drew the user’s attention)
Does NOT cover
Anything inside the model — thinking char counts, output char counts. Hooks only fire on the tool boundary.
Landing position
Alongside PR 3 (orchestrator rewrite). That’s the first PR where the runtime shape changes enough to need pre/post telemetry comparison, and where custom hooks can tag which orchestrator version was active when a tool fired.
Hook scripts can block or break sessions. A slow PostToolUse hook adds latency to every tool call; a crashing one can brick the session. Two mitigations: (1) append-only to a local file, never do network I/O inline; (2) wrap the body in { … } || true so a hook error never surfaces as a non-zero exit.

Tier 3 — SDK state observer for invoke_expert

The final 20% of the reference stats block. Exact “status flipped 38 times; entered waiting_hot 19 separate times” granularity, scoped to expert calls. This is the SLO surface for production.

TIER 3Thin wrapper around query(), emits via the Tier 1 pipeline∼3–4 h

Goal
Per-expert state-flip count, time in running vs. waiting vs. idle, retry count, token usage. Labels every metric with expert_id.
Implementation
carelink-pms/carelink/agi/observability.py — a context manager wrapping claude_agent_sdk.query() (the call PR 1 uses inside invoke_expert). Timestamps every yielded message, increments state counters on transitions, emits metrics via the OTEL pipeline from Tier 1. ~100 lines.
Metric names
  • carelink.expert.call.duration_seconds — histogram, labeled {expert_id, outcome}
  • carelink.expert.state_flips — counter, labeled {expert_id}
  • carelink.expert.retry_count — counter, labeled {expert_id, reason}
  • carelink.expert.waiting_seconds — gauge
Covers
  • Time spent in each state per expert call
  • State flip count per expert call
  • Per-expert latency histograms
  • Per-expert token usage, error count, retry count
  • Per-expert input-schema-validation-failure rate
Does NOT cover
Anything outside invoke_expert. Orchestrator-level state (idle while waiting for the user between expert calls) still comes from Tier 0 or Tier 2.
Landing position
With PR 2.1 (preferred) or PR 3, whichever hits first. PR 2.1 is the natural home because invoke_expert is already changing there — a good moment to wire in metric labels. If PR 2.1 lands without it, fold into PR 3 before the orchestrator rewrite goes live.
SDK message type drift. We hit this during Phase 1: rate_limit_event appeared without being in the documented type list. Tier 3’s observer must tolerate unknown types — log a WARN, not raise. Add a test that feeds a synthetic unknown-type message through the observer and asserts no exception. This is a hard rule.

Sequence + landing plan

The tiers stack. Each is useful alone; each adds a layer. Map to existing migration PRs — no new PR numbers introduced.

Risks & caveats

OTEL collector version mismatch. Claude Code’s metric schema evolves; pin otelcol-contrib to a recent version (≥ 0.90) and re-validate metric names after any Claude Code upgrade. Cheapest check: grep for claude_code. after every upgrade.
Hook scripts can block or break sessions. Keep the sink append-only, wrap in || true.
SDK message type drift. Tier 3 must tolerate unknown types — log WARN, not raise. Covered by a synthetic-unknown-type test.
Subagent transcripts live under /private/tmp/. That directory is ephemeral; macOS purges it on some boots. Tier 0 should run against recent sessions only, or a Tier 2 SessionEnd hook should copy them aside to ~/.claude/projects/<hash>/subagents/.
OTEL and hooks can double-count. If Tier 2 hooks forward to the same OTLP endpoint as Tier 1, tool-usage counters double. Either (a) route hooks to a separate metric namespace, or (b) stop emitting tool-usage counters via OTEL once hooks are on.
Project hashes differ across machines. ~/.claude/projects/<project-hash>/ uses a hash derived from the absolute cwd. Laptop transcripts won’t collate with CI’s unless canonicalized or copied. Not urgent.

How we’ll know each tier is done. Every criterion is empirically verifiable.

JSONL parser		 One command (python scripts/observability/parse_transcript.py <session-id>) prints the reference-stats-block equivalent for any of the 14 subagent transcripts produced during the Phase 0–1 session.
Tier 1
OTEL env flip		 claude_code.tool.usage.count{tool="Bash"} appears in a Grafana panel during a live session. claude_code.token.usage.count{type="input"} climbs as the session runs.

What this doc does NOT authorize

  • Code changes to carelink-pms/, carelink-backend/, carelink-frontend/, or the Claude Code harness (plan-only; implementation lands in later PRs).
  • New Python or npm dependencies (Tier 3 reuses whatever OTLP exporter already ships with the opentelemetry package the SDK pulls in).
  • Installed services (local collector is optional; direct-to-hosted-OTLP is supported).
  • Changes to .claude/settings.json — that lands as part of the Tier 2 PR.
Ship Tier 0 first. One hour of work unlocks 80% of the reference stats block retroactively on sessions that have already happened. The rest follows the migration’s natural cadence.
