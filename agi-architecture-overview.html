CareLink AGI — Multi-Tenancy Architecture

Reference: 2026-04-28 design synthesis. Click tabs to navigate. Click rows / layers / steps to expand.
A coherent picture of access control, agents, memories, workflows, and HITL across Platform → Org → Pharmacy → User → Session. Designed to be opened in a browser with a coworker; nothing here is committed to code yet.

Five-scope hierarchy

Everything in the system has a primary scope. Lower scopes inherit from higher ones; lower scopes cannot see across to siblings of their parent. Click a row to expand.

Platform CareLink itself code, defaults, catalog of agents/workflows/skills

The codebase that runs everywhere. Sealed and versioned at deploy time.

  • Workflow definitions: rx-flow, claims-flow, chat (YAML manifests)
  • Expert MD catalog: 8+ scoped expert markdowns under carelink/agi/agents/
  • Skill library: bundled SKILL.md files
  • Default role × verb permission templates
  • Restish CLI binary + skills binaries

All orgs run the same platform code at any given deploy. Per-org behavior comes from configuration, not from forking the platform.

Org A tenant — its own DB, FS root, GCS bucket structurally sealed from other orgs

The unit of physical data isolation. Each org has its own database; cross-org queries are physically impossible.

  • Pharmacies under it (1 to N)
  • Employee roster (users via WorkOS)
  • Workflow allow-list: which platform workflows this org can run
  • Permission overrides: org-level tweaks to default role × verb templates
  • Org-shared memories: tools/, drugs/ — readable by any pharmacy in this org
  • Audit log (org slice — fan-out to a global audit warehouse is a future concern)
Pharmacy PHI boundary inside an org policy-enforced isolation between sibling pharmacies

Where PHI lives. A user's role is per-pharmacy, not per-org — Carol can be admin at P1 and pharmacist at P2 in the same org.

  • Pharmacy config: NPI, NCPDP id, address, tax data
  • User assignments + per-user roles AT THIS PHARMACY
  • Pharmacy-scoped memories: patients/, prescribers/, insurance/
  • Pharmacy-level workflow customization (approval thresholds, disabled experts)
  • Pending HITL approvals queue: any qualified user at this pharmacy can pick up
User An identity acting at a pharmacy PharmacyStaff many-to-many; per-pharmacy roles

The actor. A user might be a member of multiple pharmacies in the same org, with different roles in each.

  • WorkOS sub (subject)
  • PharmacyStaff memberships (many-to-many): potentially multiple pharmacies
  • Per-pharmacy roles: e.g. admin@P1, pharmacist@P2, technician@P3
Session A workflow instance — one user, one pharmacy, one workflow pinned at creation; cannot change scope mid-flight

A live AGI run. Bound at creation to a single (org, pharmacy, user, workflow) tuple. Nothing in this tuple changes for the session's lifetime.

  • Lease: single-writer lock so two laptops can't drive the same session simultaneously
  • Workspace: scratch FS partition at /var/lib/carelink/agi/workspace/<org>/<pharmacy>/<sid>/
  • Memory snapshot: read-side projection of canonical memories at session start
  • Live SDK orchestrator + active experts
  • Audit events: tool_use, lease_acquired, permission_check, memory_write, approval_decided…
  • HITL state: pending questions + pending approvals (durable in DB)

Connection routing flow

Request ──▶ JWT { sub, org_id, exp }
            │
            ▼
        Org-routing middleware
            │
            ├─▶ Lookup control plane: org_id → conn_string  (cached 5min)
            │
            ▼
        Inject org-scoped DB Session into request context
            │
            ▼

What makes the org boundary structural

  • The DB connection itself is org-bound — a SQL injection in OrgX cannot reach OrgY's data.
  • FS paths are partitioned per org — no shared filesystem mount escape.
  • GCS buckets are per-org — IAM policies restrict cross-org access.
  • Backup/restore is per-org — clean tenant lifecycle.
Cost of this shape: schema migrations fan out across N databases (alembic per-org). Connection pools per-org. Backup jobs per-org. Cross-org analytics needs an explicit pipeline. These are operational realities of DB-per-tenant — not unique to AGI, but worth budgeting.
What we lose: cross-org learning. Org X and Org Y both rediscover the same FDB API quirks independently. Today's plan: option A — accept it. Future option B (CareLink-curated platform priors) is on the roadmap, not in v1.

Workflows as first-class entities

Today the runtime is rx-flow-specific. To extend to multiple workflows + AI-assisted chats, workflow becomes a top-level entity the way expert is today. A workflow is a manifest declaring orchestrator + experts + memory scope + HITL policy.

Three workflow examples

rx-flow.yaml

id: rx-flow
description: "New prescription intake → fulfillment"
orchestrator_prompt: rx-flow-orchestrator.md
experts:
  - rx-flow-intake
  - rx-flow-data-entry
  - rx-flow-clinical-checklist
  - rx-flow-dur
  - rx-flow-billing
  - rx-flow-fulfillment
  - rx-flow-shipping
  - rx-flow-rejection-classifier
required_roles_to_start: [pharmacist, technician, admin]
memory_scope:
  read:  [tools, drugs, patients, prescribers, insurance]
  write: [tools, drugs, patients, prescribers, insurance]
hitl_approval_required_for_verbs:
  submit-claim:    [pharmacist, billing_admin, admin]
  reverse-claim:   [pharmacist, admin]
  update-pt:       [pharmacist, technician, admin]

chat.yaml — the AI assistant

id: chat
description: "Open-ended AI assistant chat"
orchestrator_prompt: chat-orchestrator.md
experts: []                              # orchestrator answers directly
required_roles_to_start: [admin, pharmacist, technician, assistant]
memory_scope:
  read:  [tools, drugs]                  # safer default — no PHI
  write: []                              # chat doesn't write memories
hitl_approval_required_for_verbs: {}     # read-only domain

claims-flow.yaml — future

  - rx-flow-billing                      # ← shared with rx-flow!

  write: [tools, drugs, insurance]       # no patient/prescriber writes

  submit-medical-claim:    [billing_admin, admin]
  resubmit-medical-claim:  [billing_admin, admin]

Adding a new workflow becomes data, not code

  1. Drop workflows/my-new-flow.yaml
  2. Write workflows/orchestrator_prompts/my-new-flow.md
  3. Reference existing experts (or add new MDs)
  4. Org admin enables the workflow
  5. Frontend lists it from GET /workflows

No router changes. No frontend rebuild. No expert refactoring. The dispatcher already handles the abstract case once "workflow" is the data type.

Access control — six layers, one source of truth

Permit.io owns the canonical role × verb policy. The AGI service projects from it for fail-fast denials at six points. Click any layer to expand.

0 Org reachable connection routing middleware
JWT carries org_id → control plane DB lookup → if unknown, 404. Cross-org access is impossible because there's no connection to follow.
1 User authenticated WorkOS at API gateway
Standard WorkOS JWT validation. JWKS verification, expiry check, audience claim. Already in place.
2 Workflow startable session creation handler
Three checks at POST /sessions:
  1. Org X has workflow W enabled in org settings
  2. User role at pharmacy P intersects workflow.required_roles_to_start
  3. User is in PharmacyStaff for P
3 Expert invocable filter at session start; enforce at MCP boundary
Workflow's expert list filtered to those the invoker can use. Permit.io rule: (actor, role@P, expert) → allow|deny. Filtered set determines what the orchestrator sees in its invoke_expert enum.
4 Restish verb permitted PreToolUse hook on Bash
Today's restish_guard.py extended. On every restish carelink <verb> ... call, project against the role × verb matrix. On deny: typed envelope {"status": "permission_denied", "verb": "...", "reason": "..."} instead of running the command. Audit-logged.
5 HITL approver authorized approval-decide endpoint
Approver's pharmacy_id == approval.pharmacy_id, AND approver's roles ∩ approval.required_roles ≠ ∅. Logs both initiator and approver in audit event.
6 Backend RBAC (authoritative) existing Permit.io enforcement at endpoint
Layers 0-5 are defense-in-depth and fail-fast. Layer 6 is the single source of truth — Permit.io ABAC policy on the backend re-validates the actual restish call. If 0-5 are bypassed by a bug, 6 still holds.

Memory layer — bucket scope, append-only, TTL

Memory is the cross-session learning store. Today every session creates an empty memories/ dir and deletes it at end — no persistence happens. The proposed design makes the memories real, scoped per-pharmacy, and time-bounded.

Filesystem layout (per org)

/var/lib/carelink/agi/
├── workspace/<org_id>/<pharmacy_id>/<session_id>/ # ephemeral
└── memory/<org_id>/
    ├── org/ # shared by all pharmacies in this org
    │   ├── tools/<verb>.md # e.g. list-patients.md
    │   └── drugs/<ndc>.md # e.g. 00378-0123-45.md
    └── pharmacy/<pharmacy_id>/ # PHI scope
        ├── patients/<patient_id>.md
        ├── prescribers/<npi>.md
        └── insurance/<bin_pcn>.md

Read-side projection at session start

  1. Compute (workflow.memory_scope.read, pharmacy_id, role)
  2. Walk the canonical store, copy matching keys into workspace/memories/
  3. Write a manifest at workspace/.memory-manifest.json recording: keys included, keys excluded by role, snapshot timestamp
  4. The manifest IS the audit record (one event at session start, no per-read overhead)

Append-only writes via merge step

The expert writes freely to its workspace memories/. At session end, a merge step:

  1. Diffs each modified file vs the snapshot manifest
  2. Extracts the operator's added content (everything past the snapshotted byte range)
  3. Wraps it in a signed entry block
  4. Appends to the canonical file (open(path, 'a')) — never overwrites

Sample memory file

$ cat /var/lib/carelink/agi/memory/orgX/pharmacy/P1/patients/patient-9c.md

<!-- entry: 2026-04-28T14:02Z actor=alice role=technician session=8f3a... -->
Patient routinely provides DOB transposed (1984 vs 1948). Confirmed via ID
twice. Verify on every intake.
<!-- /entry -->

<!-- entry: 2026-04-25T09:11Z actor=bob role=pharmacist session=2d1e... -->
Name on insurance card omits hyphen ("Smith Jones" vs "Smith-Jones").
Always strip hyphen for claim submission.
<!-- /entry -->

TTL sweeper (14 days)

Inngest cron every 6h walks the memory tree per-org:

  • Parses entries (regex on <!-- entry: ... -->)
  • Drops entries older than 14d
  • If a file becomes empty: deletes it
  • Otherwise: rewrites with surviving entries + a <!-- swept: ... --> marker at top
  • Audit event: memory_swept per pharmacy with files_pruned, entries_pruned counts

drugs/<ndc>Org sharedAny session in orgAny session in org14d

Today's reality: every expert MD references memories/ as if it persists. The code path doesn't preserve it — workspace is rmtree'd at session end. The "cross-run learning" feature is currently aspirational. Implementing this is the load-bearing piece of the memory design.

End-to-end session lifecycle

Setup: OrgX has 3 pharmacies. Alice is technician @ P1. Bob is pharmacist @ P1. Alice starts rx-flow, hits a submit-claim she can't approve, walks away. Bob picks it up from his laptop.

The five lanes are the actors. Each numbered node is a step. Click any step pill below or use ← → arrow keys.

Step:

Step 1 — Alice opens her laptop and starts a session

Alice Backend

Frontend posts POST /sessions with { workflow: "rx-flow", pharmacy_id: "P1", context: {...} }. Bearer is Alice's WorkOS JWT.

Step 2 — Org routing middleware resolves the DB

Backend

JWT claim org_id=X → control plane lookup → resolves orgX_db connection. CareLink instance scoped to OrgX is injected into the request context.

Step 3 — Session creation runs all access checks

Backend
  • OrgX has rx-flow enabled in workflow allow-list
  • Alice's role at P1 is technician → ∈ workflow.required_roles_to_start
  • Permit.io projects: Alice's permitted_verbs = {list-pt, get-pt, list-drug, ...} — no submit-claim
  • Workflow's experts filtered to {intake, data-entry, clinical-checklist, dur, billing}
  • Billing expert can run in HITL-required mode (parks submit-claim, asks an approver)

Step 4 — Memory snapshot projects the workspace

Backend Storage

Backend reads canonical memory store + creates ephemeral workspace.

  • Copies /memory/orgX/org/{tools,drugs}/* (org-shared)
  • Copies /memory/orgX/pharmacy/P1/{patients,prescribers,insurance}/* (PHI scope)
  • Writes workspace/.memory-manifest.json with key list + timestamps
  • Audit event: memory_snapshot

Step 5 — Lease acquired by Alice's machine

Backend Storage

Workspace at /var/lib/carelink/agi/workspace/orgX/P1/<sid>/. Lease columns set: holder_machine_id=laptop_1, holder_actor_id=alice, expires_at=now+60s. Frontend starts heartbeat (extends lease every 15s).

Step 6 — Orchestrator runs through scoped experts

Backend SDK

intake → data-entry → clinical-checklist → dur → billing. Each expert's PreToolUse hook (restish_guard) checks each call: in expert allowlist AND in Alice's permitted_verbs. Reads succeed; writes that don't require approval succeed.

Step 7 — Billing expert wants submit-claim, hook denies

SDK

Expert generates restish carelink submit-claim --rx-id=.... PreToolUse hook checks Alice's permitted_verbs:

  • submit-claim requires HITL approval for technician role
  • Hook decision: park it (don't execute, don't deny outright)

Step 8 — Approval row written, hot window starts

SDK Storage

DB row inserted into agi_pending_approvals:

{
  verb: "submit-claim",
  initiator: "alice",
  pharmacy_id: "P1",
  required_roles: ["pharmacist", "billing_admin", "admin"],
  session_id, command, ts
}

Hot path waits 60s for decision. Audit event: approval_requested.

Step 9 — Alice walks away, cold-park

SDK Storage

Hot window elapses. SDK subprocess exits cleanly via continue_=False, stopReason=.... Session status flips to waiting_cold. Lease released (heartbeat stopped). Audit event: lease_released.

Step 10 — Bob picks up at his laptop

Bob Backend

Bob opens "Pending Approvals" tab for P1 on a different laptop. Frontend hits GET /pharmacies/P1/approvals. Sees Alice's pending submit-claim in the queue. Clicks "Approve."

Step 11 — Approval-decide runs the role check

Bob Backend Storage

POST /pharmacies/P1/approvals/{aid}/decide with Bob's bearer.

  • Decoded JWT — Bob is in OrgX
  • Bob's pharmacy_id == approval.pharmacy_id (both P1)
  • Bob's roles {pharmacist} ∩ required {pharmacist, billing_admin, admin} ≠ ∅

DB write: {decided_by: bob, decided_at: now, decision: approve}. Audit event: approval_decided with both initiator (Alice) and approver (Bob). Inngest event dispatched: agi/cold_approval_requested.

Step 12 — Inngest cold-revive executes the parked command

Inngest SDK
  • Reads OrgX DB → finds session, finds approval, reads workspace path from extra_data["workspace"]
  • Workspace lives on shared FS — Inngest worker reads it regardless of replica
  • Acquires lease (force=true since prior lease released)
  • Executes submit-claim via asyncio.create_subprocess_shell with a freshly minted bearer
  • Spawns new SDK client with resume=session_id + synthetic user-turn carrying real tool_result

Step 13 — Session resumes, completes, and merges memories

SDK Storage

Orchestrator sees submit-claim succeeded. Continues to fulfillment, shipping. On finalize_session:

  • Memory merge: workspace memories/ diff appended to canonical store with <!-- entry: actor=alice ... --> blocks
  • Audit events flushed (including the alice→bob approval handoff trail)
  • Workspace rmtree'd
  • Lease released
Every cross-cutting concern composes cleanly. No special-case logic for "this happened across two machines" — that's the default behavior of the architecture.

HITL — pharmacy-queue model

HITL approvals and questions are workflow-agnostic. The protocol is shared. Surfacing them at pharmacy level (rather than session level) makes cross-machine handoff trivial.

The pharmacy approval queue

┌──────────────────────────────────────────────────────────┐
│  Pharmacy P1 — Pending Approvals                         │
│  (visible to any qualified user at this pharmacy)        │
├──────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────┐  ┌────────────────────┐ │
│  │ rx-flow session             │  │ claims-flow session│ │
│  │ wants submit-claim          │  │ wants              │ │
│  │ initiator: Alice (tech)     │  │ submit-medical-... │ │
│  │ required_roles:             │  │ initiator: Bob (t) │ │
│  │  {pharm, billing, admin}    │  │ required_roles:    │ │
│  └─────────────────────────────┘  │  {billing, admin}  │ │
│                                   └────────────────────┘ │
└──────────────────────────────────────────────────────────┘
                          ↓
              Both rows in same DB table:
              agi_pending_approvals (per org)
                  keyed by pharmacy_id

       Decided by any user at this pharmacy whose
       roles match the row's required_roles set.

Hot vs cold path — same outcome, different timing

PathWhenMechanism
Hot Approver decides within 60s In-memory ToolApprovalGate.wait_for_decision_or_timeout resolves directly. SDK subprocess never exits. Session continues immediately.
Cold Hot window elapses, decision arrives later Subprocess exits. Status = waiting_cold. When approver decides, Inngest func_cold_approval_drain reads DB, executes the parked command, spawns fresh SDK client with resume=session_id, injects synthetic tool_result.

Why pharmacy queue beats session queue

  • Cross-machine "for free" — any qualified user at the pharmacy sees the approval, regardless of which machine the originating session ran on.
  • Workflow-agnostic — same UI surface for rx-flow, claims-flow, and any future workflow.
  • Compliance-friendly — clear separation of duties: initiator and approver are different roles, both audited.
  • Resilient to lease handoff — Alice's machine doesn't need to be running for approval to land.

routers/agi.py — extend

HTTP surface. Already exists at 1,675 LoC. Extends with: org-routing middleware (delegated), workflow-listing endpoints, pharmacy-scoped approvals endpoints, lease endpoints. Logic stays thin — delegates to service.py and the new modules.

workflows/selector — NEW

Resolves (workflow_id, invoker, pharmacy_id) into a fully-projected session config: which experts are enabled, which memory keys to copy, which approval policies apply, what orchestrator prompt to use. Called once at session creation. Pure function — consumes registry + projector + memory store + sessions/lease.

service.py — extend (thinning)

Today: 2,011 LoC god class holding everything. Target: ~500-700 LoC orchestrator that delegates to modules. resume_with_answer + resume_with_approval_result collapse to a single resume_with_synthetic_turn (~200 LoC saved). System-prompt construction moves to workflows/. Memory snapshot logic moves to memory/. Lease logic moves to sessions/lease.

workflows/registry — NEW

Loads + validates YAML manifests under workflows/definitions/. Validates: every referenced expert exists in agents/registry; every referenced restish verb is known; required_roles are valid; orchestrator prompt MD exists. Idempotent — same YAML, same registry state.

permissions/ — NEW

Three files: projector.py (Permit.io → InvokerContext), matrix.py (role × verb data shape with org overrides), audit.py (writes permission-check events). Cached projection per session; refresh on Permit.io change events.

memory/ — NEW

Four files: store.py (org/pharmacy path resolution), selector.py (workflow + role + pharmacy → keys to snapshot), merger.py (workspace diff → canonical append-only entry), ttl.py (Inngest cron sweeper). Implements the two-tier model: canonical store + per-session projection.

hitl/ — extracted from hooks.py + extended

Today's UserQuestionGate + ToolApprovalGate moved here. NEW: approvals_resource.py exposes pharmacy-scoped approval queue as a queryable HTTP resource. Decide endpoint runs role check; audits both initiator and approver.

sessions/lease.py — NEW

Single-writer DB-backed lease with heartbeat (15s extend, 60s expiry). Acquire, force-take-over, release flows. Replaces today's in-memory cl._agi_clients dict. Foundation for multi-replica deploy.

agents/registry — existing

Already at 442 LoC. 4 surviving validation gates. Will be invoked by workflows/registry to verify expert references, but otherwise unchanged.

tools.py — existing

1,173 LoC. Holds invoke_expert MCP tool, _spawn_expert_agent, retry layers. Per the Phase D spike, these stay canonical; the dormant agents= wiring at service.py:1680-1691 can be deleted (-65 LoC).

drain.py — existing

285 LoC. Shared headless receive_response() driver for live + cold-revive paths. Used by Option A asyncio task, Option C Inngest function, and the live-turn SSE pump.

contracts/rx_flow.py — existing

908 LoC, 16 typed I/O contracts. Output side deliberately loose. Stays as the source of truth for expert I/O. New workflows declare their own contracts file when their experts have new contract shapes.

Permit.io — external (existing)

Source of truth for role × verb policy. AGI calls via SDK at session start to project permissions. Already in use for endpoint-layer RBAC.

Inngest — external (existing)

Durable function runtime. Today hosts func_cold_revive_drain + func_cold_approval_drain. Will additionally host the memory TTL sweeper cron.

Shared FS — NEW infrastructure

Mount point on every backend replica. Holds workspace tmpdirs + canonical memory store. Per-org root partitions. Recommended substrate: GCP Filestore (managed NFS).

Org DB — per-tenant

One DB per org. Holds agi_sessions (with new lease columns), agi_pending_approvals (NEW), audit events, permission overrides. Resolved by org-routing middleware from JWT claim.

GCS — per-tenant

One bucket prefix per org. Holds finalized transcripts, skill bodies, skill proposals. Already in use today.

Suggested sequencing

  1. Workflows-as-first-class (foundation; everything downstream slots in)
  2. Permissions projector + matrix (largest immediate security gain)
  3. Pharmacy-level approvals resource (unblocks cross-machine HITL)
  4. Memory canonical store + merger + TTL (turns aspirational MD prose into real cross-run learning)
  5. Session lease + shared FS migration (unblocks multi-replica deploy)
  6. service.py decomposition (retroactively cleans up as logic moves out)

Request flow — what happens when Alice clicks "start session"

Request flow at a glance

Click any stage above for details. The flow runs left-to-right top row (request goes in, session spawns), then right-to-left bottom row (a single restish call gates through 4 layers and hits the backend), then down to the SSE response.

Browser

Frontend holds Alice's WorkOS JWT (short-lived). Every API call carries it in the Authorization header. Browser knows the workflow_id + pharmacy_id from the session-creation form.

Org routing middleware

Reads org_id from JWT claim, looks up the control plane DB for that org's connection string, fs root, GCS bucket. Cached 5min. Injects org-scoped DB Session into request context. Unknown org_id → 404.

Auth + actor resolution

Existing WorkOS JWKS validation. Extracts actor_id (subject). Verifies expiry + audience. Standard FastAPI dependency.

Workflow selector

Five checks: org has workflow enabled, user role at pharmacy intersects required_roles, user is in PharmacyStaff, Permit.io projects permitted_verbs, expert set filtered. Produces a fully-resolved session config: pinned experts, memory snapshot keys, approval policies.

SDK orchestrator

Spawned with the projected scope: allowed_tools narrowed, agents dict containing only permitted experts, system prompt rendered from workflow's orchestrator_prompt template, workspace mounted with memory snapshot.

Expert subprocess

Orchestrator's LLM emits invoke_expert(billing, payload). MCP tool spawns expert subprocess with its own scoped tool list. Each expert is a one-shot Claude SDK query() with the expert's MD body as system prompt.

restish_guard PreToolUse hook

Fires on every Bash invocation inside the expert. Two checks: (a) command verb in expert's allowed_restish_commands, (b) verb in invoker's permitted_verbs (with HITL routing if approval required). Deny → typed envelope; HITL → park.

Bash exec

Approved restish call runs in the workspace with HOME redirected, bearer token from workspace/.config/restish/apis.json. Output captured, returned as tool_result to the expert.

Backend RBAC (authoritative)

Permit.io ABAC at the endpoint. Verifies the bearer's subject can perform action × resource with the request's resource attributes (e.g., pharmacy_id). This is the single source of truth — even if AGI's projection is stale, this is correct.

CareLink API endpoint

Standard FastAPI route. Reads/writes the Org DB. The connection injected by org-routing middleware ensures cross-org isolation.

SSE stream back

Tool result flows back through the SDK to the orchestrator, which produces the next assistant turn. Streamed via SSE to the frontend. UI renders ChatThread blocks: tool_use, tool_result, expert_output_card, etc.

Memory flow — canonical → snapshot → write → merge

Memory flow at a glance

Two halves: at session start, canonical → workspace projection. At session end, workspace → canonical merge (append-only). The TTL sweeper runs out-of-band on a cron, pruning entries older than 14 days from canonical files.

Canonical store

Path: /memory/<org_id>/{org,pharmacy/<P>}/<bucket>/<key>.md. Each .md file is a stack of <!-- entry: actor ... --> blocks in chronological append order. Read by every session at start; written by every session at end.

Memory selector

Computes which keys to copy: intersect workflow.memory_scope.read with role-permitted buckets at the session's pharmacy. Assistant role might exclude patients/ entirely. Walks the canonical FS and selects matching files.

Workspace memories/

Selected files copied (or hardlinked) into workspace/memories/. Manifest written at workspace/.memory-manifest.json: list of keys included, list excluded by role, byte sizes, source mtimes, snapshot timestamp. The manifest IS the audit record.

Expert reads + writes

During the session, experts read these files like normal MD files. They append notes ("Patient routinely provides DOB transposed..."). They DON'T see the merge or TTL — they just see a per-key MD file.

Diff vs manifest

At session end, the merger compares each modified file to the snapshot byte range from the manifest. Anything past the snapshot is "new content from this session." Deletions or edits to pre-existing content are silently ignored — append-only invariant enforced here.

Merger wraps entry

The diff content is wrapped in <!-- entry: actor=<sub> role=<role> session=<sid> ts=<iso> --> ... <!-- /entry -->. Comment fences are human-readable; metadata is in-band. Workflow's memory_scope.write gates which buckets actually merge.

Canonical store (post-merge)

File grows by N entries. open(path, "a") with file lock. Two concurrent sessions merging the same file? Append in arrival order. No conflict resolution needed.

Workspace rmtree

Existing pattern from finalize_session. Deletes the entire workspace tmpdir. Memories already merged; transcript already uploaded; proposals already shipped.

TTL sweeper

Inngest cron, every 6h. Walks each org's memory tree. For each .md file: parses entries, drops those older than 14d, rewrites file with surviving entries (or deletes if empty), writes a <!-- swept: ... --> marker. Audit event per pharmacy with prune counts.

Permission flow — JWT to enforced action

Permission flow at a glance

One projection at session start, then six enforcement layers (L0-L6 from the Access Control tab). Layers 0-5 are fail-fast advisory; L6 is authoritative. The InvokerContext is the single object that flows through all checkpoints.

JWT

WorkOS-signed. Claims include sub (user id), org_id (which DB to use). Short-lived. Refreshed by the frontend.

Permit.io

Source of truth for role × verb × resource policy. AGI calls via SDK to query "what can this user do at this pharmacy?" Returns the projected permission set.

InvokerContext

Pydantic object: {invoker_id, roles, pharmacy_ids, permitted_verbs: frozenset[VerbScope], permitted_experts, projected_at, ttl_seconds}. Cached per-session. Refreshed on Permit.io change events.

Pinned on session row

Persisted to extra_data.invoker so cold-revive can read it without a fresh Permit.io call. Snapshot at session start; respects 4h ceiling for re-auth.

L2: Workflow startable

Three checks at POST /sessions: (1) org has workflow enabled, (2) user role at pharmacy intersects workflow.required_roles_to_start, (3) user is in PharmacyStaff at pharmacy.

L3: Expert invocable

Filter at session creation: workflow's expert list narrowed to those the invoker can use. Enforced at MCP invoke_expert handler — even if orchestrator tries an expert outside its scope, MCP rejects.

L4: Verb permitted

PreToolUse hook on Bash. On every restish carelink <verb> call, projects against role × verb matrix. Three outcomes: allow → run, HITL → park, deny → typed envelope. Audit event written.

L5: Approver auth

POST /pharmacies/{pid}/approvals/{aid}/decide. Validates approver's pharmacy_id == approval.pharmacy_id AND approver.roles ∩ approval.required_roles ≠ ∅. Logs both initiator and approver.

L6: Backend RBAC (authoritative)

Existing Permit.io enforcement at the actual API endpoint. The single source of truth. If L0-L5 had any bug, this catches it. AGI's projection is fail-fast; this is final.

Action executes or 403

Either: action runs, response back to expert, audit event "tool_use" recorded; or 403, typed envelope, audit event "permission_denied" recorded.

Audit flow — events to compliance reports

Audit flow at a glance

Many producers, one collector, two persisters, two consumers. Producers fire events at well-defined moments. The collector buffers in-process and broadcasts to SSE. At session end, events flush to Org DB. Final transcript also lands in GCS.

PreToolUse hook

Every Bash invocation. Records: tool_use on allow, permission_denied on deny, permission_check_hitl on park. Includes verb, role, decision reason.

Memory selector

One event per session start: memory_snapshot with manifest of keys included + keys excluded by role. Single snapshot at start; no per-read overhead.

Lease manager

lease_acquired, lease_revoked, lease_expired. Captures from-instance, to-instance, machine_id, actor_id. Lets you trace who held which session when.

Approval gate

approval_requested on park, approval_decided on resolve. Decided event captures BOTH initiator (who started) and approver (who decided), plus role-check result.

Memory merger

One event per modified memory file: memory_write with bucket, key, byte count, expert that wrote. memory_write_blocked_by_workflow when scope.write disallows.

TTL sweeper

Per-pharmacy event from cron: memory_swept with files_pruned, entries_pruned, bucket counts.

AuditCollector

Existing component (hooks.py:524+). In-process list with broadcast callback. Already wired for tool_use + state transitions; new event types compose without changing the broadcast contract.

Org DB persistence

Today: extra_data.events_log JSONB column. Future: dedicated agi_audit_events table for queryability. Either way, per-org DB means cross-org leakage is structurally impossible.

GCS final transcript

At session end, full SDK transcript JSONL uploaded to per-org GCS bucket. transcript_gcs_path on session row points to it. Long-term retention per the org's compliance policy.

Live SSE consumers

Frontend stats panel + agent trace panel. Pushed via the broadcast callback. No polling. Already in place.

Compliance reports

"All HITL approvals at PharmacyA in March, group by approver_role." One SQL query against the org's audit table with WHERE on pharmacy_id. The pharmacy column makes per-pharmacy reports trivially derivable.

Open questions worth resolving

None of these block the v1 architecture. Each shapes a future iteration.

1. Cross-pharmacy users at the same org

Alice is technician @ P1 and assistant @ P2 in OrgX. When she opens the frontend, does she pick a pharmacy at login or switch contexts mid-session?

Leaning: pick at login, switch only via re-auth. Simplifies session-bound pharmacy_id; avoids context-switch edge cases.

2. Workflow versioning

When rx-flow's definition changes (new expert, prompt update), do running sessions see the new version mid-flight?

Leaning: no — pin the workflow version at session creation, store on the session row. New sessions get the new version.

3. Org workflow override governance

Who in OrgX can flip claims-flow.enabled = true or override approval thresholds? Org admin only, or delegate to compliance?

4. Memory promotion / pinning

14-day TTL is naive. Some notes (a payer's quirk that's stable for years) should be evergreen. Need a "pin" mechanism that exempts entries from TTL. Phase 2.

5. Cross-org learning (option B/C)

Today: option A (each org learns independently). Future: option B (CareLink-curated platform priors for tools/drugs) or option C (anonymized cross-org write-back). Both require redaction and curation infrastructure.

6. Take-over UX during in-flight tool_use

What happens if the active session is mid-tool-use when another machine takes over? Probably: defer take-over until quiescent boundary. Edge case worth speccing.

7. Inngest concurrency vs lease

Cold-revive Inngest function might fire on any replica. How does it interact with the lease? Probably: forces lease acquisition, runs to completion, releases.

8. Connection pool sizing per org

One backend deployment talking to N org DBs needs per-org pool limits to prevent noisy-neighbor starvation. Standard practice but needs a pool config strategy.

9. Schema migration orchestration

Alembic against N DBs. Standard DB-per-tenant pain. Patterns: lockstep deploy, migration runner script, forward-compatible code. CareLink probably already has this for the core schema; AGI tables fold in.
