CareLink Sandbox Seed — Walkthrough (v2 · services-only · current-state-only)

Target: canary environment. Source: Carelink Sandbox Seed Spec.docx. Personas: 5 (Maria · David · Eleanor · Marcus · Raj). Demo: 7 steps intake → delivery. Last updated 2026-05-13.

Overview

Major pivot from v1 We are no longer writing directly to the DB. All seed work happens via PMS service calls (cl.patients.create(...), cl.inventory.products.create(...), etc.) so audit logs / FHIR mirror / Permit sync / Inngest dispatch / external integrations fire naturally and the seeded data is indistinguishable from real production data. And we are no longer seeding historical artifacts — services force now() on operational timestamps, and faking history (UPDATE-after-create) defeats the point of using services. Demo users who need historical state must generate it themselves by running real flows before the demo.

This document plans the canary sandbox seed in 6 phases. The principle: use PMS services for everything we can seed; explicitly call out everything we can't. Anything historical (delivered shipments, paid claims, completed work items, past calls / SMS / faxes, completed touchpoints, completed questionnaires, backdated audit logs) is NOT SEEDED — the demo operator generates this state via real flows before the demo, or accepts it as a demo gap.

Architecture decisions (from PMS deep-dive)

The 4-agent investigation produced these architectural facts that shape every other section.

How CareLink works

  • Facade singleton at carelink-pms/carelink/__init__.py. Integrations (FHIR, WorkOS, Permit, Asepha Voice, GCS, Stripe, ShipStation, etc.) are constructed eagerly in __init__, each gated on its credential being present — when absent, the slot is None. Domain services (cl.patients, cl.inventory, cl.shipments, cl.ivr, ...) are lazy @property instances constructed on first access.
  • Per-method DB session. Each service method opens its own async with self._db.session() as session: via the Database.session() async contextmanager. Auto-commit on clean exit, rollback on exception.
  • Audit logs are atomic with the SQL write. AuditService.log(session, ...) takes the same session as a parameter, so the audit row commits in the same transaction.
  • External integration side effects are NOT atomic with the DB write — they fire after the SQL row commits, with try/except + log on failure. If Permit sync fails, the entity still exists.
  • Inngest dispatch is best-effort and silent. Send failures are logged, not raised. If Inngest isn't registered (which it won't be in seed scripts), events go to dev-mode and are effectively dropped — this is desirable for seeds, we don't want PV1/DUR workflows running on sandbox data.
  • FHIR is hard-required. FHIRClient is always constructed, never None. Every clinical service (Patient, Practitioner, MedicationRequest, Coverage, Claim) writes to FHIR first. The sandbox needs a reachable FHIR store OR a stub. See Q open list.

The actor_id="system" convention

The literal string "system" passed as actor_id short-circuits the User lookup at carelink-pms/carelink/core/base_service.py:52 and yields a synthetic admin actor with role="admin", bypassing every authz.require() call. Audit log entries will show actor_email="system@carelink.internal" — distinguishable from real human actions in the audit trail.

Hard requirement to resolve before seed runs FHIR sandbox — every clinical service write goes through cl._fhir.create(...). Without a reachable Google Healthcare FHIR store (or a stubbed FHIRClient), the seed cannot run. See open question FHIR availability.

async def seed_pharmacy(cl: CareLink) -> PharmacyResponse:
    # cl.pharmacies.create raises ConflictError on duplicate NPI — pre-check first
    existing = await cl.pharmacies.find_by_npi("...", actor_id="system")
    if existing:
        return existing
    pharmacy = await cl.pharmacies.create(
        actor_id="system",
        data={
            "name": "Demo Pharmacy",
            "address_line1": "250 Front St",
            "city": "Columbus", "state": "OH", "zip_code": "43215",
            "phone": "614-555-0100", "fax": "614-555-0101",
            "pharmacy_type": "specialty",
            "npi": "...", "ncpdp": "...", "dea_number": "...",
        },
    )
    for staff in STAFF_SEED:
        try:
            await cl.users.create(actor_id="system", **staff)
        except ConflictError:
            pass  # user already exists
    return pharmacy

seed_03_inventory.py — service-based sketch

async def seed_inventory(cl: CareLink, pharmacy_id: UUID):
    # Locations
    fridge = await cl.inventory.stock.locations.create(
        "system", "Refrigerator A", "FRIDGE-A", "refrigerator")
    shelf  = await cl.inventory.stock.locations.create(
        "system", "Main Shelf", "SHELF-MAIN", "shelf")

    # Products — idempotent (ON CONFLICT DO NOTHING on NDC)
    for spec in PRODUCTS:                # 14 entries
        product = await cl.inventory.products.create(
            actor_id="system",
            ndc=spec["ndc"], name=spec["name"],
            list_price=spec["list_price"],
            cost_price=spec.get("cost_price"),
            manufacturer=spec["manufacturer"],
            requires_refrigeration=spec.get("requires_refrigeration", False),
        )
        # Lots via stock.receive (creates lot + stock + transaction together)
        for lot in spec["lots"]:
            await cl.inventory.stock.receive(
                actor_id="system",
                product_id=product.id,
                location_id=(fridge if spec.get("requires_refrigeration") else shelf).id,
                quantity=lot["quantity_on_hand"],
                lot_number=lot["lot_number"],
                expiration_date=lot["expiration"],
            )

    # Suppliers — no unique constraint on name, pre-check via list
    existing = {s.name for s in await cl.inventory.suppliers.list_all("system")}
    for spec in SUPPLIERS:
        if spec["name"] not in existing:
            await cl.inventory.suppliers.create(actor_id="system", **spec)

Decisions — Decided ✓

D1 · Architecture — PMS services, not direct DB writes

All seed work calls cl.*.create() service methods. This makes audit logs / FHIR mirror / Permit sync / Inngest dispatch / external integrations fire naturally and yields data indistinguishable from production.

D2 · No historical artifacts

The seed creates current state only. Services force now() on operational timestamps and we don't fight that. Demo operators generate historical context via real flows before the demo session, or accept demo gaps for items that can't be flow-generated (e.g., inbound calls).

D3 · SMS conversations — NOT SEEDED

No local table exists for SMS (all state lives in Asepha Voice). Per decision: operator generates SMS by sending real messages via the chat UI as part of demo prep.

D4 · 3 new patient sub-resource tables, FK on patient_fhir_id

patient_allergies, patient_conditions, patient_coverages all key on FHIR Patient ID (TEXT), matching the existing care_team_members pattern. See Step 1 schemas.

D5 · Sandbox config recipe — null optional integrations

ASEPHA_VOICE_API_KEY="", WORKOS_API_KEY="", CALCOM_API_KEY="", PERMIT_API_KEY="", AGI_ENABLED=false, CHAT_HOMESERVER_URL unset. See Architecture section.

D6 · actor_id="system" convention

The string literal short-circuits user lookup and yields a synthetic admin actor. Audit log entries show actor_email="system@carelink.internal".

D7 · Raj's "extras" — DROPPED

Janssen CarePath + Skyrizi welcome packet docs, welcome_call touchpoint, 4 audit logs were originally proposed as pre-seeded with FK to Raj. Cascading delete order

For each Patient marked for deletion (dependents first):

  1. Query + DELETE all MedicationRequest?patient=Patient/<id>
  2. Query + DELETE all Coverage?beneficiary=Patient/<id>
  3. Query + DELETE all CareTeam?subject=Patient/<id>
  4. Query + DELETE all AllergyIntolerance?patient=Patient/<id>
  5. Query + DELETE all Condition?subject=Patient/<id>
  6. DELETE the Patient itself

Practitioners + Medications

  • Medications — leave alone. Drug catalog is shared across patients; deletion orphans references. The 732 includes our re-run dupes (cl.products.create isn't idempotent on POST). Acceptable cruft.

Safety nets

  • Dry-run by default — list IDs that would be deleted, no DELETE calls

  • Per-resource error tolerance — if a DELETE 409s, log + continue
