First-Principles Lab · rx-flow migration

The patterns behind the prompt — and where they break.

The rx-flow migration isn't about Claude. It's six software patterns you already know from distributed systems, rediscovered in a new medium. This lab pulls them out, traces their lineage, studies their failure modes, and gives you drills for other projects.

Read linearly, or skip around. Every module includes wins, scars, and a drill you can run tomorrow.

00 · Orientation

The landscape of what you've been doing

Thesis: you're doing distributed-systems engineering where the workers happen to be language models. Every hard problem you've hit — scope, coordination, typing, failure handling, rollout — the field already has answers for. This lab connects the dots.

P · Primer

Twelve primitives you'll lean on

Purpose: the rest of this lab assumes you can reach for terms like cohesion, idempotency, and invariant without flinching. Here they are in one breath each, with an example and a pointer to where they come up. Skim and come back.

The twelve

01 · Cohesion

Things that change together should live together. A module about one concern.

Ex: a File class that opens/reads/closes — cohesive. Same class that also sends email — not.

→ Module 01 (Decomposition)

02 · Coupling

How much one thing has to know about another to work. Looser = cheaper to change.

Ex: typed function signatures = loose. Shared mutable globals = tight.

→ Module 02 (Contracts)

03 · Encapsulation

Hide internals behind an interface. Caller knows what; implementer owns how.

Ex: db.save(order) — caller never sees the SQL.

→ Module 02, 04

04 · Information hiding

Parnas, 1972. Hide design decisions, not just data. Stronger than encapsulation.

Ex: swap Redis for Postgres behind a Storage interface — callers never noticed.

→ Module 01, 06

05 · Invariant

Something always true during operation. The thing you're protecting.

Ex: "every order has a positive total." "Every expert output parses as its schema."

→ Module 02, 05

06 · Idempotency

Doing the thing twice has the same effect as doing it once. Essential for safe retries.

Ex: "mark as read" ✓. "Charge $10" ✗ (use an idempotency key).

→ Module 03 (retries)

07 · Pure function

Same input → same output, no side effects. Easy to test, compose, cache.

Ex: add(a,b) pure. send_email(to) not.

→ Module 02

08 · Determinism

Same conditions → same result. Deterministic code is cheap to reason about.

Ex: SQL query deterministic. LLM at temp=0 close. LLM at temp>0 not.

→ Module 05 (you can only falsify what's repeatable)

09 · Orthogonality

Two concerns don't interfere. Changing one doesn't force a change in the other.

Ex: adding auth shouldn't change how routing works. Telemetry shouldn't change what a function computes.

→ All modules

10 · Abstraction layer

Hides the layer below while exposing something simpler. Good ones don't leak; bad ones do.

Ex: HTTP over TCP over IP. Orchestrator over experts.

→ Module 01, 03

11 · Separation of concerns

Different reasons to change live in different places. Cousin of SRP.

Ex: CSS (how it looks) vs JS (how it behaves) vs HTML (what it is).

→ Module 01

12 · Cost of change

The real performance metric of architecture. How long from idea to shipped?

Ex: microservices faster at 100 engineers, slower at 5. The same architecture has different cost-of-change at different scales.

→ Module 01, 06

How these compose into the six modules

Each of the patterns in 01–06 is a way to turn a primitive into an enforceable shape at system scale. Cohesion becomes a bounded context. Coupling becomes a typed contract. Information hiding becomes a capability-scoped tool. Invariants become release gates. When a pattern feels arbitrary, it's because you've lost the primitive it was protecting — walk back and name the force.

The two words to internalize

High cohesion within

Each module / class / prompt is about one thing. Methods inside refer to each other. There's a single sentence that describes the module's purpose without "and."

Read it as: "does this thing have a single job?"

Loose coupling between

Modules talk through narrow, explicit interfaces. Changing one doesn't force a cascade. The less each side needs to know about the other, the better.

Read it as: "can I change this without touching its neighbors?"

If you remember only two phrases from this primer, make it those. Most architectural arguments are either about raising cohesion or lowering coupling — often both at once.

01 · Decomposition

Monolith → bounded contexts

SRP at system scale Conway's law DDD bounded contexts Information hiding
The force: a single unit serving many domains accretes all their vocabularies and edge cases, and changes whenever any of them changes. Its change rate approaches the sum of all embedded domains; review quality drops toward zero.

CareLink instance

newrx-agent is ~500 lines doing 8 jobs in one context window: patient matching, NCPDP mapping, clinical checks, claim payload, rejection classification, DUR, fulfillment, shipping. Each has its own vocab and rate of change. It became a god class the moment a second domain moved in.

Canonical mapping

DDD
Bounded context: a place where one model is authoritative. "Patient" can mean different things in intake vs. billing — healthy, not a bug.
SOLID
SRP at system scale. A reason to change ↔ a bounded context.
Parnas 1972
Information hiding. NCPDP field mapping shouldn't leak into the shipping prompt.
Conway
Systems mirror the org that built them. Bounded contexts make team topology intentional rather than incidental.

Where it succeeds

Win · Amazon, 2002
API mandate + two-pizza teams
Bezos forced services to expose themselves via contracts. Enabled AWS — but the real win was teams changing at different rates without meetings.
Win · Stripe API
Charge, Customer, Subscription — each its own context
Clean decomposition doubles as UX: the API is learnable one object at a time.

Where it fails

Loss · Prime Video, 2023
Microservices → monolith, 90% cost cut
Step Functions + Lambdas cost more in orchestration than the work. Refolded into one service. Decomposition is justified by team topology or deployability, not by default.
Loss · Segment, 2018
140+ repos to ship one customer feature
Split ran ahead of team structure. Returned to modular monolith.
Loss · Distributed monoliths
Boundaries drawn at the wrong seam
Services that share a DB, deploy together, can't test independently. You paid the tax; got none of the isolation.
Loss · Cargo-cult microservices, ~2015
5-person teams running 30 services
"Netflix does it." Without SRE, tracing, platform — the abstraction costs more than the complexity it buys out of.

The decomposition test — use before splitting

  1. Do the domains have distinct vocabularies? If "patient" means the same thing to both halves, you may not have a boundary.
  2. Do they change at different rates? If they always change together, they're one domain wearing two hats.
  3. Can you write the contract without mentioning the other side's internals? If not, your seam isn't real.
  4. Is there a team (or budget) to own it? Conway's law doesn't negotiate.

Growth path

Monolith → modular monolith → bounded contexts with contracts → independently deployable services → platform + product teams. Walk forward when the current rung concretely hurts, not further.

Deep dive — why 8 experts, not 3 or 30?

The migration chose 8 because those are the coarse phases with distinct vocab: intake uses patient/prescriber, billing uses NCPDP, shipping uses carton/carrier. Below 8 you'd have domains colliding; above 8 you'd split single-phase logic that shares invariants.

Generalization: count distinct vocabularies, not steps. A 30-step workflow may have 4 vocabularies.

02 · Boundaries

Typed contracts at boundaries

Design-by-contract Ports & adapters DTOs Schema-as-docs
The force: decomposition pays off only if the seam is real. A real seam is one you can't silently violate. Types make invariants enforceable at the boundary instead of trusting that two authors agreed in Slack.

CareLink instance

Each expert declares input_schema / output_schema (dotted paths) in YAML frontmatter. BaseStepOutput already exists at carelink/prescriptions/step_schemas/base.py; the migration extends it. The orchestrator calls invoke_expert(name, input); validation runs at the boundary. Schema failure → bounded retry, not silent drift.

What "typed" actually buys you

  1. Locality of failure. Error at the boundary, where you still have context — not three calls deeper.
  2. Parallel change. Expert internals can change freely as long as the schema holds.
  3. Machine-verifiable docs. The schema is the documentation. It can't drift.
  4. Retry affordance. Schema mismatch is a well-defined event. Prose mismatch is a judgment call.

Where it succeeds

Win · Protobuf at Google
Tens of thousands of services, cross-team
Forward/backward-compatible schemas + generated clients in every language. Cross-team changes cheap enough that teams actually make them.
Win · TypeScript in the JS ecosystem
Types at every module boundary
Ecosystem productivity gain driven almost entirely by boundary types — knowing what a function takes matters more than interior soundness.

Where it fails

Loss · Wide types that don't constrain
Dict[str, Any], "JSON blob" fields
Boundary exists on paper; in practice consumers must know internal shape. Ceremony without safety.
Loss · OpenAPI specs that lie
Spec and server drift apart
Hand-maintained specs rot. If it isn't generated from code or enforced by tests, it's a wish, not a contract.
Loss · Thrift/Protobuf fatigue in small orgs
4 PRs across 3 repos per field change
Schema tax is fixed; benefit scales with team count. Under 3 teams you're paying for coordination you don't have.
Loss · Over-typed internals
Types forced all the way down
Types belong at boundaries. Forcing them into leaf logic slows the refactor loop you need for getting logic right. Many enterprise Java codebases died here.
Loss · Contract escape via hallucination (LLM)
When the tool doesn't exist, the model invents a URL
Expert allowlisted fdb-drug-details (a command that didn't exist). Restish errored. The LLM, undeterred, fell through to guessing GET /fdb/drugs/search at the shell. Two separate hallucinated paths shipped to prod before logs surfaced them. The registry-level gate (Module 04) didn't help — it validated the allowlist but not the resolution. A typed contract has to hold at call time, not just at registration time; a model that runs out of legal options produces plausible-looking garbage instead of erroring.

A subtle twist: LLMs as typed functions

The expert is a language model. It can emit anything. Typing the output turns the LLM into "returns ExpertOutput" or "failed, retry." That's the difference between a prose assistant and a composable primitive.

Growth path

String dict → typed DTO → generated stubs → schema registry + versioning → contract tests in CI.

Deep dive — schema-first vs code-first

Schema-first (gRPC, GraphQL SDL): schema is truth. Good for many consumers, many languages, external APIs. Bad for exploratory work — schema changes become ceremony.

Code-first (Pydantic): types live next to code. Good for internal services and evolving shape. Bad when consumers are outside your repo.

rx-flow is correctly code-first. If experts ever go external, generate a schema doc from Pydantic — not the reverse.

Storyboard — Contracts, three frames

1 · Untyped seam. Caller ships a dict; callee walks into it and crashes three function calls deep. The stack trace blames the leaf, not the seam. "Works locally, explodes in prod" lives here.
2 · Typed boundary. Same bad call, different outcome. The DTO is validated at the seam; the interior is free to change shape-agnostically. Locality of failure — error is reported where context still exists.

03 · Coordination

Orchestration vs choreography

Mediator Saga Workflow engines Event choreography
The force: once you have many workers, something decides what runs next. That decision can live in one place (orchestration) or emerge from events (choreography). The common mistake is picking by habit instead of by workload.

CareLink choice

Option B: one orchestrator session calls experts via invoke_expert. Option C (saga / DAG memory across sessions) was deferred, not rejected. At migration stage, traceability and one-place-to-debug beat the parallelism choreography would buy. Option C becomes earnable after workflows show they need it.

Tradeoff matrix

AxisOrchestrationChoreography
DebuggabilityHigh — one place to lookLow — "where did the order go?"
CompensationExplicit in the sagaDistributed rollback is notoriously hard
CouplingEvery flow touches orchestratorServices know only events they care about
ParallelismBounded by orchestratorNaturally parallel
EvolutionEvery change hits orchestratorAdd/remove subscriber independently
Failure modeOrchestrator becomes god objectEvent spaghetti — no one understands flow

Where orchestration succeeds

Win · Temporal / Cadence (Uber)
Durable workflow code that looks like normal code
Uber built Cadence because event choreography made cross-service flows unobservable. Temporal now runs mission-critical workflows for the same reason.
Win · AWS Step Functions
Visual state machines for linear-ish flows
When the flow is genuinely state-machine shaped, centralizing it buys visibility, retries, and timeouts for free.

Where orchestration fails

Loss · Orchestrator as god object
Every new flow = one more branch in one file
rx-flow guards against this: orchestrator knows which expert to call, not how each one works.
Loss · Deep orchestrator nesting
Orchestrator calls orchestrator calls orchestrator
Context splits across sessions/traces; you can't reason end-to-end. Keep orchestrators shallow.

Where choreography fails

Loss · Event spaghetti at scale
"40 services subscribed to OrderCreated"
Removing a subscriber is safe. Understanding what happens is impossible. Teams back-fill orchestration layers later.
Loss · Distributed compensation
Partial failures in unrecoverable states
Saga compensation requires every participant to implement its reverse — forever. Hardest soft-skill problem in distributed systems.

Growth path

Cron + shell → queue + workers → single orchestrator (good default) → saga/workflow engine → hybrid (event bus + orchestrated islands) → event sourcing.

Deep dive — why the orchestrator is a session, not code

Conventional orchestrator = deterministic code. rx-flow's orchestrator = a Claude session with invoke_expert. Why? Routing itself is a judgment: which expert handles which rejection code is fuzzy pattern-matching, not a switch statement. Hard-coding the routing freezes today's taxonomy.

Failure mode: orchestrator hallucinates routing. Mitigations: typed outputs, bounded retries, the falsification gate (Module 05). The design is explicitly prepared to demote routing to code if the gate fails.

04 · Safety

Capability scoping & least privilege

POLA Capability security Default deny Blast radius
The force: any actor with more authority than its job needs is a latent incident. The question isn't "can it do this?" — it's "what's the smallest set of things it can do?" Addition is easy; subtraction is the invariant.

CareLink instance

Each expert declares allowed_tools and allowed_restish_commands. Today's _build_options(tools=) at service.py:508 is additive-only (list(set(base_tools) | set(tools))), silently widening every expert. PR 0.1 makes it subtractive — opt down from a baseline, not up. Bash(prefix:*) is already natively supported; the per-expert enforcement is a one-line comprehension.

def _build_options(tools=None, ...):
    base_tools = ["Read", "Grep", "Bash", "WebFetch", "Write", "Edit", ...]
    # today — union; "tools=[...]" only *adds*
    return AgentOptions(tools=list(set(base_tools) | set(tools or [])))

# calling with tools=["Read"] still yields every base tool.
# intent "restrict to Read" silently becomes "add Read to everything".
def _build_options(tools=None, ...):
    base_tools = ["Read", "Grep", "Bash", "WebFetch", "Write", "Edit", ...]
    # intersection if tools provided, else default
    allowed = [t for t in base_tools if t in tools] if tools is not None else base_tools
    return AgentOptions(tools=allowed)

# tools=["Read"] yields exactly ["Read"]. caller opts *down*.

The shape matters more than the code. Additive defaults drift toward "everyone has everything" — adding is a PR, subtracting is a negotiation.

Canonical mapping

Saltzer 1975
Principle of least authority. Every component should operate with the least privilege necessary.
Capability
Authority is an unforgeable reference you hold, not an identity matched against an ACL.
Unix
chmod / setuid / chroot — oldest POLA in wide deployment, warts and all.
Web
CSP, same-origin, CORS — layered POLA retrofitted onto a trusting substrate. Retrofitting is always harder than starting with it.

Where it succeeds

Win · AWS IAM (used well)
Resource-scoped roles; no * actions
Every compromised key in history would have done less damage with tight scoping. The model is right; the inconvenience makes people wildcard.
Win · Container capabilities / seccomp
Drop privileges at the kernel boundary
A container that can't execve or ptrace turns a 0-day into a log line.

Where it fails

Loss · Java's SecurityManager
Deprecated in JDK 17
Theoretically perfect. Almost nobody configured it right. POLA APIs have to be usable, not just sound.
Loss · "Tighten later"
Production *:* roles that outlive the deadline
Default-permissive + "we'll fix it" ≈ "we'll never." The only ratchet is default-deny.
Loss · Over-scoping blocks incident response
Oncall can't logs:GetLogEvents
POLA without break-glass is its own incident. Build auditable elevation paths — not no elevation paths.
Loss · Capability tokens leaking
Unforgeable, shareable, accidentally shared
Capabilities that survive the session become bearer tokens. Lifetime matters as much as scope.

Design APIs so the "normal" path shrinks authority. Widening should be a conscious act.

Growth path

Root everywhere → RBAC → ABAC / policy-as-code → capability tokens with lifetimes → zero trust.

Deep dive — UserQuestionGate as accidental POLA

The existing UserQuestionGate has a single _pending slot. Experts calling AskUserQuestion would clobber it. The design enforces "only orchestrator talks to human" by accident of the primitive.

Lesson: the safest capability design is often a primitive that can't be misused — not an explicit policy. When policy can be encoded in shape, it can't drift.

Storyboard — Capabilities, three frames

1 · Ambient authority. Every tool is reachable by default. A compromised expert — or a misfiring prompt — can touch anything on the allowed list. Blast radius = the union of all authorities the process ever needed.
2 · Default deny. The intake expert gets only what intake needs. Compromise narrows — an exploit of this expert can't reverse a claim or delete a patient. Subtractive defaults are a ratchet; additive defaults are a leak.

05 · Release engineering

Falsification as release gate

Canary Dual-write & diff Scientist gem Popper
The force: author confidence is a prediction; production evidence is the result. A good release gate is a falsifiable hypothesis, not a vibe. "Ship when X days of traces are clean" is science; "ship when it feels ready" is not.

CareLink instance

Phase 6 deletes the monolith — a one-way door. It gates on "14 consecutive days of clean production orchestrator traces." Regression → halt and roll back. The burden is on the new system to fail; if it doesn't, we proceed.

Canonical mapping

Popper
Falsifiability. Confidence comes from surviving attempts to break it. Releases gained confidence by surviving traffic.
GitHub Scientist
Dual-run old & new, return old, log diffs. Divergences fixed at leisure. Famous use: GitHub's permissions rewrite.
Canary
Gradual rollout with automated rollback on metric regression.
Chaos
Actively try to break your system in production under control. Netflix's Chaos Monkey.

Where it succeeds

Win · GitHub Scientist, 2016
Rewriting permissions without an outage
Old served users; new ran parallel; diffs logged. Fixed at leisure. When diffs → 0, flipped the switch. The gate is only as good as the comparison function.
Win · Google SRE's error budget
Numeric gate baked into the org
Under budget → ship faster. Over → slow down. The gate is the dev/SRE agreement — the math is easy.

Where it fails

Loss · Knight Capital, 2012
$440M lost in 45 minutes
Old feature flag repurposed; deploy missed one of eight servers. No gate, no canary, no diff. The gate you didn't build cost the company.
Loss · Healthcare.gov, 2013
Scale-tested in a room, launched to a nation
No canary, no staged rollout. The gate was a press release.
Loss · Fake observability
"We have metrics" ≠ "metrics would catch the bug"
A gate on success count can pass while new path silently corrupts data. Decide the regression signal before shipping.
Loss · Schema-locked migrations
Rollback impossible because DB changed
A gate without a real back door is one-way by accident. Migrations without a dual-write window remove your option.
Loss · The gate that only logged
Validator exists, validator speaks, nothing blocks
Two flavors, observed in one session. (1) The startup registry validator raised on bad expert commands — but the lifespan wrapped it in except Exception: logger.warning(...) and the server came up broken. (2) The test that was supposed to catch exactly this supplied the fake command names to the validator's "known" set, so the gate passed because the fixture taught it the bug was real. Fixtures are config, not decoration. A gate that logs instead of fails is a comment; a gate fed the answer key is theater. Decide at design time whether a signal halts or narrates — the default should be halt.

The hypothesis-before-deploy drill

  1. What are we claiming? A falsifiable predicate: "orchestrator completes happy-path newRx < 90s, < 0.1% prose-instead-of-JSON retries." Not "it works."
  2. What would disprove it? Define signal & threshold in advance — or you'll rationalize whatever you see.
  3. What's the rollback? Cheap flag, redeploy, or expensive migration? Cost of rollback = cost of being wrong.
  4. How long? Short windows catch quick bugs; long windows catch slow drift. 14 days covers two weekly prescribing cycles.

Growth path

Ship on vibes → staging → feature flags → manual canary → automated metric-based canary → experiments-as-infrastructure → continuous production verification.

Deep dive — why "clean traces" not "same answer as monolith"

Scientist-style byte-equal diff would fail for reasons that don't matter — the old prompt is non-deterministic. "Clean traces" is a weaker gate on purpose: acceptable time, retry rates, no stuck states, no schema-output failures. It doesn't claim "same decision."

General lesson: choose the strongest gate whose comparison function you can implement. Weakening is honest; using a strong-sounding gate you can't compute is dangerous.

The meta-pattern

Make invariants enforceable, then walk them forward. Boundaries become types. Routing becomes a scoped tool. "Safe to ship" becomes a predicate. "Don't touch legacy" becomes an exact 2-line edit. Whenever you catch yourself relying on a human to remember, look for the shape that would remember for you.

What's genuinely new here (LLMs)

  • The worker is non-deterministic. Every pattern above works better with typed contracts — LLMs make them mandatory.
  • The prompt is code. Versioned, reviewed, typed at I/O, tested, release-gated.
  • Routing can be reasoning. Doesn't have to be a switch statement — but can drift, so gates matter more.
  • Tool scoping is authority design. What an LLM can call is what it can do. POLA is a correctness primitive, not a security afterthought.
  • Hallucination is the default escape hatch. When the model can't do a thing cleanly it produces something plausible-looking instead of erroring. An undefined restish command doesn't raise — it becomes a guessed URL. Contracts have to hold at call time with real fallback behavior, not only at registration time as static validation. (See Module 02.)
  • The LLM retries without memory. Each turn starts fresh; the model will call create_product(ndc=X) as if new whether X exists or not. Write paths consumed by LLMs must be idempotent by natural key at the storage layer (DB ON CONFLICT, upsert semantics) — "the caller checks first" is a convention, not a guarantee, and the caller has no memory to keep it.

Everything else is software engineering you already know, rediscovered in a new substrate. That's the good news.

10 · Observability

Traceability — who invoked who, and was that right?

OpenTelemetry Dapper lineage Event sourcing Observability
The force: you can only optimize what you can see. Traceability is the precondition for the falsification gate (Module 05) and the explore/exploit loop (Module 11). Without it, every other pattern above degrades to opinion.

The minimal unit — a span

Every call is a span. Every related set of spans is a trace, tied by a trace ID. Context — the trace ID + parent span ID — propagates across every call, tool invocation, and event publication. When it hits an async boundary (a queue, an Inngest event), you carry the correlation ID so the downstream work joins the tree.

CareLink instance

Each invoke_expert emits a structured span: parent trace ID, expert ID, input feature hash, output shape, latency, retry count, success/fail. Tool calls become child spans. Inngest events carry the correlation ID so the async pipeline re-joins the tree. Phase 4.2 (parent/child sessions, deferred-optional) is the richer-observability version of the same idea.

Four things a trace must carry

Causal structure

Who invoked who. Parent span ID on every child. You should be able to render the trace as a tree.

Attributes, not just names

High-cardinality fields: user ID, case type, input hash, retry count, confidence score. Not just "expert=billing."

Events inside a span

Schema mismatch, retry fired, tool errored, cache hit. The granular story of the span, not just its outcome.

Outcome label (post-hoc)

Tied days or weeks later when the business outcome is known. The trace is metadata; the label is what makes it data.

The hardest part: outcome labels

The outcome of a newRx trace isn't visible in the trace itself. It happens later — a claim is approved or rejected, a patient receives medication or doesn't, a pharmacist overrides the DUR flag. You need a correlation ID from the trace to persist on the business record, so you can join back weeks later.

Trace without outcome label = ambient noise. Trace with outcome label = labeled training data.

Where it fails

"We sample 1%, miss the rare bug"
Uniform sampling loses tail events. Use tail-based sampling (keep all errors + sample the rest) or at minimum force-sample on error signals.
Loss · Traces without outcomes
Beautiful waterfall charts, no idea if the trace was correct
Latency is not quality. Without outcome labels, you optimize for speed and miss correctness regressions. Always ask: "where's the ground truth for this trace?"

Traces are often in a separate data class from production DBs. Inlining PII breaks that. Hash or redact at emission; keep a separate secure join table if you need to rehydrate.
Loss · "We have logs"
Per-process logs, no correlation, no tree
Logs say what happened in one place. Traces say who invoked whom across places. Logs ≠ traces. Most teams discover this on their first really bad incident.

The three pillars, in one sentence each

  • Logs: what happened, in one place.
  • Metrics: how often, aggregated.
  • Traces: who invoked whom, across places.

You need all three. Most AI-heavy systems ship with the first two and discover they need the third at the first serious regression.

Why this matters doubly for LLMs

  • LLM decisions are opaque from code alone — the routing emerges at runtime, it isn't in a switch statement.
  • Without traces, you cannot tell why the orchestrator chose expert X over Y.
  • Without outcome labels, you cannot tell if that choice was right.
  • Opaque AI = unfalsifiable AI = untrustworthy AI. Traceability is not a nice-to-have; it's the thing that makes the rest of the lab possible.

Growth path

Scattered print statements → structured logs with request IDs → metrics + logs + alerting (RED / USE) → distributed traces with propagation (OpenTelemetry) → high-cardinality observability with outcome labels → event sourcing for full replayability.

Deep dive — the "was this decision right?" label

For a rx-flow trace, the label could be:

  • Automatic: claim accepted by payer, no retries, no manual correction → label = 1. Rejection, retries, or manual override → label = 0 (with reason code).
  • Human: a pharmacist marks the session "correct" or flags a specific step as wrong. Adds a high-quality but expensive label.
  • Shadow comparison: for a subset, run the old monolith on the same input in parallel; compare decisions. Divergences are labeled for review.

All three contribute to the dataset the explore/exploit loop (Module 11) consumes. The labeling system is the bottleneck — invest in it early, not late.

11 · Evolution

Explore vs. exploit — avoid premature path-freezing

Multi-armed bandits Contextual bandits Policy iteration Goodhart's law
The force: early routing wins look like correctness but are often overfits to a small sample. Hardcoding them freezes today's taxonomy and closes off learning. The right move is to keep decision trees open, log every choice + outcome, and let weights emerge from evidence — then harden only the paths whose posterior is actually tight.

The trap — "it worked three times in a row"

Day 3 of the new orchestrator: in every trace, rejection-code 70 was routed to rx-flow-billing, and every claim succeeded. The instinct is: "codify it, it's clearly correct." Don't. Three is a sample. The cases so far may not represent the long tail. You've observed what succeeded; you haven't observed what would have happened if we'd routed differently. That counterfactual is what makes a decision truly "right."

with data novel cases still reach the open path

The explore–exploit tradeoff, stated plainly

  • Exploit: use the path you already believe works. Fast, confident, blind to news.
  • Explore: try a different path to learn whether it's better. Slower per-case, accumulates evidence.
  • Pure exploit (hardcode from day 1): maximum confidence, zero learning. You freeze your priors.
  • Pure explore (random routing forever): maximum learning, never benefits from what you learned.
  • The real answer: a gradient. Heavy explore early (no data); shift to exploit as evidence accumulates; keep some baseline explore forever (the world changes).

Four strategies, in order of complexity

StrategyHowWhen to reach for it
Epsilon-greedy Pick the current-best path with probability 1−ε; random otherwise. Decay ε over time. When you have a clear win metric and low-stakes arms. Cheapest bandit, ships in a day.
Thompson sampling Maintain a posterior over each arm's success rate; sample from each and pick the max. When you want self-regulating exploration — uncertain arms get explored more automatically.
UCB (upper confidence bound) Pick the arm with the highest "mean + uncertainty" score. Exploration = uncertainty bonus. When you want a deterministic bandit with explicit confidence math.
Contextual bandit Score arms based on context features (case type, payer, drug class). "Best arm" depends on the input. When different arms win in different conditions (nearly always, for routing).

What this looks like for rx-flow

  1. Phase 1 — pure exploration (orchestrator-only). The orchestrator decides every routing. Every decision + outcome is logged (Module 10). No fast paths. No switch statements.
  2. Phase 2 — posterior inspection. After N runs, aggregate by (case context, expert chosen, outcome). For each (context, expert) pair: is the posterior success rate tight and high? Or wide and uncertain?
  3. Phase 3 — seeded exploitation. For (context, expert) pairs with tight, high posteriors: seed the orchestrator's system prompt with exemplars ("for rejection-code 70, billing expert has 98% success over 400 cases"). Don't hardcode — inform. The orchestrator can still route otherwise for novel shapes.
  4. Phase 4 — fast path, with fallback. Only when a (context, expert) pair is extremely tight AND the context is narrow, add a fast path that bypasses the orchestrator. Keep a minimum exploration rate (~2%) that still routes through the orchestrator, so novel cases stay visible.
  5. Phase 5 — keep watching. If the distribution shifts (new drug class, new payer), the fast path's success rate drifts. You'll see it in traces (Module 10). Unfreeze.

Where it succeeds

Win · Netflix / YouTube ranking
Billions of bandit decisions per day
Contextual bandits drive row ordering, thumbnail selection, next-up picks. Hand-tuned rules were abandoned a decade ago because learned weights, given enough traces, are better.
Win · Google Ads auction

A small % of impressions always explore new creatives. Without it, the system collapses to a local maximum and stops learning about shifting taste.
Win · AlphaGo / AlphaZero
Weights on a decision tree from self-play
MCTS doesn't search all of Go. It allocates search budget by "visits × value," which is exploration × exploitation over a tree. Same principle, extreme scale.
Win · Duolingo spaced repetition
Which word do we review next?
Per-user, per-word bandits. Explore "might have forgotten"; exploit "definitely needs review." Pure exploitation would bore; pure exploration would exhaust.

Where it fails

Loss · No traces → no data
"Let's explore" with nothing to learn from
Without Module 10, you're not exploring — you're improvising. The loop is trace → label → weight. Break any link and the system is noise.
Loss · Confounded outcome metric (Goodhart)
Optimize "expert finished" → optimize for anything that finishes
If the outcome label is "expert returned valid JSON," the orchestrator learns to route to the most compliant expert — not the most correct. Label the business outcome, not the implementation.
Loss · Early-winner artifacts
3 wins ≠ 100 wins
Posterior width matters more than posterior mean. Tight, narrow, high = harden cautiously. Wide = keep exploring. Don't let anecdotes promote themselves to architecture.
Loss · Distribution shift
New drug class, new payer, new rejection code
Yesterday's training data doesn't match tomorrow's reality. Without a baseline explore rate, the system is blind to novelty. Keep ~2% minimum exploration, forever.
Loss · High-stakes arms
"Every exploration pull is an incident"
Medical decisions, financial trades. You can't A/B user safety. Here: explore offline (simulation, shadow traffic, historical replay), exploit online. The loop still works — it just moves.
Loss · Hardcoding the 90%
The 10% becomes invisible
Fast path handles 90%; 10% doesn't match the fast path's signature and is silently mis-routed. Always keep an open path for novel cases, and keep watching it.

The meta-principle

Decisions you can't undo are the ones to defer. Hardcoding a routing rule is a commit to a hypothesis. An open orchestrator is a hedge. Pay for the hedge until the evidence makes it clearly unnecessary. When you do harden, leave a minimum exploration rate running in parallel — the world shifts, and you want to notice.

Growth path

Pure orchestrator (no priors) → prompt-seeded with exemplars (weak priors) → epsilon-greedy fast path + orchestrator fallback → contextual bandit with features → learned policy (small classifier or fine-tuned routing model).

Each rung demands more traces, more labels, more infrastructure — and pays back in correctness and speed. Walk up when the evidence is there, not because the rung is higher.

Deep dive — how "weighting" shows up concretely

"Add weights to the decision tree" could mean many things. Concretely, in rx-flow, it might be any of:

  • Prompt exemplars. The orchestrator's system prompt grows a "what has worked" section pulled from the top-N most-similar past traces, with outcomes. No ML; just retrieval-augmented orchestration.
  • Routing prior. A lightweight table mapping (context features) → (posterior over experts). Orchestrator reads it, treats it as a suggestion. Can override when the case looks unusual.
  • Classifier-as-assistant. A small model (scikit-learn, logistic regression) predicts expert from features; orchestrator calls it and either agrees or overrides. Classifier retrained weekly.
  • Fast-path switch. For the tightest (context, expert) pairs with 99%+ posterior, bypass the orchestrator. Fall back on any uncertainty signal.

They differ in infra cost and traceability. All of them need Module 10 (traces + outcome labels) to work. None of them should be implemented before the 14-day falsification gate (Module 05) passes — you need the baseline dataset first.

12 · Dispatch

Subagent orchestration as engineering practice

Map-reduce Fork-join Parallel dispatch Cold-start contracts
The force: when a problem decomposes into slices that each fit one worker's working memory, parallel dispatch wins on wall-clock and on cost compared to sequential work. The discipline is at the seam — picking what's genuinely independent, packaging the cold-start context, and designing the integration step before you fan out.

The principle

Not every task should be done by the agent in front of you. Three questions, in order: what slices are genuinely independent? what does each slice need as cold-start context? what does integration look like? If you can't answer all three before dispatching, you'll spend more time stitching outputs back together than you saved by parallelizing them. The pattern is older than LLMs — it's fork-join with humans as the workers and a senior engineer as the dispatcher. The novelty is that you can now fork-join over agents that each carry their own context window.

CareLink instance

Three concrete dispatches in this cycle:

  • PR 1 + PR 2 in parallel. Both worktrees branched from 6173360. PR 1 added 6 columns to agi_sessions; PR 2 added AuthClaims with org_id extraction. Zero shared files. Zero shared imports. The integration step was git cherry-pick in commit order, no merge logic. Two agents finished in roughly the time one would have, and neither had to know the other existed.
  • FUSE deep-dive — 4 parallel Opus reads. One commit (b0bb495, 3,864 LoC + 926 LoC of design docs) was sliced four ways by directory: implementation, tests, infrastructure scripts, design narrative. Each slice was small enough for one agent. Synthesis happened in a single fifth pass that produced the final fuse-sandbox-deep-dive-2026-05-01.html. The slicing was deliberate — sliced by vocabulary, not by file count, so each agent's context held a coherent surface.
  • Test agent + verification agent post-staging. Two agents with deliberately different objectives. Test agent ran the suite; verification agent audited the staged diff against acceptance criteria. They have complementary failure modes: the test agent caught the cold-revive monkeypatch regressions in PR 4a (the verification agent missed them because it focused on the diff); the verification agent caught the spec deviation around db: Database threading in PR 3 (the test agent missed it because the new behavior passed its tests). Always dispatch both. They fail in different directions.

Where it succeeds

Win · PR 1 + PR 2 parallel dispatch
No shared files, same baseline, mechanical merge
Both PRs grounded on commit 6173360. The cherry-pick into main was a 30-second operation per PR. The orchestrator never had to mediate a conflict — that's the signature of a real seam.
Win · Code reads larger than one window
3,864 LoC sliced 4 ways
Whole-commit context wouldn't fit in one Opus invocation; sliced by vocabulary it fit four. The synthesis pass was the only one that needed to see all four reads — a much smaller surface than the original code.
Win · Complementary auditors
Test runner + diff auditor in parallel
Each catches what the other misses. Both ran the same input; their questions differed. Dispatching one and trusting it would have shipped a regression.

Where it fails

Loss · "Independent" with hidden state at the seam
PR 1 created a column; PR 3 consumed it
PR 3 needed the lease columns from PR 1 to compile. Dispatching them in parallel would have forced PR 3 to mock the column or fail — neither produces useful work. Sequential, no choice. Test for independence is "would either compile if the other had not run?" — if no, you don't have parallelism, you have ordering.
Loss · Integration cost > parallelism win
4 agents × 1,000 LoC ≠ 1 agent × 4,000 LoC
Four independently-authored 1,000-line diffs may pull in conflicting style choices, redundant helpers, mismatched assumptions. The integration step costs more than the parallel speed-up bought. Dispatch to read in parallel; consider hard before dispatching to write in parallel.
Loss · Cold-start prompt isn't self-contained
"Make the change like we did last time" sent to a fresh agent
The agent has no "last time." Without file paths, repo state, baseline commit, and acceptance criteria spelled out, the agent will guess plausibly and ship something that misses the actual ask. Cold-start is everything; assume amnesia, write to it.
Loss · Dispatcher hallucinates the seam
Slicing by file count instead of by vocabulary
"Each agent reads 7 files" sounds even. But if files 1-3 are infrastructure and 4-7 are application logic, neither slice has a coherent surface. Slice by what changes together, not by what's convenient to count. Same lesson as Module 01.

13 · Refactor

Refactoring under a live codebase

Strangler fig (class scale) Facade Attribute lookup semantics Pass-through proxies
The force: when you split a god class into N classes behind a facade, callers and test fixtures depend on the attribute lookup semantics of the old surface — not just its method signatures. Preserving "obj.method(...) works" is easy; preserving "monkeypatch.setattr(obj, '_method', ...) still reaches the new internals" is the part that bites.

The principle

A facade is more than method delegation. It must preserve the shape of the object as callers see it — including the dynamic shape (setattr, getattr, hasattr, monkeypatch, pickle). The cleanest split makes __getattr__ route reads to the right inner class and __setattr__ route writes to the inner class that owns that attribute. Without the write side, a caller that mutates the facade silently mutates the wrong object — and the failure surfaces three function calls deep, far from the seam.

CareLink instance

PR 4a split AGIService (2,223 LoC) into:

  • service.py — facade, 444 LoC. Holds no logic; routes to backend or worker.
  • backend.py — BackendAGIService, 858 LoC. Zero claude_agent_sdk imports (verified by a test that grep-asserts).
  • worker/service.py — WorkerAGIService, 1,531 LoC. Owns all SDK calls.

Method delegation was straightforward. Two failure modes surfaced during verification:

  1. Cold-revive monkeypatch on _build_options didn't reach the worker. The test does monkeypatch.setattr(agi, "_build_options", mock). The old method lived on AGIService, which was now a facade. The worker's body still calls self._build_options(...), where self is the worker — not the facade the test had patched. Two objects, two attribute lookups, no sharing.
  2. Module-bound import of _import_sdk didn't reach the worker callsite. The worker had from carelink.agi.service import _import_sdk. monkeypatch.setattr("carelink.agi.service._import_sdk", ...) updated service's namespace — but the worker's local binding was frozen at import time. The patch reached one module's name; the worker held a stale pointer.

The fixes are general enough to be worth memorizing.

Pattern 1 — facade __setattr__ pass-through

The facade's __setattr__ checks: does the worker have a class-level attribute by this name? If yes, write to the worker. If no, write locally on the facade. Read-through (__getattr__) alone doesn't suffice because read-through fires only on missing attributes — once the test sets the attribute on the facade itself, the worker's body still reads its own self. and never sees the facade.

class AGIService:
    """Facade. Routes to worker or backend; preserves attribute lookup semantics."""

    def __init__(self, ...):
        self._worker = WorkerAGIService(...)
        self._backend = BackendAGIService(...)

    def __getattr__(self, name):
        # Called ONLY when normal lookup fails — i.e. the attribute
        # isn't on the facade. Bail on internals to avoid recursion.
        if name in ("_worker", "_backend"):
            raise AttributeError(name)
        if hasattr(self._worker, name):
            return getattr(self._worker, name)
        raise AttributeError(name)

    def __setattr__(self, name, value):
        # Internals set normally on the facade.
        if name.startswith("_") and name in ("_worker", "_backend"):
            object.__setattr__(self, name, value)
            return
        # If the worker class has this attribute, propagate the write.
        # This is what makes monkeypatch.setattr(facade, "_build_options", mock)
        # reach the worker's self._build_options(...) callsite.
        worker = self.__dict__.get("_worker")
        if worker is not None and hasattr(type(worker), name):
            setattr(worker, name, value)
            return
        object.__setattr__(self, name, value)

The trick is "if the worker class has this attribute" — written before the instance attribute is set, so we look at the class, not the instance. This makes the pass-through stable against the worker's own __init__ ordering.

Pattern 2 — module-attribute lookup for monkeypatch transparency

from x import y creates a local binding that's a snapshot of x.y at import time. monkeypatch.setattr("x.y", new) rebinds x.y but doesn't reach the consumer's local y. The fix: import the module, call attributes off it at runtime.

# worker/service.py
from carelink.agi.service import _import_sdk

def _hydrate_session(...):
    sdk = _import_sdk()       # local binding — frozen at import time
    ...

# test
monkeypatch.setattr("carelink.agi.service._import_sdk", fake_sdk)
# the worker's _import_sdk reference is the OLD function; patch missed it
# worker/service.py
from carelink.agi import service as _service_module

def _hydrate_session(...):
    sdk = _service_module._import_sdk()   # runtime attribute lookup
    ...

# test
monkeypatch.setattr("carelink.agi.service._import_sdk", fake_sdk)
# the worker now resolves _import_sdk via _service_module at call time
# the patch on service._import_sdk reaches it

Where it succeeds

Win · Django QuerySet refactor
Multi-class internals behind a single attribute surface
Django's QuerySet has been quietly composed of many helpers for a decade. Callers rarely notice because the attribute lookup semantics are preserved — every method appears on the QuerySet, even if its body lives elsewhere.
Win · Strangler fig at class scale
Same shape; can roll back to the monolith by removing the facade
The facade is the bridge from Module 06. Until callers migrate to the inner classes, the facade is load-bearing. Once they do, you delete it. The pattern is reversible.

Where it fails

Loss · Read-through only
__getattr__ without __setattr__
Tests do monkeypatch.setattr(facade, "_method", ...); the patch lands on the facade; the worker's body never sees it. The test passes when run in isolation (the stub is called directly) and fails when run as part of the flow. The hardest kind of bug — green diff, red CI on integration.
Loss · __getattr__ recursion
Forgot to bail on _worker in __getattr__
Python calls __getattr__ when a normal lookup fails. self._worker in __getattr__ calls __getattr__ recursively if _worker hasn't been set yet — infinite loop, stack overflow before any test prints. Always check self.__dict__ for the well-known internals; bail on missing.
Loss · Hidden state coherence bugs
Pass-through masks a write going to the wrong object
If the facade and the worker both have a _cache, the pass-through routes the write to the worker — but the facade's _cache is the one that should have been updated. Pass-through is a tool, not an answer; if the inner classes share state, you still owe a coherent state model.
Loss · Facade lives forever
Callers never migrate to the inner classes
If the facade preserves the surface perfectly, there's no pressure to migrate off it. The "temporary" scaffolding becomes architecture. Pair the facade with a migration deadline (Module 06). Pass-through is a bridge; bridges must die.

Growth path

Junior — knows method delegation: "def foo(self, x): return self._inner.foo(x)". Senior — knows the dynamic shape: __getattr__, __setattr__, module-attribute lookup, and the test fixtures that depend on each. Architect — designs the split so the pass-through is scaffolding, removable once callers migrate to the inner classes directly. Every __getattr__ is a temporary patch; if it's still there a year later, the migration stalled.

14 · Decision durability

Living plan documents

ADRs RFCs Status badges Decision archaeology
The force: architectural docs go stale within weeks of being "finished." A doc written once and never edited becomes confidently wrong — it tells future readers what was true at one moment, not what's true now. The only docs worth maintaining are the ones designed to be edited as decisions ratify, PRs ship, and understanding evolves.

The principle

Treat the canonical plan as code: status badges, version control, machine-checkable acceptance criteria. When a decision is ratified, the doc flips from Pending to Decided with the chosen option highlighted and the runner-up archived in a fold-away. When a PR ships, its card flips to Merged with a Shipped paragraph capturing what actually landed (which often differs from what was planned). The doc's structure forces clarity — you can't claim "we're done" without flipping a status badge, and you can't flip a badge without writing what was decided.

CareLink instance

agi-implementation-plan.html evolved across a multi-day session:

  • Day 1: 8-PR plan + 7 open decisions + 9 follow-up questions + 8 FUSE-specific questions. Status banner reads "All decisions Pending."
  • Day 2: Decisions resolve one by one. Each "Pending" badge flips to "Decided" with a green Resolution box capturing the chosen option, the date, and the rationale. Runner-up options stay in the card but are no longer load-bearing.
  • Day 3: PR 4 is realized to be too large for a single dispatch; the card splits into 4a/4b/4c. PR 4a ships and gets a "Merged" badge + Shipped paragraph noting two surface deviations (__getattr__/__setattr__ pass-through, _import_sdk module-attribute indirection) that weren't in the original spec.
  • Day 4: Decision 3 is strengthened mid-session — bucket-per-org → GCP project-per-org. The Resolution box gets edited in place (with a "(strengthened 2026-05-04)" date stamp), not replaced or duplicated. Future readers see the path the decision took.

The status banner at the top of the doc tells a new reader, in 30 seconds, what's locked vs. in-flight. Without the banner, every reader reconstructs that state from the cards.

15 · Investigation

Pre-flight investigation — don't assume

Hypothesis check State verification Cheap before expensive Forensic discipline
The force: every operation rests on a premise about current state. When the premise is wrong, the operation produces output that's hard to attribute. Pre-flight investigation costs minutes; acting on a wrong premise costs hours. The discipline isn't being clever — it's reliably running the cheap check before the expensive action.

The principle

Before any non-trivial change, verify the assumed starting state. git rev-parse HEAD, alembic current, \d table, ls -la, git log --all -S 'mystery-string'. The checks themselves are mechanical and cheap. The discipline is to actually run them — and to read what they say, instead of skimming for the answer you expected. Most disasters are preceded by a 30-second check that, run before the action, would have prevented them.

CareLink instance — the orphan-revision saga

At the start of the shipping cycle, the local DB was stamped at alembic revision e1f3a5b7c9d2. That revision didn't exist in any branch's migration files. A naive agent would have run alembic upgrade head, hit a confusing "revision not found" error, and either reported it as a blocker or — worse — destroyed local state trying to "fix" it.

What actually happened:

  1. git log --all -S 'e1f3a5b7c9d2' — found the revision string in a stash from a prior agent. The stash had a note: "DDL is in place; fix with stamp."
  2. Schema-diffed local DB against the model on a fresh DB. Confirmed equivalence — the columns exist, the indexes exist, the structure matches.
  3. alembic stamp --purge c4d6e8f0a2b3 — moved the stamp to the closest valid ancestor without re-running the chain.
  4. PR 1 then ran alembic upgrade head cleanly, applying the new lease-columns migration on top.

Total investigation time: ~5 minutes. Time saved: hours. The "fix" was knowing what to read; the work itself was three commands.

Where it succeeds

Win · The orphan-revision saga
Read the stash before running migrations
A 30-second git stash list revealed a prior agent's deliberate workaround. Without that read, the next agent would have re-discovered the chain bug the hard way.
Win · "What baseline are you on?"
PR 3 dispatch from 0c02964, not HEAD
Naming the exact commit hash in the dispatch prompt avoided the agent picking up uncommitted local changes from a sibling worktree. Worktrees and stashes hide state from each other; checking is free.
Win · Verifying staged blob materializes to working tree
After git reset --soft, the index was right, the file wasn't
A wc -l on the supposedly-edited file revealed the working tree had reverted to old content despite the index being staged correctly. git cat-file -p <blob> > file rematerialized it. Caught by reading file size, not by trusting the operation reported success.

Where it fails

Loss · Investigation as ritual
Running checks but not reading them
If git status runs and the agent skims for the answer it expected, the unexpected lines (the 4 untracked docs, the staged file from a sibling) get ignored. Investigation only works if you read what it tells you, even when boring.
Loss · Investigation as substitute for understanding
Knowing what state is, not why
Running alembic current tells you the revision. It doesn't tell you why the chain has a drop-before-create bug at ix_tasks_authored_on. The investigation is necessary, not sufficient — follow up with "what produced this state?" or you'll just defer the bug to next time.
Loss · Skipping when "obvious"
"It's just a small change"
The cheap check skipped is exactly the one that would have caught the dependent state. Pre-flight must be habit, not judgment — judgment is what produces the disaster you're trying to prevent.
Loss · Wrong working directory
cd /Users/sangeet/src/carelink-ui-mockups followed by a monorepo command
The Bash tool's cwd persists. A cd for one operation leaks into the next call. git worktree remove ran from the mock-ui directory and failed silently (or more often, found nothing to remove). Use git -C <abs path> for cross-worktree operations; never trust the inherited cwd.

Pre-flight checklists by operation

OperationCheap checks before you act
DB migration alembic current · alembic heads · count migrations under versions/ · schema-diff vs model on fresh DB · search stashes for the revision string if it's unrecognized
Cherry-pick git status on source AND target worktrees · git rev-parse HEAD on both · git log --oneline <commit>^..<commit> to confirm the SHA exists where you think it does
Worktree creation git worktree list first · check that the target directory doesn't already exist · check the source branch is up to date · verify the baseline commit is reachable
Subagent dispatch Read the worktree's actual state once from the dispatcher's perspective: git log --oneline -5, ls on the file directories the agent will touch, confirm forbidden zones haven't been pre-modified
Edit a file wc -l <file> before and after · if the file has been touched recently by a different process (worktree, agent), verify your edit is editing the version you think it is

Growth path

Junior — runs git status before starting work. Senior — runs git status AND understands what each line means (untracked vs ignored vs modified vs staged). Architect — designs systems where the answer to "what state is this in?" is unambiguous: alembic_version table, status enums, well-named git branches, HEAD commits that summarize their content. The best systems make pre-flight investigation easy because their state is legible.

16 · Audit

Reading someone else's code

Cherry-pick discipline Severity rubrics Sharp-edge auditing Decision-ready output
The force: code you didn't write hides assumptions the author had and you don't. When you inherit a branch — to cherry-pick from, evaluate, or integrate — read it with structured suspicion. Capture every "this looks fragile" as a sharp edge with a severity rating. Without severity, "take everything" or "take nothing" are your only options. With severity, "take the library, fix S1-S4, defer S5-S10" is the right call.

The principle

Auditing isn't reading harder; it's reading differently. You're not asking "do I understand?" — you're asking "what would fail, and how badly?" Each finding gets named, located (file:line), severity-tagged, and paired with a fix sketch. The discipline is in the output shape: a 10-item list with no severity is useless because it can't be acted on; a 3-item "data-loss class" list with fix sketches is decision-ready. Severity rubrics force calibration — you can't rate everything "high," because then "high" means nothing.

CareLink instance — the FUSE deep-dive

The cycle inherited a long-running branch (inbound-voice-fuse-sandbox @ b0bb495): 21 files, 3,864 LoC of implementation, 926 LoC of design docs, written by a different engineer over a different time period. Decision required: cherry-pick into PR 5? All of it? Some of it? Skip it?

4 parallel Opus subagents read non-overlapping slices (Module 12). Output synthesized into fuse-sandbox-deep-dive-2026-05-01.html: 10 sharp edges, ranked S1-S10 by data-loss risk, each with file:line, fix sketch, and a click-to-expand body.

Severity rubric — what each tier means

TierDefinitionSequencing
Data loss A code path that can lose or corrupt user data. Requires recovery from backups or version history. Block any production use. Fix before merge.
High Operationally serious — security exposure, capacity exhaustion, irrecoverable state without manual intervention. Block any non-dev environment. Fix before launch.
Medium Quality/operability issue. Surfaces under load or specific scenarios. Workarounds exist. Track as follow-up; don't block initial ship.
Low Cleanup, polish, or nice-to-have. Not surfaced unless you go looking. File as ticket; fix when the area is touched anyway.

Sample findings from the FUSE audit

Each finding has a one-line title, location, severity, body, and fix. The form is the value — without it, an audit is a vibes report.

S1 · Data loss
_load_manifest swallow-and-empty
workspaces/manager.py:644-652. Bare except Exception → empty() can't distinguish a 404 (legit empty) from a 503 (transient → should retry). A transient GCS read failure during _checkpoint produces an empty manifest, then unconditionally uploads it over the real one. Fix: distinguish 404 from 5xx; on 5xx, raise + log + abort the checkpoint.
S2 · Data loss
No if-generation-match on manifest upload
manager.py:537-542. Two concurrent checkpoints from different pods both download v(N), both bump to v(N+1), both upload — last writer wins; the other's blob uploads orphan. Per-session asyncio.Lock doesn't span processes. Fix: pass if_generation_match=existing.generation; on precondition-failed, re-load and retry-with-merge.
S4 · High
WorkspaceManager is one-org-per-process
manager.py:147. org_id baked at construction. Cannot serve multiple orgs from one process — incompatible with the locked 5-scope hierarchy. Fix: replace org_id: str with scope_resolver: Callable[[Session], MemoryScope]. ~4 hours.
S9 · High
systemd unit + IaC gaps
scripts/sandbox-fuse.service. No hardening; hardcoded User=sangeet; bucket creation, IAM, gcsfuse install all out-of-band; manual ADC rotation = degradation until a human re-auths. Fix: 30-min hardening pass + workload identity migration + Terraform modules for any non-dev environment.

The 10 findings are sequenced: S1-S4 (data-loss class) block any production use; S5-S8 (operability) block production beyond a single dev VM; S9-S10 (deployment) block any non-dev environment. Sequencing is the audit's value-add — without it, "10 problems" looks the same as "3 problems."

The cherry-pick decision, made after ranking

With severity in hand, the cherry-pick decision is mechanical:

  • Take the library. workspaces/{manifest,schemas,manager}.py + tests. The data-loss class fixes (S1-S4) are concrete and small — patch them in the cherry-pick PR.
  • Leave the consumer. carelink-agi/app/ is a parallel runtime that conflicts with Decision 1 (the BackendAGIService + WorkerAGIService split). It's a different architectural commitment.
  • Defer FUSE enable. The mount scripts ship in PR 5 disabled. S5 (cross-session memory propagation lag), S9 (systemd hardening), S10 (versioning lifecycle) are real but addressable separately.

Without the audit, "take it" or "leave it" were the only options on the table — both wrong.

Where it succeeds

Win · Linux kernel maintainership
Patches reviewed with structured rubrics for decades
Maintainers don't ad-hoc-evaluate patches; they apply known criteria (style, ABI compat, test coverage, security implications). Severity is implicit but consistent. The longest-lived audit culture in software.
Win · OWASP DREAD / STRIDE
Threat models with named severity tiers
"This vulnerability is a STRIDE-T at DREAD 6/10" produces decisions ("fix this sprint" vs "track in backlog") that "this looks bad" can't.
Win · CVE numbering
Industry-wide consistent severity vocabulary
CVSS scoring isn't perfect, but it lets dozens of organizations triage thousands of vulnerabilities consistently. The reduction in coordination cost is the whole point.

Where it fails

Loss · Severity by gut feeling
Everyone rates everything "high"
When the rubric isn't calibrated, severity inflates. The data-loss tier becomes meaningless because all 20 findings are tagged "data loss." Severity is a finite resource — spend it.
Loss · Findings without remediation
"Concerns" with no fix sketch
A bare list of concerns is a complaint. To act, the next reader needs "what to do about it" — even a one-line sketch is enough. Audit output is decision-ready or it's noise.
Loss · Misattribution to the author
Severity tagged on style preference, not data risk
Auditing under-disciplines becomes a way to rate someone else's choices as "wrong." Severity should describe what could happen, not how the author wrote it. Aesthetics aren't S1.
Loss · Audit can't reproduce author's context
"This is wrong" — actually a misunderstanding of the constraints
Sometimes the author had context the audit can't reproduce (a deployment constraint, a dependency you can't see). When confidence is low, the right move is "ask the author" — not lower the severity, not raise it. Audits work; they don't replace conversations with the author.

Audit output shape — what makes it decision-ready

  1. Title. One line. Names the issue, not the file.
  2. Location. file:line. Concrete, copy-pasteable.
  3. Severity. One of {data-loss, high, medium, low}. Calibrated against the rubric, not against feelings.
  4. Body. 2-4 sentences. Names the failure mode, the trigger, and the consequence.
  5. Fix sketch. One line of "what to do." If you can't sketch a fix, you don't understand the issue yet.
  6. Sequencing. "Block before merge" / "Block before non-dev" / "Track as follow-up" — derived from severity, not added independently.

Growth path

Junior — reads code linearly, end-to-end, hoping understanding emerges. Senior — reads with structured questions ("what assumptions does this make?", "what fails if X is null?"). Architect — designs the audit format so the output is decision-ready: sharp edges + severity + remediation + sequencing. The senior audit answers a question the dispatcher already asked; the architect's audit produces a question the dispatcher should be asking.
