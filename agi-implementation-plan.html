What this document is for

We've spent several weeks designing the CareLink AGI runtime architecture. We've decided what to build. This document is about how to actually ship it — turning architectural decisions into pull requests, scheduling them, and surfacing the questions that still need answers.

In one paragraph (no jargon)

Right now, when a pharmacist starts using the AI assistant, everything runs inside one server process. That works for one customer, but doesn't scale to many customers, and doesn't survive server restarts gracefully. We're moving to a setup where the AI runs in a separate dedicated compute pool, paused sessions can be picked up by any server, and each customer's data lives in its own database. This document is the plan for getting from here to there.

The headlines

  1. Today is already one-database-per-deployment, just without the orchestration to manage many of them. Each running CareLink instance serves one customer. The migration is "build the routing layer that turns N independent deployments into one routed deployment," not "split a shared database."
  2. The compute side has a clean rollout strategy. A configuration flag (compute_mode) lets us keep the current code working while the new code goes in alongside it. Switch the flag, the new path activates. Nothing big-bang.
  3. The biggest single change is in routers/agi.py — about 10 endpoints need to change from "look up the AI in process memory" to "look up which server is hosting it and forward the request there." This is the dispatch boundary. It's the most consequential code change, but it's well-bounded.
  4. The second-biggest change is the workflow engine globals. A bunch of workflow code uses module-level variables ("set this once at process startup and forget it"). For multi-customer, those need to become per-customer factories.
  5. Two tracks can ship in parallel: the worker-pool track (move AI runtime to dedicated compute) and the multi-tenant track (per-customer database routing). They share early foundations but diverge after.

Now vs Target — the system shape

In one paragraph (no jargon)

Today, everything runs in one server process — the website talks to one backend, which holds the AI conversations in its own memory. If that server restarts, conversations vanish. If a second customer joins, they share the same database. The new shape splits this into specialized pieces — a thin "request handler" backend, a separate "AI runner" compute pool, and storage that any server can reach. Each customer gets their own database. Nothing is shared except the platform itself.

What's BUILT today

  • The AGI runtime works end-to-end for one customer in one process
  • Inngest functions for cold-revive (HITL pause/resume) — already work cross-process
  • WorkOS authentication + Permit.io role checks (single-tenant)
  • GCS plumbing for transcripts and skills
  • Tests for invoke_expert per expert (hermetic, mocked SDK)

What's MISSING and needs to be built

  • Lease primitive (single-writer per session, DB-backed)
  • Worker pool — separate Cloud Run service with its own container image
  • Backend dispatcher — knows which worker is hosting which session
  • Memory canonical store + selector + merger (the actual persistent learning layer)
  • Filestore mount for session workspaces (replacing /tmp)
  • Control plane DB + org provisioning workflow
  • JWT extension to carry org_id + middleware to route per-org
  • CareLinkRegistry (per-org service bag instead of one singleton)
  • AuthorizationService refactor (pass tenant per-method-call, not per-construction)
  • Workflow engine globals → per-org factories
  • Admin tooling (provisioning APIs, migration runner against N DBs)

The 9 layers — what depends on what

In plain English

You can't build the roof before the walls. These are the 9 layers, in build order. Layers 1-6 are the AI worker-pool migration. Layers 7-9 are the per-customer (multi-tenant) work. After Layers 1-2, the two tracks can advance in parallel.

1Foundation: schema + control plane
Add lease columns to AGISession. Build the central registry of customers → databases.

What gets built:

  • 5 new columns on agi_sessions for tracking which server hosts which conversation (Alembic migration)
  • NEW: control plane database — a small central database with one table mapping org_id → connection_string, storage_paths, HIPAA_tier
  • JWT auth layer extended to extract org_id from the token

Why first: nothing else can route per-customer until the routing data exists.

↓
2Routing primitives
Every request is now scoped to a specific customer.

What gets built:

  • CareLinkRegistry — when a request arrives, look up the right database connection for the customer it's for
  • AuthorizationService refactor — instead of being constructed per customer, the org_id is passed per method call
  • get_carelink dependency reads from request state (which has org_id from the JWT middleware)

Why second: services need to know which customer they're acting for before any per-customer logic works.

↓
3Lease primitives + worker entrypoint
The new AI compute pool gets its HTTP surface.

What gets built:

  • sessions/lease.py — 4 methods: acquire, refresh, release, lookup. DB-backed.
  • worker/entrypoint.py — the FastAPI app that runs INSIDE each worker pod, exposing internal endpoints like POST /worker/sessions/spawn, POST /worker/sessions/{id}/turn, etc.
  • worker/dispatcher.py — backend-side companion. Looks up "which worker holds this session?" and forwards the request via internal HTTP.

Why third: can be built and deployed independently. Doesn't replace anything yet — exists alongside the in-process path.

↓
4Storage decoupling
Session workspaces move from /tmp to shared storage. Skills come from a mount, not a copy.

What gets built:

  • workspace_base becomes environment-driven (dev still uses /tmp; prod uses Filestore mount)
  • _prepare_workspace uses deterministic per-(org, session) paths so any server can find the workspace
  • Skills bundle mounted via GCS FUSE read-only — instead of copying ~150MB per session
  • NEW memory/store.py, memory/selector.py, memory/merger.py — the canonical learning layer (the thing that's currently aspirational in expert MD prose but doesn't exist in code)

Why fourth: needs the control plane to know which storage paths each customer uses.

↓
5Dispatch boundary refactor (the big one)
The 10 endpoints that hold the AI in process memory now route to the worker pool.

What gets built:

  • Every endpoint in routers/agi.py that today does cl._agi_clients[session_id] now does "look up lease → forward request via HTTP to that worker"
  • _msg_to_sse_events extracted from routers/agi.py to a shared module (worker uses it too)
  • broadcast_stats_event in-process pub/sub becomes Redis pub/sub for cross-replica fan-out
  • Dispatcher streams server-sent events from worker → backend → browser

Why fifth: needs lease primitives (Layer 3) and storage (Layer 4) in place. This is the load-bearing change — it's where today's behavior actually changes.

↓
6Cold-revive worker-pool aware
Pause/resume of conversations works across server restarts.

What gets built:

  • _drive_cold_revive calls dispatcher RPC instead of running the AI in-process
  • _drive_cold_approval same pattern
  • _execute_parked_bash still runs Python subprocesses — but on the worker that holds the lease, with workspace path on shared Filestore

Why sixth: cold-revive already works cross-process; this just changes who owns the SDK client.

— Below this line: the multi-tenant track. Can advance in parallel with Layers 3-6 once Layers 1-2 are done. —

7Per-org workflow engine factories
The biggest tenancy refactor: module-level globals become per-customer factories.

What gets built:

  • set_workflow_dependencies in integrations/inngest/functions.py: replace module globals with a per-org factory
  • Same pattern in document_processing.py, fax_sweeps.py, scheduled_dispatcher.py, agi_cold_revive.py, rx_flows/_orchestrator_bridge.py
  • Each Inngest event payload must carry org_id for routing

Why seventh: requires Layer 2 (CareLinkRegistry) and is the biggest single tenancy refactor.

↓
8Provisioning + admin tooling (greenfield)
When a new customer signs up, what happens?

What gets built (entirely NEW):

  • NEW: carelink/provisioning/orchestrator.py — sequences org creation steps (allocate org_id, create DB, run migrations, provision Filestore subdir, create GCS buckets, seed Permit.io roles)
  • NEW: alembic-against-N-databases runner (one CLI command runs migrations against every customer's DB)
  • NEW: carelink-backend/app/routers/admin.py — admin-only API for provisioning

Why eighth: can be built later — manual provisioning gets us to the first customer; automation is needed when we have many.

↓
9Permit.io refactor + pharmacy projection
Role-based access checks become pharmacy-aware AND tenant-safe.

What gets built:

  • AuthorizationService.check accepts pharmacy_id in context (today it's there but ignored)
  • PermitClient permission cache key changes from role to (tenant, role) — fixes a cross-tenant cache collision bug
  • ~80-100 callsites that do _authz.require/check need to thread actor.pharmacy_id into the context

Why ninth: pre-existing gap (called out in 340B notes), aligns with our pharmacy-scoped HITL and memory design.

Prompt — what this PR accomplishes

This is the conversion. Up through PR 6, every PR shipped behind compute_mode="in_process" default — production behavior was unchanged. Code shape — endpoint dispatch branch

@router.post("/agi/sessions/{session_id}/query")
async def query_session(
    session_id: str,
    body: QueryRequest,
    cl: CareLink = Depends(get_carelink),
    claims: AuthClaims = Depends(get_current_auth_claims),
):
    if cl.agi._agi_config.compute_mode == "in_process":
        # Today's path — direct in-memory client lookup
        client = cl.agi._find_active_client(session_id)
        if client is None:
            raise HTTPException(404, "session not found / not active")
        return StreamingResponse(
            _serialize_sse(client.query(body.message)),
            media_type="text/event-stream",
        )
    else:
        # Worker pool path — lease lookup + dispatcher proxy
        lease = await SessionLease.lookup(cl.db, session_id)
        if lease is None:
            raise HTTPException(404, "session not found / lease expired")
        upstream = cl.agi._backend.dispatcher.query(
            session_id=session_id,
            worker_base_url=lease.worker_base_url,
            message=body.message,
            actor_id=claims.sub,
            org_id=claims.org_id,
        )
        return StreamingResponse(
            proxy_sse(upstream, heartbeat_seconds=15),
            media_type="text/event-stream",
        )

Prompt — what this PR accomplishes

The Inngest cold-revive flow (used when a HITL approval comes back days after the original session went idle) currently spins up an in-process AGI client to drive the resumed session. After PR 7, this is the LAST place the in-process path is hardcoded for production traffic. PR 8 makes cold-revive use the dispatcher RPC just like the live path.

Cold-revive's structure is unchanged — same Inngest function shape, same DB lookups, same workspace hydrate. Only the "how it talks to the worker" changes:

  • Today (in_process): _drive_cold_revive constructs an in-process AGIService, calls resume_with_answer directly. Workspace re-hydrates from local /tmp if cold-revive event lands on the same machine; fails otherwise.
  • After PR 8 (worker_pool): _drive_cold_revive calls the dispatcher's resume_with_answer RPC. Acquires (or refreshes) the session lease via SessionLease.acquire — picks any available worker, not necessarily the original. Workspace hydrates from PR 5's per-org canonical bucket (machine-independent).

Why this is needed now: without PR 8, cold-revive is the one production path that requires the original machine to still be alive — defeats the purpose of multi-replica deploy. Once PR 8 ships, multi-replica scale-out is unblocked (single-replica still works fine; multi-replica gains failover).

What surprised us — 5 findings worth flagging

In plain English

The two AI agents reading the codebase surfaced things we hadn't budgeted for. Some are bugs waiting to happen; others change the migration story. None are blockers, but all are worth knowing before we start writing code.

1Today is already one-DB-per-deployment, just without orchestration

The reframe: we'd been talking about "shared DB → split into per-org DBs" but the actual current shape is different. Each running prod deployment serves exactly one customer because Settings.database_url is baked at process startup. The migration isn't "split a shared DB" — it's "build the orchestration that turns N independent deployments into one routed deployment."

Why this matters: the migration story is much cleaner than I'd been describing. There's no "untangle co-mingled data" step. Each existing deployment can become an entry in the control plane registry by writing one row.

What we should verify: is this actually how production runs today? If there are multiple customers in one DB somewhere, the migration story changes significantly. Question for stakeholders.

2Streaming AI responses through backend → worker is a connection-pool concern

The risk: when an AI conversation runs (which can take minutes for an rx-flow), the backend holds an open HTTP connection to the worker the entire time, just to forward the streaming response to the browser. At hundreds of concurrent conversations, the backend's HTTP client connection pool needs to be sized correctly.

Why this matters: if we under-size the connection pool, conversations queue up and stall waiting for connection slots. We need to load-test this before going to production with multiple replicas.

What needs to happen: dispatcher uses httpx.AsyncClient with explicit pool sizing (recommend max_keepalive=300, max_connections=500 as starting point). Validate under expected concurrent-session load.

3PermitClient permission cache has a cross-tenant collision bug

The bug: Permit.io cache is currently keyed only by role (e.g., "pharmacist"). But Permit.io supports per-tenant role customization — "pharmacist" at Org A might have different permissions than "pharmacist" at Org B. Today's cache shares them.

Why this matters: in single-tenant production today this is invisible. The first multi-tenant customer with a customized role definition will surface this as a permission bug. Could be either over-permissive (security concern) or under-permissive (broken UX).

What needs to happen: change cache key from role to (tenant, role). Small fix (~10 lines in integrations/permit/client.py) but must land before any second customer goes live. Goes in PR T4.

4Lease ticker scoping for long expert calls

The risk: the lease has a 60-second expiry, refreshed every 20 seconds by a ticker running alongside the AI conversation. But individual "expert calls" (e.g., the billing expert running a complex check) can take longer than 60 seconds in some cases. If the ticker is scoped to the conversation drain (not the expert spawn), an expert call that takes 90 seconds could lose its lease mid-call.

Why this matters: rare but ugly failure mode. Lease loss mid-call surfaces as session crash with a confusing error message.

What we should do: raise lease TTL to 5 minutes (simple fix), OR move the ticker to a session-scoped daemon (more complex). Recommend the simple fix first; revisit if any expert routinely exceeds 5min. Goes in PR 3.

5Connection pool math gets dangerous at scale

The risk: SQLAlchemy default connection pool is 5 base + 10 overflow per database. With per-org DBs across multiple replicas, the math is "N orgs × ~15 connections × M replicas" = potentially thousands of Cloud SQL connections. Cloud SQL connection limits are not unlimited.

Why this matters: at ~20 customers and 4 backend replicas, we'd already be at 1,200 connections — close to default Cloud SQL limits.

What needs to happen: drop pool size per org to 2-3 base. Deploy PgBouncer in front of Cloud SQL. Connection management at multi-tenant scale is a real ops surface, not a freebie. Open decision below — do we deploy PgBouncer?
