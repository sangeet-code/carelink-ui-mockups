Rx-Flow Option B: Intake PoC, Expert Registry & MCP-vs-Native Dispatch

A walkthrough of where the rx-flow Option-B migration sits today, and what the architecture looks like if we replace the MCP invoke_expert wrapper with the Claude Agent SDK's native Agent tool. Drift handling is compared side-by-side for both paths.

Branch: feature/ai-orchestration-with-domain-agents Phase 0 + 1 shipped MCP path = today Native path = dormant, wired

What this document is

The rx-flow workflow is mid-migration from a single monolithic-prompt orchestrator (Option A) to an orchestrator + 8 scoped domain experts (Option B). PR 1 landed the first end-to-end expert (the Intake PoC) and the supporting Expert Registry. The orchestrator dispatches experts via a custom MCP tool invoke_expert. The Claude Agent SDK's native Agent tool is wired but unused — activating it would simplify the dispatch path at the cost of validation/retry guarantees. This doc lays out both paths and how each handles LLM-side and subprocess-side drift.

Glossary

MCP (Model Context Protocol)

A JSON-RPC protocol the Claude Agent SDK uses to expose custom tools to a running model. In this codebase the invoke_expert tool is registered on an in-process MCP server attached to the orchestrator's ClaudeAgentOptions(mcp_servers=...).

Native Agent tool

A first-class SDK tool that takes { subagent_type, prompt } and dispatches a subagent subprocess directly. Subagents come from ClaudeAgentOptions(agents={id: AgentDefinition}). No MCP layer involved.

Expert

A scoped subagent defined by a Markdown file with YAML frontmatter under carelink-pms/carelink/agi/agents/*.md. Each expert owns a narrow domain (intake, billing, DUR, fulfillment, shipping, clinical-checklist, rejection-classifier, data-entry).

Drift

The gap between what the orchestrator (or expert subprocess) is supposed to emit and what it actually emits. Output drift = wrong schema. Tool drift = calling a disallowed verb. Subprocess drift = silent crash. Each path handles each kind differently.

The Intake PoC

PR 1 of the Option B migration. The first scoped domain expert + the dispatcher that routes to it + the typed contracts that keep its inputs and outputs honest. Landed on commits aeff02a + 645727d.

What it does

rx-flow-intake is a scoped subagent that takes a parsed NCPDP NewRx message + a pharmacy_id and resolves four references against CareLink's records:

  1. Patient — match by last name + DOB. Create if safe; else escalate via HumanQuestion.
  2. Prescriber — match by NPI, fall back to name. Create if safe.
  3. Drug / inventory product — match by NDC, fall back to search-drugs by GCN if DAW=0 and no NDC hit.
  4. Insurance — match against the patient's record using the NewRx insurance segment.

Once those resolve, intake calls create-prescription to mint the rx_id and returns a structured IntakeExpertOutput. Anything ambiguous becomes a human_questions[] entry instead of a guess.

Why "PoC"

It's the first end-to-end Option-B expert dispatch. Intake exercises every piece the rest of the experts will use: registry load, frontmatter parse, input/output Pydantic validation, subprocess spawn, restish allowlist enforcement, schema-drift retry, fallback envelope. If intake's plumbing is sound, the other 7 experts plug in with config-only changes (a new .md file + matching contracts).

Status note. The Pydantic input schema for BillingExpertInput recently exposed an under-specified case: it required insurance_id but didn't accept the ncpdp_insurance dict that intake emits when no local Insurance record was matched. That's a contract gap, not a dispatch issue — fixed today with a model_validator requiring exactly one of the two.

Dispatch: MCP today vs Native if we flip

Same orchestrator LLM, same expert subprocess. The route between them is what changes. Click between the views.

MCP path — 4 hops, full validation/retry

Today, the orchestrator emits invoke_expert(expert_id, input). The SDK routes that to the in-process MCP server registered on ClaudeAgentOptions(mcp_servers=...). Our handler does the heavy lifting before the expert subprocess is even spawned, and again after it returns.

  1. Orchestrator LLM emits tool_use invoke_expert(expert_id="rx-flow-intake", input={...}). system-prompt-templates.ts:541-542 (instructs the model)
  2. SDK dispatches to MCP server via the configured mcp_servers={"agi": ...} map. service.py:1690 (mcp_servers kwarg)
  3. MCP tool handler runs — _invoke_expert(args) at tools.py:871, which calls into the dispatcher. tools.py:843-870 (sdk.tool registration), tools.py:871-930 (handler body)
  4. Input validation — spec.input_schema.model_validate(input). Pydantic raises immediately if the orchestrator passed a malformed input. Caught in the dispatcher; orchestrator gets a typed error envelope. tools.py:734
  5. Subprocess spawn (with one transient retry) — _try_spawn_with_subprocess_retry calls _spawn_expert_agent, which builds an sdk.ClaudeAgentOptions per-expert (system_prompt = expert body, allowed_tools, model, restish allowlist hook on Bash) and runs sdk.query(prompt, options). ProcessError / CLIConnectionError / rate-limit are classified; transient → backoff + retry once, permanent → typed envelope. tools.py:543-679 (retry wrapper), tools.py:108-329 (spawn)
  6. Subprocess runs the expert — claude CLI in a sandboxed workspace, executes Bash → restish carelink <verb>, emits a final JSON line.
  7. Output extraction — _extract_final_json(raw) grabs the last JSON object from the subprocess's stdout and parses it. tools.py:387-407
  8. Output validation (with one schema-drift retry) — Pydantic output_cls.model_validate(parsed). On failure, _retry_once_on_validation_error re-spawns the expert with the original input + a nudge message containing the ValidationError + the full JSON schema. Validates the second result. tools.py:410-500 (retry), tools.py:447-451 (nudge prose)
  9. Fallback envelope on double failure — if both attempts fail validation, the dispatcher returns {"status": "failed", "step_name": expert_id, "error": str(first_error), "raw": raw}. The orchestrator reads status == "failed" and routes to HITL. tools.py:787-792
  10. Orchestrator receives a Python dict — typed, validated (or a typed failure envelope). Drift is contained at the expert boundary; the orchestrator never sees malformed JSON.

Native path — 1 hop, hook-based validation only

If we delete invoke_expert and update the orchestrator's system prompt to use the SDK's native Agent tool, the route collapses. The SDK already has agents={id: AgentDefinition} populated by load_all_experts(registry) at service.py:1681,1691 — it just isn't being asked.

  1. Orchestrator LLM emits tool_use Agent(subagent_type="rx-flow-intake", prompt=""). (system prompt would change to teach this form)
  2. SDK looks up the agent definition in options.agents[subagent_type] — already populated by load_all_experts(registry). agents/loader.py:28-59 (load_all_experts), service.py:1691 (kwarg)
  3. SDK spawns the subagent subprocess directly — using the AgentDefinition's {description, prompt, tools, model}. Same workspace + env. No Python-side dispatcher in between.
  4. Subprocess runs the expert — identical to the MCP path. Bash → restish.
  5. PostToolUse hook intercepts the tool_result — a hook registered on Agent tool calls receives the raw tool_response. This is the only place we can do output validation now. The hook can updatedMCPToolOutput to mutate the result, but it cannot directly trigger a retry — the orchestrator LLM has to choose to re-invoke on the next turn based on what the hook leaves in the tool_result. claude_agent_sdk/types.py:280, 384 (PostToolUse + updatedMCPToolOutput)
  6. Orchestrator LLM receives a ToolResultBlock — text-shaped, possibly mutated by the hook. The LLM has to decide whether the tool_result indicates success or whether to re-invoke. The hook can write a structured error message into the tool_result content to nudge the LLM, but cannot guarantee a retry.
  7. Subprocess crash → orchestrator session tears down — on the native path, a subagent ProcessError isn't returned to the orchestrator as is_error: true. The SDK propagates it. SubagentStop hook fires but cannot prevent the parent from also failing. There is no "transient retry" equivalent. claude_agent_sdk/_internal/* (no error envelope on subagent crash)
What's lost from MCP path. Three things: (1) Pydantic input validation BEFORE spawn — moves to a PreToolUse hook, doable but a different surface. (2) One-shot output-validation retry — gone; the LLM has to retry. (3) Transient subprocess retry — gone; subprocess crash goes straight to session teardown.

Trace a single intake call — step by step

An operator triggers rx-flow on a NewRx fax. Same input on both paths; only the dispatch route differs. Steps where MCP and Native diverge are tagged inline.

  1. Operator action lands at FastAPI — POST /api/agi/sessions with the parsed NCPDP message + pharmacy_id. Auth resolved (actor_id), bearer minted with 1h TTL. backend router → AGIService.create_session
  2. AGIService prepares the session — creates workspace dir, mints bearer, persists AGISession row with status=active, builds ClaudeAgentOptions via _build_options. service.py::create_session + _build_options
  3. ClaudeAgentOptions is assembled — every shared resource gets wired into one object: system_prompt (orchestrator template), allowed_tools (orchestrator's lean set), mcp_servers={"agi": build_mcp_server(...)}, agents=load_all_experts(registry), hooks (PreToolUse for AskUserQuestion + Bash approval, PostToolUse for audit, SubagentStop for audit), env (HOME redirected, restish JSON, bearer in env), cwd (workspace), permission_mode=bypassPermissions. service.py:1684-1749
  4. Orchestrator ClaudeSDKClient spawns — long-lived. Reads system prompt from frontend template, sees invoke_expert in the toolset (or Agent on Native).
  5. Orchestrator LLM emits a tool_use for the first expert.
    • MCP today invoke_expert(expert_id="rx-flow-intake", input={...}) — the input dict carries ncpdp_message, pharmacy_id, and any folded human_answers from a prior HITL turn.
    • Native if flipped Agent(subagent_type="rx-flow-intake", prompt="") — same payload, encoded as text instead of an object property.
  6. Dispatch routes the call.
    • MCP SDK looks up "agi" in mcp_servers, invokes the registered invoke_expert tool's handler. Handler runs input Pydantic validation (line 734); on fail, returns typed error to the orchestrator without spawning anything.
    • Native SDK looks up "rx-flow-intake" in agents={}. Pre-spawn validation only happens if a PreToolUse hook on Agent tool calls is registered (would need to be added).
  7. Subprocess spawns.
    • MCP _spawn_expert_agent builds a fresh ClaudeAgentOptions per spawn: system_prompt = spec.body (preamble already prepended), allowed_tools = spec.allowed_tools, hooks = {PreToolUse: [restish_allowlist_hook(spec)]}, model from spec, env from same builder. Calls sdk.query(prompt, options).
    • Native SDK reads AgentDefinition directly: prompt + tools + model. Spawns with the same env / workspace as the orchestrator. The restish allowlist hook would still work because it's registered by the SDK on subagent spawn through the agents= mechanism — but the wiring is different: today it's wired in _spawn_expert_agent's per-spawn options; on Native it'd live on the parent's hooks={} with a HookMatcher scoped to subagent tool calls.
    tools.py:108-329 (MCP spawn) / agents/loader.py + SDK internals (Native)
  8. Subprocess cold-starts — claude CLI in workspace. Reads system prompt (preamble + body), discovers skills via SDK skill-loader, reads memories/ dir on disk in workspace.
  9. Expert LLM emits Bash tool_use — e.g., echo '{"last_name":"Smith","dob":"1980-01-01"}' | restish carelink list-patients. PreToolUse allowlist hook fires inside the expert subprocess: checks list-patients against spec.allowed_restish_commands → allowed. If the verb is mutating (e.g., create-prescription), the Bash approval hook also fires — operator decides hot or it cold-parks. tools.py:236-249 (allowlist) + hooks.py (approval gate)
  10. Bash command runs — bearer token from env authorizes restish; backend serves the response; expert reads stdout JSON.
  11. Expert may emit AskUserQuestion — flows up to the orchestrator's ClaudeAgentOptions.hooks registered PreToolUse for AskUserQuestion. Hot path: gate blocks subprocess until operator answers via REST. Cold path: persist to DB, fire Inngest, return synthetic answer on revive. Identical on both paths — the gate is wired on the orchestrator-side options.
  12. Expert finishes its work + emits the final JSON line (last line of stdout = single-line JSON validating IntakeExpertOutput).
  13. Output capture.
    • MCP _extract_final_json grabs the last JSON object from the raw subprocess stdout. Pydantic output_cls.model_validate(parsed). On fail: _retry_once_on_validation_error re-spawns with a nudge containing the schema. Returns either the validated dict or a typed failure envelope.
    • Native SDK delivers the subagent's result as a ToolResultBlock on the orchestrator's stream. PostToolUse hook (would-be-added) does its own _extract_final_json + Pydantic validate, then either passes through or rewrites the tool_result to flag drift. No automatic re-spawn.
  14. Trace + audit — every message (orchestrator + expert) flows into ExpertTraceRegistry. AuditCollector records the tool invocation. MCP path has per-message granularity because _spawn_expert_agent's loop publishes each SDK message; Native path drops to start/stop granularity via SubagentStart/Stop hooks.
  15. Result reaches the orchestrator.
    • MCP Python dict, typed or fallback envelope. Orchestrator branches on output.status.
    • Native ToolResultBlock with text content (possibly mutated by the hook). Orchestrator parses prose to decide what's there.
  16. Orchestrator decides next step — typically dispatches the next expert (e.g., rx-flow-data-entry) with the merged state. Loop continues until rx-flow reaches a terminal state (claim submitted / HITL escalation / hard failure).

DIVERGES ▶

DIVERGES ▶

    raw = input_data["tool_response"]["content"][0]["text"]
    try:
        parsed = _extract_final_json(raw)
        output_cls.model_validate(parsed)
        return {}  # passthrough
    except Exception as e:

DIVERGES ▶

MCP — auto-retry inside one tool call

  f"Your previous output did not match the required "
  f"schema. Error: {first_error}. Return a single-line "
  f"JSON object that validates against this schema: "
  f"{schema_json}"

  • Operator never sees the drift — invisible recovery
  • Cost: ~10s extra wall-clock + ~1× expert tokens

Native — LLM-mediated retry across turns

// Orchestrator's next turn sees:

// Orchestrator LLM has to DECIDE to re-invoke:

    emit Agent(subagent_type="rx-flow-intake", prompt=...)
  • Two orchestrator turns, two subagent spawns
  • Operator pays for an extra round-trip in latency + tokens
  • If orchestrator misreads or skips the marker, broken output flows downstream into billing

Impact: this is the biggest practical difference. MCP turns drift into invisible 10s blips; Native turns drift into operator-visible turns + a contained-failure risk.

Drift handling — both paths, scenario-by-scenario

"Drift" here means: the LLM (orchestrator or expert) emitted something that doesn't match the contract. Or the subprocess died. Or a tool was called wrong. Each path catches some, misses others. This is where the trade-offs are concrete.

Output-shape drift

D1. Output JSON missing a required field

Common

e.g., expert emits {"rx_id": "...", "patient_id": "..."} with no step_name.

MCP path Handled

  • Pydantic ValidationError at tools.py:766
  • _retry_once_on_validation_error re-spawns expert with nudge: "Your previous output did not match the required schema. Error: ... Return a single-line JSON object that validates against this schema: ..."
  • Most drift cases recover on retry 2
  • If retry also fails: typed envelope {"status":"failed", "error": "...", "raw": "..."} → orchestrator escalates to HITL

Native path Lossy

  • PostToolUse hook can run output_cls.model_validate(parsed) and mutate the tool_result with an error note
  • Hook cannot trigger a retry directly — orchestrator LLM must read the mutated tool_result and choose to re-invoke
  • If orchestrator misses the cue and proceeds, downstream experts get garbage input
  • Compensating: explicit error prose in tool_result + orchestrator system-prompt rule "if tool_result starts with [SCHEMA_ERROR], re-invoke"

D2. Wrong type on a field

Common

e.g., at_a_loss: "true" string instead of true bool; days_supply: "30" string.

MCP path Handled

  • Pydantic strict-mode reject; same retry path as D1
  • Output contracts are deliberately loose where coercion is safe (e.g., notes accepts str OR list[str]) — fewer false positives

Native path Lossy

  • Hook can validate + mutate, but no automatic retry
  • If the orchestrator interprets the mutated result as success, the wrong-type field flows downstream — billing reads days_supply as string and crashes deeper in the call chain

D3. Malformed JSON (truncated, prose around it)

Less common

e.g., expert emits prose explanation followed by JSON; or hits max_turns mid-emit.

MCP path Handled

  • _extract_final_json(raw) grabs the last JSON object from raw text — tolerates surrounding prose
  • Truncation → JSONDecodeError → same retry path

Native path Manual

  • Hook needs its own JSON-extractor (port the one from tools.py:387)
  • Truncation: hook can detect, but again — no retry mechanism, only nudge-via-mutated-result

Tool-use drift

D4. Expert calls a disallowed restish verb

Common during prompt-tuning

e.g., billing expert calls get-pharmacy when its allowlist doesn't include it (real example from earlier today).

MCP path Handled

  • PreToolUse hook on Bash inside the expert subprocess — build_restish_allowlist_hook at tools.py:236-249
  • Returns a structured error to the expert LLM that names the violation and the allowed list
  • Expert often self-corrects on next turn

Native path Handled

  • Same PreToolUse-on-Bash hook; allowlist enforcement is identical because it lives inside the expert subprocess, not in the dispatch layer
  • Independent of MCP vs Native

D5. Mutating Bash command requires operator approval

Hot or cold path

e.g., expert calls create-prescription. Hot: operator clicks approve in <60s. Cold: operator takes longer; subprocess exits.

MCP path Handled (hot + cold)

  • Hot: ToolApprovalGate + build_bash_approval_hook blocks until decision
  • Cold: today's just-shipped Part C — gate persists pending approval to DB, fires Inngest agi/cold_approval_requested; drain executes the parked command + injects synthetic tool_result on resume

Native path Handled (same)

  • Independent of dispatch route — approval gate lives inside the expert subprocess as a PreToolUse hook on Bash; cold path is Inngest-driven and doesn't care if the parent was MCP or Native

Subprocess drift

D6. Transient subprocess error (rate limit, network blip, GCS hiccup)

Common in production

MCP path Handled

  • classify_subprocess_failure distinguishes transient (RateLimitError, CLIConnectionError) from permanent
  • Transient → backoff + retry once at tools.py:659-677
  • Most blips invisible to orchestrator

Native path Degraded

  • SDK propagates ProcessError up to the orchestrator session
  • No automatic retry layer; depends on whether the SDK itself retries internally for that error class
  • If session tears down, cold-revive recovers on next operator action — but UX is "rx-flow died, click retry" instead of "transparent recover"

D7. Permanent subprocess error (OOM, claude CLI missing, malformed prompt)

Rare, surfaces as bug

MCP path Handled with envelope

  • Classified as non-transient → build_subprocess_failure_envelope(...)
  • Envelope: {status: failed, step_name, error, stderr_tail, raw: None}
  • Orchestrator branches on status == "failed" — never sees raw exception

Native path Session teardown

  • SubagentStop hook fires post-hoc with transcript path, but cannot return a synthetic tool_result to the parent
  • Parent session typically dies — recovery via cold-revive (re-spawn from operator action)
  • Loses the clean "expert failed, route to HITL" branching the orchestrator currently has

D8. Expert subprocess hangs (no progress, no exit)

Edge case

MCP path Bounded by SDK timeout

  • SDK's max_turns + per-call timeout caps wall-clock
  • On timeout: ProcessError → classified → retry or envelope

Native path Bounded by SDK timeout

  • Same SDK-level timeout applies
  • Session may receive a tool_result with timeout indication, or session may tear down depending on SDK internals

Orchestrator-side drift (independent of dispatch path)

D9. Orchestrator builds wrong input shape for an expert

Real today

e.g., the case earlier today: orchestrator passed insurance_matched: false + ncpdp_insurance: {...} but billing's input schema required insurance_id.

MCP path Caught at dispatcher

  • Pydantic ValidationError at tools.py:734 BEFORE spawn
  • Wasted-spawn cost = 0; error returned synchronously to the orchestrator's tool_result
  • Orchestrator can re-attempt with corrected input on the next turn

Native path PreToolUse hook

  • Hook can intercept Agent tool_use, validate input against expert's input schema, return permissionDecision: "deny" with reason
  • Orchestrator sees deny + reason in tool_result; same self-correction loop as MCP
  • Equivalent if the hook is wired correctly — this gap closes cleanly

Decision matrix

When to keep MCP, when to flip to Native, when to run both.

If reliability matters more than simplicity

Keep MCP — current state

The MCP path's automatic transient retry + automatic schema-drift retry + typed fallback envelope are real reliability features. On a flaky network or under intermittent rate-limit pressure, they hide blips that the native path would expose. In a clinical workflow where an expert failure can stall an rx, "transparent recover" is worth the LoC overhead.

Keep MCP if: production reliability is the #1 priority and you don't have observability to measure subprocess crash rates yet.

If simplicity matters more, or you want sibling parity

Flip to Native — Part D execution

~−400 net LoC. 4 hops → 1 hop. Matches the openaiagentssdk sibling pattern. PreToolUse hook + PostToolUse hook can rebuild input/output validation. The retry losses are real but bounded — most production drift is schema-shape drift, which the hook still catches; the LLM-driven re-invoke is slower than the auto-retry but typically works.

Flip if: you've measured subprocess crash rates and they're <1% over a representative window, AND you have telemetry to spot regressions if they appear.

If you want to measure first

Run both behind a feature flag — recommended

One PR adds a config flag agi.dispatch_mode = "mcp" | "native". Default mcp. Run a fraction of sessions on native. Compare:

  • Subprocess crash rate on each path (already trace-able via SubagentStop + existing audit)
  • Schema-drift rate (count ValidationError events on MCP; count PostToolUse-mutation events on native)
  • Mean turns-to-recover after drift (MCP = always 2, native = LLM-dependent)
  • End-to-end rx-flow success rate

After 1–2 weeks of data, decide. If native holds: delete invoke_expert + retry helpers + the flag (the cleanup is one more PR). If native degrades: keep MCP, document the experiment, and the native wiring stays dormant for a future attempt.

What this means for the in-flight branch

  1. Don't bundle MCP-removal with the in-flight ExpertRegistry / 2.x PRs. Let those land clean.
  2. Land the cheap wins independent of the path decision: prefer restish carelink <verb> for any new domain access in experts; never add a new in-repo MCP server without a documented reason. Add this as a standing rule in CLAUDE.md.
  3. Add the dispatch-mode flag as its own PR after PR 2.x lands. It's a 1-line config + a 5-line if/else in service.py::_build_options. Cheap to add, expensive to forget.
  4. Defer the deletion until the data is in. The dormant native wiring is ~20 LoC and keeps the option open at zero ongoing cost.
