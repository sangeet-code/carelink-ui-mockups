340B Drug Pricing Program

The 340B program (Section 340B of the Public Health Service Act, 1992) allows qualifying healthcare organizations — called covered entities — to purchase outpatient drugs at significantly reduced prices from manufacturers. CareLink operates as a contract pharmacy, dispensing 340B drugs on behalf of covered entities. This module is currently at 0% implementation and is a critical compliance blocker for 340B participation.

Key 340B Concepts

ENTITY

Covered Entity (CE)

HRSA-registered organization (hospital, FQHC, DSH, CAH, etc.) that qualifies for 340B pricing. Identified by an OPA ID (Office of Pharmacy Affairs). Must maintain eligibility through annual recertification.

PHARMACY

Contract Pharmacy

External pharmacy (like CareLink) that dispenses 340B drugs on behalf of a covered entity under a written contract. Must track all 340B dispenses separately and report to the CE's Third Party Administrator (TPA).

PRICING

Ceiling Price

Maximum price a manufacturer can charge for a 340B-eligible drug, calculated by HRSA using AMP (Average Manufacturer Price) minus the URA (Unit Rebate Amount). Published quarterly. Typically 25-50% below WAC.

BILLING

Split Billing

Decision engine at point-of-dispense that determines whether an Rx qualifies for 340B pricing or standard billing. Considers patient eligibility, drug source, payer type (Medicaid exclusion), and inventory availability.

Step 1: Rx Arrives

Prescription arrives via NCPDP SCRIPT eRx (NewRx) or manual data entry. Check for OfficeOfPharmacyAffairsID in the SCRIPT message — if present, the prescriber is at a covered entity and the Rx is flagged for 340B evaluation.

  • EXISTS eRx intake via RxLinc SCRIPT mailbox
  • EXISTS OfficeOfPharmacyAffairsID field defined in NCPDP SCRIPT XSD
  • GAP Parse and store OPA ID from incoming eRx
  • GAP Auto-flag Rx for 340B evaluation when OPA ID present

Step 2: Patient 340B Eligibility Check

Determine if the patient qualifies as a "patient of the covered entity." HRSA's three-prong test requires:

  • 1. Established relationship — Patient has records at the covered entity
  • 2. CE provider encounter — The prescribing provider is employed by or contracted with the CE
  • 3. Healthcare responsibility — The CE maintains responsibility for the patient's care

Implementation: New Patient340BEligibility model linking patient → covered entity with eligibility dates. Service checks eligibility at dispense time.

  • GAP Patient340BEligibility model (patient_id, covered_entity_id, effective_date, expiry_date, verification_method)
  • GAP Eligibility determination service with CE EHR data import

Step 3: Drug 340B Eligibility Check

Not all drugs are 340B-eligible. Check the NDC against the HRSA ceiling price list and verify 340B stock is available.

  • EXISTS ProductPricing.cost_basis JSONB — add 340b_ceiling_price
  • GAP HRSA ceiling price list import (quarterly update)
  • GAP NDC-level 340B eligibility check service

Step 4: Duplicate Discount Check (CRITICAL)

Federal law prohibits using BOTH 340B pricing AND Medicaid drug rebates for the same prescription.

If the patient's primary payer is Medicaid, the pharmacy MUST use standard (non-340B) inventory and standard billing. Using 340B inventory for a Medicaid patient creates a "duplicate discount" — the manufacturer would be paying the 340B discount AND the Medicaid rebate. This is the #1 compliance risk in 340B.

  • GAP Payer type identification (Medicaid BIN/PCN lookup)
  • GAP DuplicateDiscountCheck model — audit record of each check
  • GAP Hard-stop preventing 340B dispense for Medicaid patients

Step 5: Inventory Source Decision

The split billing engine selects the inventory source. If all checks pass, dispense from 340B lot (FEFO within 340B lots). If any check fails, dispense from standard lot.

  • EXISTS InventoryStockService with FEFO logic in decrement()
  • GAP Filter lots by acquisition_source before FEFO sort
  • GAP Fallback: if no 340B stock, use standard and flag for 340B PO replenishment

Step 6: Dispense from Selected Source

Use the existing InventoryStockService.adjust() to decrement from the chosen lot. The InventoryTransaction records the unit cost at the 340B ceiling price.

  • EXISTS InventoryStockService.adjust() with lot tracking
  • EXISTS InventoryTransaction records unit_cost and balance_after
  • GAP Record acquisition_source on the transaction
  • GAP Record covered_entity_id on the transaction for HRSA audit

Step 7: Submit Claim with 340B Pricing

When dispensing from 340B inventory, the claim is submitted with basis_of_cost="06" (340B Program Price) and ingredient_cost set to the 340B acquisition cost.

  • EXISTS ClaimRequest.basis_of_cost field — just set to "06" instead of "01"

Step 8: TPA Accumulation Record

After dispensing, record the transaction in the TPA accumulator for the covered entity's compliance reporting. The TPA reconciles 340B dispenses with replenishment orders.

Inventory Segregation

340B requires tracking drug acquisition source. Two approaches exist: virtual segregation (recommended) uses database-level tracking; physical segregation uses separate storage locations. CareLink's existing lot system supports both.

RECOMMENDED

Virtual Segregation

Same physical shelf, different lot records tagged by acquisition_source. Most contract pharmacies use this approach.

  • How it works: Each InventoryLot gets an acquisition_source field ("standard" or "340b"). At dispense, the split billing engine filters lots by source before applying FEFO.
  • Pros: Lower operational overhead, no duplicate shelf space, leverages existing composite unique constraint
  • Cons: Requires disciplined data entry when receiving 340B POs
  • CareLink fit: InventoryStock already has composite unique (product_id, location_id, lot_id) — different lots on the same shelf are already separate stock records
ALTERNATIVE

Physical Segregation

Separate storage locations for 340B vs standard stock. Simpler conceptually but operationally heavier.

  • How it works: Create dedicated InventoryLocation records (e.g., "SHELF-340B-A1"). All 340B stock goes to 340B locations. Standard stock to standard locations.
  • Pros: Easy to audit (physical inspection), clear separation
  • Cons: Duplicate shelf space, more complex receiving, harder for high-volume pharmacies
  • CareLink fit: InventoryLocation model supports this but doubles location management overhead

Compliance & Audit

340B has significant compliance requirements enforced by HRSA audits. Contract pharmacies must maintain comprehensive records and exchange data with covered entities' TPAs.

HIGH RISK

Duplicate Discount Prevention

Cannot bill Medicaid rebate AND use 340B pricing. The #1 cause of 340B audit findings. Need real-time payer check before selecting inventory source.

Model: DuplicateDiscountCheck — records every check with result, payer info, timestamp

HIGH RISK

HRSA Audit Trail

Every 340B dispense must be traceable: patient eligibility determination → inventory source → claim submission → pricing. HRSA audits cover 3+ years of records.

Leverage: Existing InventoryTransaction + AuditLog + new acquisition_source fields

REQUIRED

TPA File Exchange

Contract pharmacies send accumulator files to the CE's TPA (e.g., 340B ESP, Sentry, Macro Helix). TPA sends back replenishment authorizations. Typically daily or weekly.

Model: TPAAccumulatorRecord + file export service (CSV/pipe-delimited per TPA spec)

REQUIRED

Ceiling Price Compliance

Acquisition cost must not exceed HRSA ceiling price. Validate at PO receipt time. HRSA publishes ceiling prices quarterly — need automated import.

Store in: ProductPricing.cost_basis["340b_ceiling_price"] + quarterly update service

REQUIRED

Contract Pharmacy Agreements

Valid written contract required between pharmacy and each covered entity. Track effective dates, renewal schedules, and auto-alert on expiry.

Model: CoveredEntityContract with pharmacy_id, effective/expiry dates, TPA config

IMPORTANT

Patient Definition Compliance

"Patient of the covered entity" has specific HRSA-defined criteria: established relationship, CE provider encounter, ongoing care responsibility. Pharmacy must document eligibility determination.

Model: Patient340BEligibility with verification_method, effective/expiry dates
