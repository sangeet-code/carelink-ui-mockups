CareLink AGI — Storage Architecture & Session Lifecycle

Generated 2026-04-30. Companion to agi-architecture-overview.html and agi-sandbox-memory-architecture.html.
Deep dive on GCS FUSE for memory storage, the two-tier storage architecture (Filestore for hot session workspace + GCS for persistent canonical memory), and the full end-to-end session lifecycle from frontend click through cold-revive to post-task cleanup. Each lifecycle phase has its own storage-layer interaction diagram.

The architectural insight

"GCS FUSE for memory" implicitly assumes one storage tier serves all our needs. But our workload has two distinct storage roles that want different primitives:

Two-tier storage with explicit role separation
Filestore (hot session workspace) + GCS (persistent canonical memory + cold archive). Optional GCS FUSE for the read-only skills bundle.
  • Filestore (NFS RWX, ~$0.45/GiB-mo) — hosts active and paused session workspaces. Sub-ms latency. Bounded volume (only active+paused sessions). Mounts on every backend replica so cold-revive on a different replica works natively.
  • GCS (object storage, ~$0.020/GiB-mo) — hosts canonical memory (per-org bucket prefix), final transcripts, audit archives. Unbounded volume (years of accumulated memory entries). Accessed via native API at session start (read snapshot) and session end (merge writes).
  • GCS FUSE — a useful tool for the skills bundle (read-only, large, rarely changes — caching wins). NOT the primary mechanism for canonical memory writes (POSIX gaps + write-amplification).
  • GCS Coldline — automatic tier-down via Object Lifecycle Management for entries older than 30d. Compliance retention at $0.004/GiB-mo.

Key insight: storage and compute decouple cleanly

The two storage tiers live in our GCP project, not inside any sandbox vendor's primitives. Cloud Run worker pools, GKE Autopilot, Daytona, or self-hosted Firecracker can all mount the same Filestore + read/write the same GCS bucket. Compute becomes vendor-replaceable; storage stays stable. This is the migration property the previous discussion landed on, made concrete.

Where GCS FUSE specifically wins for us

  • Skills bundle as read-only mount: 150MB tree, mounted once per replica, cached aggressively, symlinked into per-session workspace. Avoids the per-session skills copy that's currently the slowest part of provisioning.
  • Per-org canonical memory as a "browseable" view (optional): admins can ls /mnt/memory/orgX/pharmacy/P1/ on a maintenance pod for ad-hoc inspection without writing tooling.

Where GCS FUSE specifically loses for us

  • SDK transcript writes: Claude CLI writes to ~/.claude/projects/<cwd>/<sid>.jsonl hundreds of times per turn. GCS FUSE write-amplification (each write = full object rewrite under the hood) would be expensive and slow.
  • POSIX flock / atomic append: our memory merger needs reliable concurrency control. GCS conditional generation (if-generation-match) via API is more reliable than FUSE's translation layer.
  • Listing a busy directory: O(N) GCS list ops add up. Our merger walks the workspace memories dir at session-end — local FS list is microseconds, FUSE list is per-page-of-objects API calls.

System topology

End-to-end. From the operator's browser to the durable storage layer, with the sandbox compute fleet in between.

Notable properties of this topology

  • Frontend only ever talks HTTPS to Backend. Never directly to sandbox or storage. The org_id claim in the JWT is the routing key.
  • Backend is Cloud Run regular service (request-driven, scales to zero on idle). Different from the sandbox tier which is worker pools (always-on, pull-based).
  • Sandbox fleet is workload-specific. Active sessions hold a lease. waiting_cold sessions consume zero compute — only their workspace persists on Filestore.
  • Inngest functions handle durable cross-replica work: cold-revive (any replica) and TTL sweeper (cron). They write to GCS canonical and Filestore equally because both are shared resources.
  • Storage layer is the durable backing. Compute is interchangeable; storage is stable.

How the tiers interact

SESSION START (pre-task)
  1. Create workspace dir on Filestore: /workspace/<org>/<P>/<sid>/
  2. Memory selector reads canonical from GCS (per-org bucket)
     → Filters by workflow.memory_scope.read + role + pharmacy
     → Copies subset to Filestore: workspace/memories/...
     → Writes manifest to workspace/.memory-manifest.json
  3. Skills bundle: symlinked from GCS-FUSE-mount → workspace/.claude/skills/
  4. Bearer minted → workspace/.config/restish/apis.json

SESSION ACTIVE (during work)
  - All reads/writes hit Filestore. GCS is NOT touched.
  - SDK JSONL transcript writes to Filestore (sub-ms).
  - Memory edits append to Filestore workspace/memories/...
  - Skills reads cached from GCS FUSE (sub-ms after first hit).

SESSION END (post-task)
  1. Memory merger walks Filestore workspace/memories/
  2. For each modified file: extract added bytes (vs manifest snapshot)
  3. Wrap in <!-- entry: --> block with actor + role + ts
  4. Atomic write to GCS canonical via if-generation-match
  5. Final transcript uploaded to GCS transcripts bucket
  6. Filestore workspace rmtree'd

LATER (cron)
  - TTL sweeper walks GCS canonical, prunes entries > 14d
  - Object Lifecycle auto-moves entries > 30d → Coldline
  - Object Lifecycle auto-deletes Coldline > 7yr (compliance window)

Why not GCS for everything?

If we put workspace state on GCS (whether via FUSE or API), three things break:

  1. SDK transcript JSONL writes hundreds of times per turn. Each FUSE write = full object rewrite under the hood. Latency + cost both bad.
  2. Per-session lock semantics — fcntl.flock and atomic open/close that the SDK assumes don't translate cleanly to object storage.
  3. Latency budget on hot ops — every memory read during the session would be 50-300ms instead of sub-ms.

By having Filestore as the hot tier and GCS as the cold tier, each storage primitive does what it's designed for. The boundary is exactly two events per session: snapshot read at start, merge write at end.

Why not Filestore for everything?

If we put canonical memory on Filestore too, two things bite:

  1. Cost. Filestore Enterprise is 22x more expensive per GB than GCS Standard. Canonical memory grows unbounded over time. At a year of accumulation, you're paying for terabytes at $0.45/GiB instead of $0.020/GiB.
  2. Migration. If we ever leave GCP, GCS-to-S3 is a one-day rsync. Filestore-to-EFS is significantly more involved (need to spin up intermediate compute, copy via NFS+rsync).

GCS FUSE deep dive

GCS FUSE (now branded "Cloud Storage FUSE") is a FUSE filesystem driver that mounts a GCS bucket as a directory. Originally a Google open-source project, now a supported GCP product with native Cloud Run + GKE integration.

What it actually is

  • Mounts a GCS bucket at a local mount point (e.g., /mnt/canonical-memory/)
  • Translates POSIX FS operations to GCS API calls
  • Caches metadata, file content, and stat info locally for performance
  • Available as a Cloud Run volume mount (GA 2024) — no need to install the FUSE driver yourself
  • HIPAA-eligible (it's just GCS underneath)

The caching layer matters a lot

Two caches are configurable:

  • Metadata cache: stat info (file existence, size, mtime). TTL configurable. Default short (~1 minute) to avoid serving stale stats.
  • File content cache: actual file bytes. Stored on local SSD or in memory. Configurable size (max-size + per-file size). LRU eviction.

With aggressive caching, FUSE for read-heavy mostly-static content (like our skills bundle) is essentially free after the first read. With minimal caching or write-heavy patterns, every operation hits GCS API.

Costs (often-forgotten)

FUSE adds GCS API call costs that wouldn't exist with Filestore:

  • Class A operations (writes, list, etc.): $0.05 per 10K ops
  • Class B operations (reads): $0.004 per 10K ops
  • For a session that reads ~50 small files at start (50 GET ops) and writes ~5 at end (5 PUT ops): ~$0.0000025 per session in API costs. Negligible per session.
  • BUT — at 5K sessions/day × 50 reads + 5 writes = 275K ops/day = $0.001/day in API costs. Still negligible.
  • The real cost issue is if you mistakenly use FUSE for high-frequency operations (SDK transcript: hundreds of writes per turn × thousands of turns/day = millions of ops = nontrivial dollars). Don't.

HIPAA & security

  • FUSE driver itself is just a translation layer — no separate compliance scope
  • Underlying GCS is HIPAA-eligible
  • Encryption at rest (Google-managed or CMEK) handled at GCS layer, transparent to FUSE
  • Encryption in transit handled at GCS API layer, transparent to FUSE
  • IAM enforcement happens at GCS layer — FUSE inherits the GCS service account's permissions

Where GCS FUSE makes sense for us

Skills bundle (read-only, large, slowly-changing). Mount the skills GCS bucket at /mnt/skills/<version>/ on every Cloud Run replica. Read-only flag set. Aggressive metadata + file content caching. Symlinked into per-session workspace at provisioning time. Saves the 100-200ms per-session skills copy entirely.

Where it does NOT make sense

SDK transcript JSONL writes. The Claude CLI writes to ~/.claude/projects/<cwd>/<sid>.jsonl hundreds of times per turn. Each write would trigger a full object rewrite via FUSE. Use Filestore (real append) for this.
Active workspace memory writes during session. Same write-amplification issue. Use Filestore.
Memory merger at session end. While technically possible via FUSE, the merger needs concurrency control (multiple sessions ending simultaneously, both touching the same canonical file). GCS native API + if-generation-match is more reliable than FUSE + workaround locks.

Verdict on GCS FUSE for our use case

Use it for one specific role: the read-only skills bundle mount. That role plays to all of FUSE's strengths (read-heavy, cacheable, large enough that copying-per-session is wasteful). Avoid it for everything else where the POSIX gaps or write-amplification would bite.

End-to-end lifecycle flow

Six phases, each with its own storage interaction profile. Click any phase pill below to step through, or use ← → keys.

Phase:
1 Session created
User clicks "start rx-flow" in the browser. Backend authenticates, picks an org DB, creates a session row, assigns a sandbox replica.

What this means

  • The session now has a row in the org DB and a workspace path on Filestore.
  • Lease is held by replica X — only X can spawn the orchestrator subprocess for this session.
  • No GCS canonical interaction yet. That happens in Phase 2.
  • Skills mount is a replica-level concern (mounted once at replica boot, not per-session).
2 Pre-task injection
Skills, agent definitions, memory snapshot, orchestrator system prompt — all assembled into the workspace before the SDK subprocess spawns. Prep is the expensive bit; execution is fast once it starts.

The four inputs in detail

hot

persistent

What's NOT happening this phase

  • No SDK subprocess running yet
  • No orchestrator turn yet
  • No restish API call yet
  • No expert spawn yet

Pre-task is purely setup. Once the workspace is fully assembled, the orchestrator subprocess spawns with cwd=workspace and starts running.

Orchestrator runs. Calls experts. Experts call restish. Memories get written. The hot loop where 90% of session wall-clock happens.

Untouched. The session is operating against a frozen snapshot.

Why this phase is on Filestore-only

  • Frequent writes (transcript JSONL, memory appends, restish output captures) need real POSIX append semantics
  • Latency matters at the per-turn level — 50-300ms GCS round-trip per write would slow turn-time noticeably
  • The session is "operating against a frozen snapshot" of canonical memory — no need to revalidate against GCS during work
  • Append-only invariant is enforced at the merge boundary (Phase 6), not during this phase
4 HITL pause (waiting_hot → waiting_cold)
An operator question or mutating-Bash approval blocks the session. Hot wait for ~60s; if no answer, cold-park (subprocess exits, workspace persists).

Why workspace persists during cold-park

  • The Filestore export is shared across all backend replicas. Whether the original replica X is still alive doesn't matter.
  • The session row points at the workspace path: extra_data["workspace"] = "/workspace/<org>/<P>/<sid>/"
  • The SDK CLI's own session state (.jsonl transcript) is in the workspace. Any future SDK client with resume=session_id AND cwd=workspace picks up where it left off.
  • Memory writes from this session HAVE NOT YET been committed to canonical. They're staged in workspace/memories/. The merger doesn't run until Phase 6.

What gets lost if we abandoned cold-park (TTL hits 7 days)

  • The workspace's in-progress memory writes never reach canonical (the merger doesn't run for abandoned sessions)
  • The Filestore directory is rmtree'd
  • The session row is marked abandoned; audit event logged with timestamp
  • Any side-effects already executed (restish writes that happened earlier in the session) are NOT rolled back — they're already persisted in CareLink's domain DB
5 Cold revive
Operator answers. Inngest event fires. Any backend replica picks up the workspace from shared Filestore and spawns a fresh subprocess via SDK resume.

Storage operations during cold revive

hot
Filestore: workspace path access
Replica Y mounts the same Filestore export as replica X did. Reads workspace contents (skills, agents, memories, manifest, JSONL transcript, apis.json). Sub-ms.
hot
Filestore: bearer rewrite
Re-mint internal HS256 token; overwrite apis.json. Single small write.
idle
GCS canonical
Untouched. Memory state lives in workspace; merge happens only at finalize.
skills

Cached on replica Y from prior sessions. No re-fetch.

What makes this work — the load-bearing properties

  • Filestore is RWX-NFS shared across all replicas. X and Y mount the same export at the same path.
  • Workspace path is recorded on the session row. Y reads it from DB; doesn't need to know what X did.
  • SDK resume reads its own JSONL transcript from the workspace. Continuity is the SDK's responsibility, not ours.
  • Lease forces single-writer. Y took the lease force=true; if X was alive and writing, X gets a stop signal.
6 Post-task / finalize
Orchestrator emits final result OR session ends. Memory merger runs. Audit flushed. Final transcript uploaded. Workspace destroyed.

The append-only invariant in code

The merger enforces "append-only" at the GCS write boundary, not at the OS level. If an expert tried to delete or modify pre-existing canonical content (via Edit tool), the merger silently drops the change. Only the post-snapshot additions get propagated:

snapshot_sha = manifest[key].snapshot_sha256
current_sha = sha256(workspace_file[:snapshot_bytes])

if current_sha != snapshot_sha:
    log_warning("Append-only violated; ignoring modified pre-snapshot content")
    # Fall through to extract additions only

if workspace_file.size > snapshot_bytes:
    added = workspace_file[snapshot_bytes:]
    new_entry = wrap(added, actor, role, session_id, ts)
    gcs_atomic_append(canonical_path, new_entry)

Concurrency safety

Two sessions finalizing simultaneously, both writing to the same canonical file:

  • Session 1 reads canonical at generation N, computes N+1, writes with if-generation-match=N → succeeds
  • Session 2 reads canonical at generation N (before Session 1 wrote), computes N+1, writes with if-generation-match=N → 412 Precondition Failed
  • Session 2 retries: reads canonical at generation N+1, computes N+2, writes with if-generation-match=N+1 → succeeds

Optimistic concurrency control via GCS generations. No flock needed. Distributed-aware.

Why this is much cheaper than "GCS for everything" or "Filestore for everything"

  • Filestore for everything: canonical memory grows unbounded; even at 10TB you'd be paying $4.5K/mo just for storage. With our two-tier design, GCS soaks up the unbounded growth at 22x lower per-GB cost.
  • GCS for everything: would mean putting workspace state on FUSE. Hot-path latency tanks (50-300ms per workspace read instead of sub-ms). API costs explode (every JSONL transcript write = a GCS PUT). Workable but slow + more expensive at hot scale.

Why GCS-as-canonical helps migration

GCS objects are S3-compatible at the protocol level. gsutil rsync to S3 is a single command. The memory format itself (signed entry blocks in MD files) is text — fully portable. If we ever need to leave GCP, the canonical memory store is the easiest piece to move.

Filestore-to-EFS is harder (network mount + rsync, not single-command), but Filestore content is bounded (~500GB) so it's still tractable.

Open verifications before committing

  1. GCS conditional generation match throughput at 5K-session fleet — at session-end peaks, how many CAS retries do we expect? Typically <1% but worth a load test.
  2. Cloud Run NFS no-locking impact on Claude SDK — verify the SDK doesn't internally rely on flock for transcript JSONL writes. Smoke test required.
  3. GCS Object Lifecycle delete semantics with active references — does the lifecycle rule fire on objects currently being read by FUSE? (Should be no impact since FUSE caches; but verify.)
  4. Per-org GCS bucket isolation cost — is one bucket-per-org cleaner than path-prefix-per-org? Bucket-per-org gives natural IAM boundaries but adds management overhead. Path-prefix is simpler.
  5. Filestore Enterprise IOPs at peak — 5K sessions × 50 ops/sec average = 250K IOPs/sec total. Filestore Enterprise tier is rated for higher; verify our specific share size aligns.

Cloud Run worker pools — operational deep dive

Worker pools went GA in 2026 as the Cloud Run primitive for long-running, pull-based, non-HTTP workloads. They're a meaningfully different shape than Cloud Run services — different timeout model, different scaling primitive, different cost calculation. This tab is the operational reference: where each knob lives, what each ceiling means, when we'd push past it.

Verdict: sufficient for our planned workload through ~10x scale. The 32 GiB per-instance memory cap is the operative ceiling but doesn't pinch — we scale wider, not deeper. Compute (8 vCPU) is far above what we'll use. No request-timeout problem (worker pools don't have one).

The five timeout knobs

Worker pools have a meaningfully different shape than Cloud Run services here. The famous "60-min request timeout" doesn't apply.

#TimeoutCloud Run servicesCloud Run worker pools
1Request timeoutHard cap (default 5 min, max 60 min)N/A — pools are pull-based, not request-driven
2Container startup probe timeoutConfigurable per deploymentSame — applies during instance bring-up
3SIGTERM → SIGKILL grace periodDefault 10s, configurableSame — load-bearing for our drain-on-scale-down
4Instance lifetimeEffectively unbounded; rotates on deploySame — instances run continuously until deploy/scale-down/infra-recycle
5VPC + NFS mount timeoutsDefault ~30s for connector; NFS retry-boundSame

The big architectural difference: Cloud Run services have the famous 60-minute hard request cap. Worker pools don't have that — they're not serving requests, they're processing pulled work. Our orchestrator subprocess can run for hours or days inside a single worker pool instance with no request-shaped ceiling cutting it off mid-flight.

What can still kill a long-running orchestrator

  • Deploy of new revision (replicas drained one at a time, SIGTERM grace applies)
  • Infrastructure-side recycling (Google rotates underlying VMs periodically, ~weekly typical)
  • Instance crash (OOM, segfault)
  • Manual scale-down

For all of these, our session-recovery model (lease + cold-revive on a different replica via shared Filestore) handles it cleanly. The instance dies; the session resumes elsewhere.

Where each knob is configured

1. Container startup probe

gcloud run worker-pools update agi-orchestrator-pool \
  --startup-probe-period-seconds=10 \
  --startup-probe-failure-threshold=10 \
  --startup-probe-timeout-seconds=30

2. SIGTERM grace period (recommended: 300s)

Default is 10 seconds, which is too aggressive for our workflow — finalize_session needs time to run the memory merger + audit flush + workspace teardown. Recommended: 300s (5 min) so a graceful drain can finalize in-flight sessions without losing memory writes.

gcloud run worker-pools update agi-orchestrator-pool \
  --termination-grace-period=300s

3. Instance memory + CPU + execution environment

gcloud run worker-pools update agi-orchestrator-pool \
  --memory=32Gi \
  --cpu=8 \
  --execution-environment=gen2

The --execution-environment=gen2 flag is critical — gen1 uses gVisor (more restrictive, fewer syscalls); gen2 uses a microVM (full Linux). For our subprocess-heavy workload, gen2 is required.

4. Scaling

gcloud run worker-pools update agi-orchestrator-pool \
  --min-instances=2 \
  --max-instances=50

Worker pools don't have "request concurrency" the way services do. Our orchestrator-per-session pattern means each instance hosts N sessions; we control N at the application layer (via the lease + per-replica capacity check).

5. Volume mounts (Filestore + GCS FUSE)

gcloud run worker-pools update agi-orchestrator-pool \
  --add-volume=name=workspace-fs,type=nfs,location=10.0.0.5:/workspace \
  --add-volume-mount=volume=workspace-fs,mount-path=/var/lib/carelink/agi/workspace \
  --add-volume=name=skills,type=cloud-storage,bucket=carelink-skills-prod \
  --add-volume-mount=volume=skills,mount-path=/mnt/skills

Cloud Run "Cloud Storage volumes" are GCS FUSE under the hood — GCP just rebrands them.

Required NFS mount option for fast cold start: pass --volume-options=tcp on the NFS volume or your mount adds ~3s on cold start (default mount tries other transports first).

Memory ceiling: 32 GiB per instance

Per-instance density math

ComponentRAM
Orchestrator subprocess (Claude CLI, Node)50-150 MB
Active expert subprocess (when running)50-150 MB
Workspace tmpfs (skills/agents/proposals/transcripts)5-50 MB
Container baseline (Python runtime, libs)~200 MB total
Per-session peak150-300 MB

Sessions per instance ceiling: (32 GiB - 200 MB baseline) / 250 MB per session ≈ 130. We'd cap at 100/instance for headroom.

Three ways the ceiling actually bites

(1) OOM kills are not graceful. When an instance hits its memory limit, the kernel SIGKILLs the highest-RSS process. There is no SIGTERM. There is no graceful drain. Whatever orchestrator/expert was on the wrong end loses its in-flight work.

For us this means: a single session that grows beyond expectation can take down its sibling sessions on the same instance. Mitigations:

  • Per-session memory cap enforced in application layer (kill orchestrator if it exceeds 500 MB)
  • Conservative session/instance density (100, not 150)
  • Watchdog that detects ballooning sessions and migrates them via lease handoff before they OOM

(2) Tmpfs counts against memory. Cloud Run's per-instance ephemeral disk is in-memory tmpfs. That includes anything we write outside the mounted volumes. If skills get baked into the container image and copied per-session into ephemeral disk, 100 MB per session × 100 sessions = 10 GB just in tmpfs. This is why GCS FUSE for skills is a real perf + cost win — keeps skills out of tmpfs entirely.

(3) Buffered transcript writes count too. The SDK's transcript JSONL eventually gets fsynced to Filestore (which doesn't count against instance memory) but the buffered-but-not-yet-fsynced content does. Hundreds of writes per turn means kilobytes of buffer per session at any moment — small but multiplies.

What 32 GiB doesn't let us do

  • Run >130 sessions on one instance (forced to scale wider, which is fine)
  • Run a single huge orchestrator with massive context (fine for us — sessions are bounded)
  • Cache large reference data in-process (mitigated by GCS FUSE for skills)

Compute ceiling: 8 vCPU per instance

Less interesting than memory because our workload is mostly idle CPU.

Per-session CPU profile

Our orchestrator is mostly waiting on the LLM (Anthropic API) or the database (restish backend). Real CPU work happens in bursts:

  • JSON parsing of LLM responses (~10-50ms)
  • Pydantic validation of expert outputs (~5-20ms)
  • restish CLI startup (Go binary cold start, ~50-100ms)
  • skill execution (Python runtime spin-up, ~50-200ms when applicable)

Per-session steady-state CPU: <0.05 vCPU average, with bursts to 0.5-1 vCPU during LLM-response processing. 100 sessions × 0.05 vCPU avg = 5 vCPU steady-state. Comfortable on 8 vCPU. Bursts can briefly saturate but Cloud Run "always-on CPU" mode handles this.

Three ways the compute ceiling bites

  1. Always-on CPU is required for worker pools — no request boundaries to tie CPU allocation to. Pricing factors this in (always-on is ~2x request-time pricing per vCPU-hr). Our cost estimates already use the always-on rate.
  2. Burstable, not sustained. Cloud Run gives you up to 8 vCPU but doesn't guarantee sustained availability. Under cluster pressure, a "noisy neighbor" instance could see throttling. Mitigation: keep utilization below 70% per instance.
  3. No NUMA / CPU pinning. GKE lets you pin pods to specific cores or NUMA nodes for deterministic perf. Cloud Run doesn't expose this. For most workloads this doesn't matter; for memory-bandwidth-sensitive workloads it can.

Deployment playbook — from current code to running in production

The operational steps to take what exists in carelink-pms/carelink/agi/ today and ship it as a Cloud Run worker pool deployment. Concrete file paths, gcloud commands, IAM permissions, and a 5-day shipping plan.

Current state vs target state

What stays the same

  • The frontend (no API contract changes)
  • The backend HTTP routes (POST /sessions, POST /messages, etc.)
  • The expert MD files (carelink-pms/carelink/agi/agents/*.md)
  • The contracts (carelink-pms/carelink/agi/contracts/rx_flow.py)
  • The MCP tool implementations (tools.py)
  • Inngest cold-revive functions (already wired)

What changes

  1. Split the backend: HTTP-bound bits stay in the Cloud Run service; orchestrator-bound bits move to a worker pool.
  2. Replace cl._agi_clients dict with lease-routed RPC: backend service holds the lease registry; worker pool replicas register on bring-up; backend dispatches "drive this session" jobs to whichever replica holds the lease.
  3. Move workspace from /tmp to Filestore mount: config.py workspace_base changes from /tmp/agi-workspace to /var/lib/carelink/agi/workspace (or wherever Filestore is mounted).
  4. Add memory selector + merger: at session start, pull from GCS canonical (NEW). At session end, merge back to GCS canonical (NEW).
  5. New deployment artifact: container image for the worker pool (different from the backend service image).

The worker pool entrypoint is a long-running Python process that:

  1. Registers itself with the backend service (lease registry) on startup
  2. Heartbeats to the backend every ~15s
  3. Listens for "drive this session" jobs (via internal HTTP, or pub/sub, or pull-queue)
  4. Spawns orchestrator subprocesses for assigned sessions
  5. Drains gracefully on SIGTERM

What "deployed and useful" looks like

On the day this is "deployed and useful for our currently-working agents":

  • An operator clicks "start rx-flow" in the browser
  • Backend creates a session row, dispatches to a worker pool replica
  • Worker pool replica: prepares workspace on Filestore, pulls memory snapshot from GCS, spawns Claude SDK orchestrator
  • Orchestrator runs through the 8 rx-flow experts (existing MDs, no changes)
  • Memories accumulate in workspace during session
  • HITL hits → cold-park → Inngest revive on a possibly-different replica → continues
  • Session finalizes → memory merger writes to GCS canonical → workspace rmtree'd
  • Next rx-flow session at the same pharmacy reads the previous session's memories from canonical
